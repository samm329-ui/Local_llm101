# Memory Bandwidth for Local AI Hardware 

## Chapter Introduction

In the previous chapter, we explored how Large Language Models (LLMs) work, how they consume memory, how quantization changes memory usage, how different runtimes behave, and how to choose models that fit available hardware.

Eventually, every local AI user reaches a new question:

> **"What hardware should I buy?"**

For many beginners, the answer seems obvious.

Buy the computer with the biggest memory.

If one machine has **128 GB** of memory while another has only **32 GB**, then surely the 128 GB machine must be dramatically faster.

This sounds reasonable.

Unfortunately, it is wrong.

As soon as you begin running modern language models locally, you discover something surprising.

A machine with **less memory** can sometimes generate responses several times faster than another machine with a much larger memory pool.

Likewise, two systems that appear almost identical on paper may produce completely different inference speeds.

Why does this happen?

Because modern AI hardware is not judged only by **how much memory it has.**

It is also judged by **how quickly it can use that memory.**

This chapter introduces one of the most important—and most misunderstood—hardware concepts in local AI:

**Memory Bandwidth.**

Throughout this chapter, we will learn why memory capacity and memory bandwidth solve completely different problems, why software matters just as much as hardware, why many marketing specifications are poor indicators of real AI performance, and how experienced local AI users evaluate hardware before making purchasing decisions.

By the end of this chapter, you should be able to look at a hardware specification sheet and understand what actually matters for running modern language models efficiently.

---

# Learning Objectives

By the end of this chapter, readers will be able to:

* Understand the difference between memory capacity and memory bandwidth.
* Learn why two computers with similar memory capacities can perform very differently.
* Understand why local AI inference is frequently limited by memory movement rather than computation.
* Learn a practical framework for evaluating local AI hardware.
* Understand why software optimization is just as important as hardware specifications.
* Recognize why marketing numbers often fail to predict real-world language model performance.

# 1. Introduction — Why Memory Matters More Than You Think

Earlier chapters introduced memory planning, model selection, quantization, runtimes, and VRAM estimation.

Those chapters answered an important question:

> **Can this model fit on my hardware?**

This chapter answers the next question:

> **If it fits, why does one computer generate tokens much faster than another?**

The answer is **memory bandwidth**.

Most beginners compare computers using only memory capacity.

Experienced practitioners compare three things:

* Memory Capacity
* Memory Bandwidth
* Software Stack

Together, these determine the real-world performance of a local AI system.

---

# 1.1 Why Bigger Memory Does Not Always Mean Faster AI

One of the most common assumptions made by beginners is:

> **"More memory automatically means a faster AI computer."**

This idea sounds logical.

If one computer has **128 GB** of memory and another has **32 GB**, then surely the larger machine must be faster.

Unfortunately, local AI does not work this way.

Running a language model involves two completely different questions.

The first question is:

> **Can the model fit into memory?**

The second question is:

> **Once the model fits, how quickly can the hardware use that memory?**

These are different engineering problems.

Memory **capacity** solves the first.

Memory **bandwidth** solves the second.

---

## 1.1.1 Capacity Decides Whether the Model Fits

Memory capacity describes how much information the computer can store at one time.

Examples include:

* 16 GB
* 24 GB
* 32 GB
* 64 GB
* 128 GB

If the combined memory required for:

* Model weights
* KV Cache
* Runtime overhead
* Temporary tensors
* Batch or concurrency overhead

is larger than the available memory, the model cannot run.

Conceptually:

```text
Available Memory

        │

        ▼

Can The Model Fit?
```

Capacity answers only one question:

> **Can the model fit?**

It does **not** answer:

> **How fast will it run?**

---

## 1.1.2 Bandwidth Decides How Fast the Hardware Can Work

After the model fits, another limitation appears.

The processor must continuously read billions of model parameters stored inside memory.

The speed at which those parameters move from memory to the processor is called **memory bandwidth**.

Memory bandwidth measures:

> **How much data can be transferred every second.**

It is commonly measured using:

* GB/s (Gigabytes per Second)
* TB/s (Terabytes per Second)

Conceptually:

```text
Memory

        │

Data Transfer

        │

        ▼

Processor
```

Higher bandwidth allows the processor to receive more data every second.

Lower bandwidth forces the processor to wait.

This is why people often summarize the idea like this:

> **Capacity decides whether the model fits.**

> **Bandwidth decides whether the hardware feels alive.**

---

## 1.1.3 Why Dedicated GPUs Can Beat Much Larger Unified-Memory Systems

This explains one of the most surprising observations in local AI.

A computer equipped with an **RTX 5090** or **RTX PRO 6000 Blackwell** may contain significantly less memory than a unified-memory machine such as a:

* Mac Studio M3 Ultra
* DGX Spark
* Ryzen AI Max (Strix Halo)

Yet the dedicated GPU often generates tokens dramatically faster.

Why?

Because once the model fits inside memory, the GPU can move that data much more quickly.

Unified-memory systems remain valuable because they can load much larger models.

However, they usually provide lower memory bandwidth than high-end dedicated GPUs.

As a result:

* Larger models may fit.
* Token generation is often slower.
* Multi-agent workflows become more difficult because multiple models compete for the same memory bandwidth.

# 1.2 A Better Mental Model for Local AI Hardware

Instead of thinking only about memory size, experienced practitioners think in terms of three interacting components.

```text
Local AI Hardware

        │

        ├── Capacity

        ├── Bandwidth

        └── Software Stack
```

These three pillars explain almost every hardware comparison in modern local AI.

---

## 1.2.1 Capacity

Capacity determines:

> **Can the model fit?**

---

## 1.2.2 Bandwidth

Bandwidth determines:

> **How quickly can memory feed the processor?**

A larger bandwidth allows the processor to spend more time computing and less time waiting.

---

## 1.2.3 Software Stack

The software stack includes:

* Inference runtimes
* GPU drivers
* Attention kernels
* Quantization implementations
* Compiler optimizations
* Memory management

Even extremely powerful hardware performs poorly if the software cannot efficiently utilize it.

Together these three ideas produce a simple mental model:

```text
Local AI Performance

=

Capacity

×

Bandwidth

×

Software Stack
```

Capacity determines what fits.

Bandwidth determines how hard the hardware can breathe.

The software stack determines how much of the specification sheet you actually experience.

That is the mental model.

Not:

* "AI PC."
* "NPU TOPS."
* Or whatever marketing slogan happens to be popular this year.

# 1.3 The Hardware Number You Should Actually Care About

Especially in the age of AI agents, memory bandwidth has become one of the most useful first-pass measurements of local AI hardware.

Memory bandwidth is **not** the same as **tokens per second**.

Many other factors influence inference speed:

* Model architecture
* Quantization
* Runtime quality
* Context length
* Attention kernels
* Driver optimization

However, memory bandwidth remains one of the easiest ways to separate hardware into meaningful performance tiers before running benchmarks.

It is not a complete answer.

It is a very good starting point.

# 1.5 Side Note — The Memory Tax People Mix Up

One of the biggest reasons people become confused when comparing AI hardware is that they unintentionally combine two completely different specifications into a single idea.

They look at a hardware specification sheet, see a large memory number, and immediately assume that the hardware must also be fast.

Unfortunately, this is one of the most common mistakes in local AI.

Memory **capacity** and memory **bandwidth** solve different engineering problems.

Treating them as though they are the same leads to poor purchasing decisions and misleading hardware comparisons.

Conceptually:

```text
Memory

        │

        ├── Capacity

        │      │

        │      ▼

        │   What Fits

        │

        └── Bandwidth

               │

               ▼

        How Fast It Runs
```

Many discussions on the internet unknowingly mix these two ideas together.

As a result, people confidently compare completely different machines while overlooking the specification that actually explains the performance difference.

This hidden misunderstanding can be thought of as the **memory tax**—the extra confusion people pay because they compare only memory size while ignoring memory speed.

## 1.5.1 Same Topic, Completely Different Trade-offs

Consider several modern AI systems.

An **RTX 5090** is NVIDIA's flagship consumer graphics card designed for demanding AI, rendering, and gaming workloads.

An **RTX PRO 6000 Blackwell** is NVIDIA's professional workstation GPU built for enterprise AI, scientific computing, and content creation.

Although one contains **32 GB** of VRAM and the other **96 GB**, both provide approximately the same memory bandwidth:

**1792 GB/s**

That means both can move data through memory at roughly the same speed.

The difference is capacity.

The RTX PRO 6000 can load substantially larger models before running out of memory.

The RTX 5090 reaches its memory limit sooner, even though both GPUs feed the processor equally quickly.

Now consider another category.

A **DGX Spark** is a compact AI development system built by NVIDIA.

Rather than targeting maximum raw bandwidth, it focuses on providing a large unified memory pool, a tightly integrated CUDA software stack, and a convenient development environment.

It offers:

* 128 GB unified memory
* Approximately 273 GB/s memory bandwidth

Similarly, **Ryzen AI Max**, also known by its architecture codename **Strix Halo**, is AMD's high-end Accelerated Processing Unit (APU) designed for AI laptops and compact workstations.

Unlike traditional GPUs, it shares a unified memory pool with the CPU, allowing as much as approximately **96 GB** to be allocated for graphics workloads depending on system configuration.

Apple approaches the problem differently.

The **Mac Studio M3 Ultra** combines Apple's high-bandwidth unified memory architecture with very large memory capacities.

Depending on the configuration, it supports up to **512 GB** of unified memory while providing approximately **819 GB/s** of bandwidth.

Each of these machines solves a different problem.

Conceptually:

```text
Large Capacity

        ≠

High Bandwidth

        ≠

Best Choice
```

Every hardware platform represents a different compromise between:

* Memory capacity
* Memory bandwidth
* Cost
* Power consumption
* Software ecosystem
* Expandability

There is no universally perfect machine.

The correct choice depends entirely on the workload.

## 1.5.2 Bandwidth Is Not the Whole Story

After reading the previous sections, it may be tempting to conclude:

> "Bandwidth is the only specification that matters."

This would simply replace one misunderstanding with another.

Bandwidth is extremely important.

It is **not** the only factor.

Real-world inference performance also depends on:

* Memory capacity
* Compute performance
* Software optimization
* Quantization
* Runtime quality
* Attention kernels
* Context length
* Model architecture

Bandwidth is best viewed as the **fastest way to stop being confused**, not as the complete answer.

It provides an excellent first approximation.

The remaining specifications explain the details.

# 1.6 What This Looks Like in Practice

Although every workload behaves differently, modern AI hardware naturally falls into several practical bandwidth tiers.

These tiers are not official standards.

They are useful mental models that help estimate how different systems are likely to perform.

---

## 1.6.1 Below Approximately 150 GB/s

Systems below roughly **150 GB/s** generally belong to the **thin-and-light** category.

Examples include:

* Ultrabooks
* Lightweight laptops
* Battery-focused devices
* Many AI PCs

These machines remain useful for:

* Small language models
* Basic assistants
* Offline note taking
* Educational experimentation

However, they should not be expected to compete with dedicated workstation GPUs running large language models.

Conceptually:

```text
< 150 GB/s

        │

        ▼

Portable

Efficient

Limited Throughput
```

---

## 1.6.2 Around 250–300 GB/s

Once memory bandwidth reaches approximately **250 to 300 GB/s**, unified-memory systems begin to feel much more practical.

Examples include:

* DGX Spark
* Mac mini M4 Pro
* Ryzen AI Max (Strix Halo)

These systems often provide enough bandwidth for comfortable local inference while simultaneously offering significantly larger shared memory pools than traditional consumer GPUs.

This makes them attractive for users running models that simply cannot fit into 24–32 GB of dedicated VRAM.

---

## 1.6.3 Around 450–650 GB/s

The **450–650 GB/s** range represents what many practitioners consider the modern workstation tier.

Examples include:

* Mac Studio M4 Max
* MacBook Pro M5 Max
* AMD Radeon AI PRO R9700
* Tenstorrent Blackhole p150

These systems balance:

* Strong bandwidth
* Professional workloads
* Larger models
* Higher throughput
* Better multitasking

For many developers, this range represents a comfortable balance between cost and capability.

---

## 1.6.4 Beyond 800 GB/s

Once bandwidth exceeds approximately **800 GB/s**, hardware enters the enthusiast and enterprise performance category.

These systems are expensive.

They are also extremely capable.

High-bandwidth hardware shines when workloads involve:

* Large models
* Continuous inference
* Multiple concurrent users
* Agent systems
* Long-running development sessions

Conceptually:

```text
Higher Bandwidth

        │

        ▼

Higher Throughput

        │

        ▼

Higher Cost
```

## 1.6.5 One Market? Not Really.

Modern local AI is often described as though every computer belongs to the same market.

In reality, this is misleading.

Today's ecosystem consists of several overlapping markets:

* Thin-and-light laptops
* Consumer desktops
* Professional workstations
* Unified-memory development systems
* Enterprise AI servers

Each category is optimized for different priorities.

Comparing them directly without understanding those priorities often leads to incorrect conclusions.

# 1.7 Discrete GPU Reality — Still the Bandwidth Kings

A **discrete GPU** is a dedicated graphics processor with its own high-speed memory.

Unlike integrated graphics or unified-memory systems, discrete GPUs do not share their primary memory with the CPU.

This allows them to achieve extremely high memory bandwidth.

If the model fits entirely inside GPU memory—or multiple GPUs are connected using technologies such as **NVLink** or coordinated through **Tensor Parallelism** over modern **PCIe Gen 5** interconnects—discrete GPUs remain the fastest solution for most local AI workloads.

Examples include:

* **RTX PRO 6000 Blackwell** — Professional NVIDIA workstation GPU with **96 GB VRAM** and approximately **1792 GB/s** bandwidth.
* **RTX 5090** — NVIDIA's flagship consumer GPU with **32 GB VRAM** and approximately **1792 GB/s** bandwidth.
* **RTX 4090** — Previous-generation flagship with **24 GB VRAM** and approximately **1008 GB/s** bandwidth.

AMD also provides powerful AI-capable GPUs.

Examples include:

* **Radeon RX 7900 XTX** — Consumer flagship featuring **24 GB VRAM** and approximately **960 GB/s** bandwidth.
* **Radeon PRO W7900** — Professional workstation GPU with **48 GB VRAM** and approximately **864 GB/s** bandwidth.
* **Radeon AI PRO R9700** — AI-focused professional GPU with **32 GB VRAM** and approximately **640 GB/s** bandwidth.

Intel has also entered the professional AI GPU market.

Examples include:

* **Arc Pro B65** — Approximately **32 GB VRAM** and around **608 GB/s** bandwidth.
* **Arc Pro B60** — Approximately **24 GB VRAM** and around **456 GB/s** bandwidth.

Conceptually:

```text
Discrete GPU

        │

Dedicated VRAM

        │

Very High Bandwidth

        │

Very High Throughput
```

Discrete GPUs excel because they can move enormous amounts of model data every second.

Their primary limitation is capacity.

If the model cannot fit into available VRAM, performance usually drops dramatically once memory spills outside the GPU.

# 1.8 Apple Silicon Reality — Balancing Capacity and Bandwidth

Apple approaches local AI differently from traditional GPU vendors.

Rather than pursuing the absolute highest memory bandwidth, Apple combines:

* Unified memory
* Relatively high bandwidth
* Large memory capacities
* Excellent power efficiency
* Quiet operation

This makes Apple systems attractive for developers who prefer a single integrated machine rather than managing multiple discrete GPUs.

Representative bandwidth figures include:

* Mac mini M4 — approximately **120 GB/s**
* MacBook Air M5 — approximately **153 GB/s**
* Mac mini M4 Pro — approximately **273 GB/s**
* MacBook Pro M5 Pro — approximately **307 GB/s**
* MacBook Pro M5 Max — up to **614 GB/s**
* Mac Studio M3 Ultra — approximately **819 GB/s** with configurations reaching **512 GB** of unified memory

Apple systems are particularly attractive when users need:

* One compact workstation.
* Quiet operation.
* Extremely large unified memory.
* Large models without GPU sharding.

Their primary weakness is raw throughput.

When maximum token generation speed or high concurrency is the highest priority, dedicated workstation GPUs generally remain faster.

---

# 1.9 DGX Spark — A Developer Appliance Rather Than a Bandwidth Monster

The **DGX Spark** occupies an interesting position within the local AI ecosystem.

Its specifications include:

* 128 GB unified memory.
* Approximately 273 GB/s memory bandwidth.
* Tight integration with NVIDIA's CUDA software ecosystem.

On paper, its bandwidth appears modest compared with high-end workstation GPUs.

Its value comes from somewhere else.

DGX Spark emphasizes:

* A coherent memory architecture.
* Strong software compatibility.
* Integrated development workflows.
* Easy experimentation with NVIDIA's AI stack.

It also supports emerging technologies such as **NVFP4**, a low-precision numerical format designed to improve inference efficiency on supported hardware and software stacks.

Although promising, many NVFP4 optimizations are still maturing, and practical benefits depend heavily on runtime support.

DGX Spark should therefore be viewed primarily as a **developer appliance** rather than a machine designed solely to maximize tokens per second.

Its strength lies in the overall platform, not simply in raw memory bandwidth.
