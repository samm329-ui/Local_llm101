# GPU Memory Math for LLMs: The Complete Mental Model (2026 Edition)

Most people who start running AI models locally make the same assumption.

They hear that a model is "7B" or "70B" and immediately ask:

"How much VRAM do I need?"

At first glance, this sounds like a simple question. Surely there must be a chart somewhere that says:

* 7B model → X GB VRAM
* 13B model → Y GB VRAM
* 70B model → Z GB VRAM

And while those charts exist, they only tell part of the story.

The reality is that two models with the same parameter count can require completely different amounts of memory depending on how they are stored, loaded, quantized, and executed.

To understand why, we need to start much further back than most explanations do.

We need to start with what a language model actually is.

---

## Chapter 1: What Is a Large Language Model?

When people hear terms like ChatGPT, Llama, Qwen, DeepSeek, Claude, Gemma, or Mistral, they often imagine something mysterious.

In reality, every modern Large Language Model (LLM) is a very large neural network.

A neural network is a mathematical system inspired loosely by the structure of biological brains. It consists of layers of interconnected mathematical units that process information and learn patterns from data.

Imagine teaching a child language by exposing them to billions of books, websites, conversations, articles, and documents.

Over time, the child begins to understand grammar, facts, relationships between ideas, and patterns in language.

A language model learns in a somewhat similar way, except instead of biological neurons, it uses mathematical parameters.

These parameters are the most important concept in the entire discussion.

Everything eventually comes back to them.

---

## Chapter 2: Parameters – The Actual "Knowledge" of the Model

Every neural network contains an enormous number of numerical values.

These numbers are called parameters, sometimes referred to as weights.

A parameter is simply a learned value that influences how strongly one part of the network affects another.

For example, a parameter might contain a value such as:

0.823

or

-1.742

or

0.00013

Individually, these numbers mean almost nothing.

Collectively, billions of them encode the behavior of the model.

When someone says:

"This is a 7B model."

they mean:

"This model contains approximately 7 billion parameters."

Similarly:

* 13B = 13 billion parameters
* 32B = 32 billion parameters
* 70B = 70 billion parameters
* 405B = 405 billion parameters

The larger the parameter count, the more information the model can potentially represent.

However, larger parameter counts also mean larger memory requirements.

This brings us to the next question.

How much memory does each parameter require?

Before answering that, we need to understand how computers store information.

---

## Chapter 3: Bits, Bytes, and the Language of Storage

At the lowest level, computers understand only two states:

0

and

1

These states are called bits.

A bit is the smallest unit of information in computing.

You can imagine a bit as a light switch.

OFF = 0

ON = 1

That is all a single bit can store.

A single bit is not very useful on its own, so computers group bits together.

Eight bits form one byte.

For example:

10110110

contains eight individual bits and therefore represents one byte.

From there, storage units grow larger:

* 1 Kilobyte (KB) = 1024 bytes
* 1 Megabyte (MB) = 1024 KB
* 1 Gigabyte (GB) = 1024 MB
* 1 Terabyte (TB) = 1024 GB

When your GPU advertises:

8 GB

12 GB

24 GB

48 GB

80 GB

it is describing how many billions of bytes of memory are available.

That memory is called VRAM.

---

## Chapter 4: RAM vs VRAM

Most people are familiar with RAM.

RAM stands for Random Access Memory.

RAM is connected primarily to the CPU.

When your operating system, browser, game, video editor, or code editor is running, data is temporarily stored in RAM.

VRAM stands for Video Random Access Memory.

VRAM is attached directly to the GPU.

Traditionally, VRAM was used primarily for graphics:

* Textures
* Geometry
* Shaders
* Frame buffers

However, modern AI workloads use GPUs for mathematical computation.

As a result, AI models are usually loaded into VRAM.

For local LLM inference, VRAM is often the single most important hardware limitation.

Not CPU.

Not SSD.

Not internet speed.

Memory.

If the model does not fit in memory, it does not run.

---

## Chapter 5: Why Model Size Alone Is Not Enough

Imagine two models.

Both contain 70 billion parameters.

At first glance, they seem identical.

But one occupies 140 GB.

The other occupies 35 GB.

How is that possible?

The answer lies in how each parameter is stored.

The parameter count tells us how many numbers exist.

The precision format tells us how many bits are used to store each number.

This is where FP16, BF16, FP8, INT8, GPTQ, AWQ, NF4, and GGUF enter the picture.

---

## Chapter 6: Floating Point Numbers

Neural networks rely heavily on decimal values.

Examples include:

0.25

3.14159

-1.87

0.00004

Computers use special numerical formats to represent these values.

These are called floating-point formats.

The most common historically has been FP32.

FP stands for Floating Point.

The number indicates how many bits are used.

FP32 means:

32 bits per value.

Since 8 bits equal one byte:

32 bits = 4 bytes.

Therefore every parameter consumes 4 bytes.

A model with one billion parameters would require approximately:

4 GB.

A 7B model would therefore require approximately:

28 GB.

This is why FP32 is rarely used for inference today.

It is expensive.

---

## Chapter 7: FP16 and BF16

To reduce memory usage, the industry moved toward 16-bit formats.

FP16 stands for Half Precision Floating Point.

BF16 stands for Brain Floating Point 16.

Both use 16 bits per parameter.

Since:

16 bits = 2 bytes

every billion parameters requires approximately:

2 GB.

This leads to one of the most important rules in AI memory estimation:

Every 1 billion parameters stored in FP16 or BF16 requires roughly 2 GB of memory.

Examples:

7B → 14 GB

13B → 26 GB

70B → 140 GB

405B → 810 GB

BF16 generally provides better numerical behavior during training, but from a memory perspective, BF16 and FP16 are nearly identical.

---

## Chapter 8: FP8 and INT8

The next major step is reducing precision even further.

FP8 uses:

8 bits per parameter.

INT8 also uses:

8 bits per parameter.

Since:

8 bits = 1 byte

every billion parameters requires roughly:

1 GB.

Examples:

7B → 7 GB

13B → 13 GB

70B → 70 GB

405B → 405 GB

Immediately we have reduced memory requirements by half compared to FP16.

But engineers kept pushing further.

---

## Chapter 9: Quantization – The Trick That Makes Local AI Possible

Quantization is the process of storing parameters using fewer bits.

Instead of using:

16 bits

or

32 bits

we might use:

8 bits

4 bits

3 bits

or even 2 bits.

The goal is simple.

Reduce memory usage while preserving as much model quality as possible.

Think of image compression.

A raw photograph may occupy tens of megabytes.

A compressed JPEG may occupy only a fraction of that space while appearing almost identical to the human eye.

Quantization applies a similar philosophy to neural network weights.

The weights become approximations of their original values.

Some precision is lost.

A large amount of memory is saved.

---

## Chapter 10: The Power of 4-Bit Models

Four-bit quantization is particularly important because it makes large models accessible to consumer hardware.

Four bits equals:

Half a byte.

Therefore:

1 billion parameters require approximately 0.5 GB.

Examples:

7B → 3.5 GB

13B → 6.5 GB

70B → 35 GB

405B → roughly 200 GB

This is why a consumer GPU can sometimes run models that would otherwise require enterprise hardware.

---

## Chapter 11: GPTQ, AWQ, and NF4

Many people treat these as different model formats.

They are not.

They are quantization methods.

GPTQ stands for Generative Pre-trained Transformer Quantization.

Its goal is to compress models aggressively while preserving accuracy.

AWQ stands for Activation-Aware Weight Quantization.

Instead of treating every weight equally, AWQ identifies weights that matter most and protects them during compression.

NF4 stands for NormalFloat4.

It was popularized by QLoRA and is specifically designed around the statistical properties of neural network weights.

All three methods attempt to answer the same question:

How can we use fewer bits without damaging model quality too much?

---

## Chapter 12: GGUF Is Not Magic

A common misconception is that GGUF is a quantization method.

It is not.

GGUF is a file format.

Think of it like:

MP4 for video.

ZIP for archives.

PDF for documents.

GGUF for language models.

Inside a GGUF file, many quantization schemes may exist:

Q2_K

Q3_K

Q4_K

Q5_K

Q6_K

Each represents a different balance between quality and memory usage.

Approximate memory consumption per billion parameters:

Q2_K ≈ 0.33 GB

Q3_K ≈ 0.43 GB

Q4_K ≈ 0.56 GB

Q5_K ≈ 0.69 GB

Q6_K ≈ 0.82 GB

A 70B model stored as Q4_K therefore requires roughly:

70 × 0.56 ≈ 39 GB

for weights.

However, GGUF's efficiency is closely tied to runtimes such as llama.cpp.

Move the same model into a different framework and memory behavior may change dramatically.

---

## Chapter 13: The One Formula That Explains Everything

After all of these details, everything collapses into a single mental model:

VRAM ≈ Parameters × (Bits per Weight ÷ 8)

That is the entire game.

Every memory estimate begins here.

Examples:

7B FP16:

7 × (16 ÷ 8) = 14 GB

13B 4-bit:

13 × (4 ÷ 8) = 6.5 GB

70B FP8:

70 × (8 ÷ 8) = 70 GB

Almost every memory estimate used by AI engineers starts from this equation.

Unfortunately, this is only the beginning.

Because the model itself is not the only thing occupying memory.

---

## Chapter 14: The VRAM Tax Nobody Talks About

Many beginners calculate weight memory perfectly and still encounter:

CUDA Out Of Memory

Why?

Because weights are only one component of total memory usage.

Modern inference systems also require memory for:

* KV Cache
* Activations
* Batching
* Concurrency
* Framework overhead
* CUDA Graphs

These hidden costs often determine whether a deployment succeeds or fails.

---

## Chapter 15: KV Cache – The Silent Memory Killer

KV stands for Key-Value Cache.

When a model processes text, it creates internal representations of every token it has already seen.

Rather than recomputing those representations repeatedly, the model stores them.

This stored information is called the KV Cache.

You can think of it as the model's working memory for the current conversation.

Longer conversations require larger KV caches.

Which means more VRAM.

A 128K context window can consume dramatically more memory than a 4K context window, even with the exact same model.

This is one of the most common reasons memory usage surprises people.

---

## Chapter 16: Context Windows

A context window defines how many tokens the model can consider at once.

Examples include:

4K

8K

32K

128K

1M

Larger contexts allow the model to remember more information.

The tradeoff is memory consumption.

Long context means large KV cache.

Large KV cache means higher VRAM usage.

---

## Chapter 17: Activations

Activations are temporary values produced during computation.

Think of them as scratch paper used while solving a math problem.

The final answer is not the scratch paper.

But you still need the scratch paper while working.

Activations occupy memory during execution and contribute to total VRAM requirements.

---

## Chapter 18: Batching and Concurrency

Suppose one user sends a request.

The GPU processes it.

Now imagine one hundred users sending requests simultaneously.

The GPU must maintain state for all of them.

Batching refers to grouping multiple requests together to improve hardware utilization.

Concurrency refers to the number of requests being processed at the same time.

Both improve throughout.

Both consume additional memory.

Large production systems often use more memory for active users than for model weights themselves.

---

## Chapter 19: Framework Overhead

Different runtimes manage memory differently.

Transformers.

vLLM.

TensorRT-LLM.

llama.cpp.

Each has different allocation strategies, caching systems, optimization techniques, and execution paths.

As a result, identical models may consume different amounts of memory depending on the runtime.

---

## Chapter 20: CUDA Graphs

CUDA Graphs allow GPU workloads to be recorded and replayed efficiently.

The benefit is lower latency and more stable throughout.

The cost is additional reserved memory.

This tradeoff is often worthwhile in production deployments.

---

## Chapter 21: The MoE Trap

Mixture-of-Experts models create another layer of confusion.

Consider an "8×7B" model.

Many people immediately calculate:

56B parameters.

But only a subset of experts may be active for each token.

This means:

Memory cost and compute cost become different concepts.

The total parameter count affects memory requirements.

The active parameter count affects computational workload.

Confusing these two leads to wildly inaccurate estimates.

---

## Chapter 22: Dense Models vs MoE Models

Dense Model:

Every parameter participates in every token computation.

MoE Model:

All parameters may exist in memory, but only some participate in computation for a given token.

This distinction becomes increasingly important as MoE architectures become more common.

---

## Chapter 23: Multi-GPU Systems and Tensor Parallelism

What happens when a model requires more memory than one GPU provides?

You split the model across multiple GPUs.

Suppose a model requires:

140 GB

and your GPUs each have:

80 GB

You can distribute the model.

This technique is called Tensor Parallelism.

Instead of one GPU storing everything:

GPU 1 stores part of the model.

GPU 2 stores another part.

Together they provide enough memory to run the system.

---

## Chapter 24: What Actually Fits?

Ignoring overhead and focusing only on weights:

8 GB VRAM:

* ~3–4B FP16
* ~6–8B FP8
* ~12–16B 4-bit

12 GB VRAM:

* ~5–6B FP16
* ~10–12B FP8
* ~18–24B 4-bit

16 GB VRAM:

* ~7–8B FP16
* ~13–16B FP8
* ~25–32B 4-bit

24 GB VRAM:

* ~10–12B FP16
* ~20–24B FP8
* ~35–48B 4-bit

48 GB VRAM:

* ~20–24B FP16
* ~40–48B FP8
* ~70–96B 4-bit

80 GB VRAM:

* ~35–40B FP16
* ~70–80B FP8
* ~140–160B 4-bit

These are weight estimates only.

Real deployments require additional memory.

---

## Chapter 25: The Rule Professionals Actually Use

First estimate weight memory.

Then add overhead.

A common rule of thumb is:

Add approximately 10–30% extra VRAM.

However, if you are using:

* Long context windows
* Large KV caches
* High concurrency
* Agent systems
* Tool calling workflows
* Multi-user serving

you may need significantly more than that.

## The Final Mental Model

Forget the giant compatibility charts.

Forget memorizing dozens of model sizes.

Forget trying to remember every benchmark.

Everything starts with one question:

How many parameters are there?

Then:

How many bits are used to store each parameter?

Once you know those two numbers, you can estimate the weight memory of almost any model in seconds.

After that, account for:

* KV Cache
* Activations
* Batching
* Concurrency
* Runtime overhead
* CUDA Graphs

That is the complete memory story.

At its core, local AI is not really about models.

It is about memory.

The people who understand memory stop guessing whether a model will fit.

They can usually predict it before downloading the model at all.
