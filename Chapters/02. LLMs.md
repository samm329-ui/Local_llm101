# LLMs 101: A Practical Guide

## Introduction

Large Language Models are often discussed in terms of model size, GPU requirements, context windows, quantization methods, serving engines, and benchmark scores. While these topics are important, they are much easier to understand once the core mechanism of an LLM becomes intuitive.

At the center of every modern decoder-only language model is a simple loop. Text is converted into tokens. Those tokens are processed by a Transformer. Attention determines which earlier tokens are relevant to the current prediction. A Key-Value (KV) cache stores previously computed information so the model does not need to recompute the entire sequence repeatedly. The model then predicts a next token, appends it to the sequence, and repeats the process.

Everything else builds upon this foundation.

Concepts such as VRAM consumption, quantization, context length, chat templates, retrieval systems, serving engines, and model selection are ultimately consequences of the mechanics governing token generation. Understanding the loop first makes these later topics easier to reason about because they stop appearing as isolated technical concepts and instead become practical implications of how inference works.

This chapter focuses on the model side of LLMs. It introduces inference, tokens, tokenization, context windows, model weights, attention, KV caching, prefill, decode, and next-token prediction. The goal is to build intuition for how modern language models operate before moving on to deployment, hardware, and software considerations.

# Chapter 1: The LLM Inference Loop

## Learning Objectives

By the end of this chapter, readers will be able to:

* Explain the fundamental inference loop used by decoder-only language models.
* Understand the role of tokens during language generation.
* Distinguish between model weights, context, and KV cache.
* Describe how next-token prediction works.
* Explain the relationship between logits, probabilities, and decoding.
* Understand why generation speed is measured in tokens per second.
* Recognize the practical impact of context windows and token counts.
* Understand why tokenization influences memory usage, latency, and model behavior.

## 1.1 Start With the Loop

The most useful way to understand an LLM is to begin with the generation loop itself.

Many people imagine that a model receives a prompt, internally constructs a complete answer, and then reveals that answer to the user. This is not how a decoder-only language model operates.

Instead, generation occurs incrementally.

The model repeatedly predicts a single token. Once that token has been selected, it becomes part of the sequence. The model then predicts another token based on the updated sequence. This process continues until generation stops.

The loop can be described as:

1. Convert text into tokens.
2. Feed tokens into the model.
3. Process the sequence through the Transformer.
4. Use attention to determine which previous tokens are relevant.
5. Compute scores for possible next tokens.
6. Apply a decoding strategy.
7. Select one token.
8. Append the token to the sequence.
9. Update the KV cache.
10. Repeat.

Every response generated by a language model—from a single word to an entire book chapter—is ultimately produced through repeated execution of this loop.

Understanding this process provides the conceptual foundation for everything that follows.

## 1.2 What an LLM Actually Does

Running a model is called "Inference".

Inference is the process of using a trained model to generate predictions from an input sequence.

For a decoder-only language model, inference consists of repeatedly predicting the most appropriate next token based on the tokens already present in the sequence.

This process continues until one of several stopping conditions occurs:

* The model generates a stopping token.
* A predefined token limit is reached.
* The user manually stops generation.

A critical point is that the model is never generating an entire answer simultaneously.

Instead, every new token becomes part of the context that influences future predictions.

As a result, generation is inherently sequential.

The current state of the sequence determines what can be generated next.

This sequential behavior explains why language models can adapt their output continuously as generation progresses.

## 1.3 Weights, Context, and Working Memory

Three concepts are central to understanding inference:

### Model Weights

Model weights contain the "Patterns learned" during training.

They represent the model's accumulated knowledge about language, structure, formatting conventions, and relationships between tokens.

During inference, the weights remain fixed.

The model is not learning from the prompt while generating text. Instead, it is applying the patterns already encoded within its parameters.

Weights determine what the model has learned.

### Context

Context is the information currently visible to the model.

This includes:

* User prompts
* Conversation history
* System instructions
* Previously generated tokens

The context changes continuously throughout generation because each new token is appended to the sequence.

Context determines what the model is currently looking at.

### KV Cache

The Key-Value cache acts as a form of working memory during inference.

Without a KV cache, the model would need to recompute information about all previously processed tokens every time a new token was generated.

Such repeated computation would be inefficient and would dramatically reduce generation speed.

The KV cache stores previously computed information so that subsequent generation steps can reuse it.

This makes long conversations and long-form generation practical.

The weights provide learned knowledge.

The context provides current information.

The KV cache enables efficient continuation.

Together, these components make modern inference feasible.

## 1.4 The Mathematical View

An LLM can be viewed as a learned function.

Its purpose is to map a sequence of tokens to a probability distribution over possible next tokens.

The relationship can be expressed as:

f(theta, sequence) → probability distribution over next token

Where:

* theta represents the model weights.
* sequence represents the prompt and all generated tokens so far.

The output of the model is not text.

The output is a set of scores describing the likelihood of potential next tokens.

Text generation emerges because this prediction process is repeated many times.

Understanding this distinction is important because it highlights what the model is fundamentally doing: predicting what token should come next.

## 1.5 Logits, Probabilities, and Decoding

When the model evaluates a sequence, it produces raw numerical scores for every token in its vocabulary.

These raw scores are known as logits.

Logits are not probabilities.

They represent the model's relative preferences among possible next tokens.

A normalization process converts these logits into probabilities.

The resulting probability distribution indicates how likely each candidate token is relative to all others.

However, probabilities alone do not generate text.

A decoding process must select a specific token.

Decoding transforms a probability distribution into an actual token choice.

Once selected, that token is appended to the sequence, becoming part of the context for the next prediction.

Generation therefore consists of a cycle of prediction and selection.

The model predicts possibilities.

Decoding chooses one possibility.

The loop then continues.

## 1.6 Prefill and Decode

From a user's perspective, generation speed is often experienced in two distinct phases.

### Prefill

Before the first generated token appears, the model must process the entire prompt.

This phase is known as prefill.

During prefill, the model converts the prompt into internal representations and builds the information needed for subsequent generation.

Long prompts increase the amount of work required during this stage.

As a result, users may experience a noticeable pause before output begins.

A document containing thousands of tokens can produce substantial prefill latency even before the model generates its first word.

### Decode

After prompt processing is complete, the model enters the decode phase.

During decode, the model repeatedly predicts and generates one token at a time.

Each generation step updates the KV cache and extends the sequence.

Because decode controls how quickly text appears on screen, users often perceive decode speed as responsiveness.

However, prefill latency can be equally important when working with large prompts.

Understanding the distinction helps explain why long documents may feel slow even on systems that generate tokens rapidly once decoding begins.

## 1.7 Tokens

Language models do not operate directly on human-readable words.

They operate on tokens.

A token is a unit of text represented internally by an integer identifier.

Tokens can represent many different types of textual content.

Examples include:

* Entire words
* Word fragments
* Punctuation symbols
* Whitespace-prefixed text
* Byte-level fallback representations
* Special control markers

A token might correspond to a complete word such as "hello."

It might also correspond to only part of a word.

Long or uncommon words are frequently divided into multiple tokens.

Consequently, words and tokens should never be assumed to be equivalent.

The distinction is fundamental because every major resource consumed by an LLM is measured in tokens rather than words.

## 1.8 Tokenization

Tokenization is the process of converting text into tokens and converting tokens back into text.

The tokenizer serves as the translation layer between human language and the model's internal representation.

Different model families use different tokenizers.

Common approaches include:

* BPE-style tokenizers
* SentencePiece-style tokenizers

Because tokenizers differ, identical text may produce different token counts across different models.

A document that consumes one amount of context in one model may consume significantly more context in another.

This difference affects practical considerations such as memory usage, prompt length, context consumption, and latency.

Tokenization is therefore not merely a preprocessing step. It directly influences how efficiently a model can represent information.

## 1.9 Vocabulary Size

Every tokenizer operates using a vocabulary.

The vocabulary contains all token entries recognized by the tokenizer.

Vocabulary size affects how text is represented.

A larger vocabulary may allow some text patterns to be represented using fewer tokens.

However, vocabulary size also influences other parts of the model, including embedding and output projection structures.

This means vocabulary design involves trade-offs rather than simple improvements.

It also explains why comparing tokens-per-second performance across different model families can be misleading.

Different tokenizers may represent the same text with different numbers of tokens, making direct comparisons imperfect.

## 1.10 Why Tokens Matter

Tokens are the fundamental unit of work during inference.

They determine:

* Context window consumption
* KV cache growth
* Prompt processing cost
* Memory requirements
* Latency
* Efficiency for code and multilingual text
* Correct interpretation of chat formatting markers

Once text is viewed as tokens rather than words, many aspects of LLM behavior become easier to reason about.

Long context stops appearing magical and instead becomes a measurable computational cost.

Every additional token contributes to the work the model must perform.

## 1.11 Context Windows

A context window defines the maximum number of tokens the model can attend to simultaneously.

Modern models support a wide range of context lengths.

Examples include:

* 8K tokens
* 32K tokens
* 128K tokens
* 256K tokens
* Larger server-oriented configurations

However, supported context length should not be confused with practical context length.

A model may technically accept a large context while becoming slower as that limit is approached.

Similarly, coherence may degrade at extreme sequence lengths.

Therefore, context support should be evaluated in terms of actual workloads rather than advertised limits alone.

A context window is not free.

It consumes memory, increases computation, and can influence generation behavior.

Understanding token costs makes context windows easier to evaluate realistically.

## Practical Example

Consider a user submitting a large document for analysis.

The text is first converted into tokens.

The model processes those tokens during prefill.

Internal state is established and the KV cache is populated.

The model then computes logits for possible next tokens.

These logits are transformed into probabilities.

A decoding strategy selects a token.

The selected token is appended to the sequence.

The KV cache is updated.

The model repeats the process.

What appears to the user as a streaming response is actually a sequence of repeated next-token predictions occurring one token at a time.

## Common Misunderstandings

A common misconception is that an LLM writes an entire answer before displaying it. In reality, generation occurs one token at a time.

Another misunderstanding is that tokens are equivalent to words. Tokenization frequently divides words into smaller components, making token counts and word counts fundamentally different measurements.

Many users also assume that large context windows guarantee strong performance throughout the entire range. In practice, supported context length does not necessarily imply equal speed, cost, or quality at every size.

It is also common to think that responsiveness is determined only by generation speed. Prompt processing time can be equally important, particularly when working with long documents.

Finally, the KV cache is often mistaken for stored conversation text. It is actually a storage mechanism for previously computed information that allows efficient continuation of generation.

## Key Takeaways

Modern decoder-only language models operate through a repeated next-token prediction loop.

Text becomes tokens. Tokens are processed by the Transformer. Attention determines relevant context. The KV cache preserves previously computed information. The model predicts a next token, appends it to the sequence, and repeats the process.

Weights determine what the model has learned.

Context determines what it is currently examining.

The KV cache enables efficient inference.

Tokens are the fundamental unit of work, influencing memory usage, latency, context length, and generation cost.

Understanding these mechanics provides the foundation required for reasoning about all higher-level aspects of language model behavior and deployment.

## 1.13 Transformers

Most modern Large Language Models are based on the Transformer architecture. While users typically interact with models through chat interfaces and observe only the generated text, the actual computational machinery responsible for producing those outputs is the Transformer itself.

Most local chat-oriented language models are decoder-only Transformers. A decoder-only Transformer generates text by repeatedly predicting the next token while looking back at tokens that have already appeared in the sequence. Future tokens do not yet exist, so the model can only use information that is already present within the context window.

Everything discussed earlier in this chapter—including tokens, tokenization, model weights, context windows, KV caches, configuration settings, and chat templates—exists to support the computational engine underneath. Those concepts determine what information enters the model and how that information is represented. The Transformer is the system that processes those representations and converts them into predictions.

A useful way to think about the Transformer is as a stack of processing layers. Each layer receives a representation of the sequence, transforms that representation, and passes the result to the next layer. Through repeated transformation, the model gradually builds increasingly useful internal representations that can ultimately be used for next-token prediction.

A simplified Transformer layer contains several major components:

* Token embeddings
* Positional information
* Self-attention
* Feed-forward computation
* Layer normalization
* Residual connections
* Output projection

The relationship between these components can be visualized as follows.

### Token Embeddings

The first step inside the Transformer is the embedding stage.

Language models do not operate directly on token IDs. A token ID is merely an integer identifier. Before meaningful computation can occur, each token ID must be converted into a vector representation.

Embeddings perform this conversion. They transform discrete token identifiers into dense numerical representations that can participate in the mathematical operations performed throughout the network.

Embeddings therefore act as the bridge between symbolic language and numerical computation.

### Positional Information

Token embeddings alone do not contain information about order.

A sequence such as:

> The dog chased the cat

contains many of the same tokens as:

> The cat chased the dog

but the meanings differ because the token order differs.

The Transformer therefore requires positional information.

Many modern language models use Rotary Position Embeddings (RoPE). RoPE encodes position by rotating internal representations according to token location within the sequence. This allows the model to distinguish between tokens appearing at different positions while preserving useful relationships between them.

Without positional information, the model would struggle to understand sequence structure.

### Self-Attention

Once tokens have been embedded and positional information has been incorporated, the model applies self-attention.

Self-attention allows each token representation to examine previous token representations and determine which information is relevant.

Not every earlier token contributes equally to the next prediction. Some tokens may be highly relevant while others contribute very little. Self-attention allows the model to assign different levels of importance to different parts of the sequence.

This mechanism is one of the primary reasons Transformers can maintain coherence across long passages of text.

### Feed-Forward Computation (MLP)

After attention processing, the representation passes through a feed-forward network, often referred to as an MLP.

The MLP performs dense nonlinear computation that transforms the representation generated by attention.

A useful intuition is that attention determines what information matters, while the MLP reshapes that information into a more useful form.

A large fraction of a model's parameters often reside within these feed-forward layers. As models become larger, these layers account for a substantial portion of the total parameter count.

### Layer Normalization and Residual Connections

Modern language models may contain dozens or even hundreds of Transformer layers.

To maintain stable computation across such depth, Transformers use layer normalization and residual connections.

Layer normalization helps stabilize numerical values as they flow through the network.

Residual connections allow information from earlier stages to continue influencing later stages rather than being completely replaced.

Together, these mechanisms help information propagate effectively through deep networks.

### Output Projection

After the sequence has passed through the Transformer stack, the final hidden representation must be converted into next-token scores.

The output projection layer performs this conversion.

The result is a set of logits over the model vocabulary. These logits later become probabilities, which are then used by the decoding process discussed earlier in this chapter.

### Transformer Recap

The Transformer can be summarized as a sequence of repeated operations.

Tokens become vector representations through embeddings. Positional information keeps token order intact. Attention connects relevant parts of the sequence. Feed-forward networks reshape representations. Layer normalization and residual pathways stabilize computation. Output projection converts the final hidden state into logits over the vocabulary.

## 1.14 Attention

Attention is the mechanism that allows a token to determine which earlier tokens matter for the next prediction.

It is also one of the primary reasons local inference can become memory-sensitive.

When a model processes a sequence, it must preserve information about previous tokens so that future tokens can attend to them. As context length grows, the amount of information that must be stored and accessed can increase substantially.

For local inference workloads, attention design therefore has a direct impact on memory usage, context length behavior, and KV cache growth.

### Figure 1.10 — Conceptual View of Self-Attention

```text
Sequence

[The] [dog] [chased] [the] [cat]

                         ▲
                         │
                  Current Token

                         │
         ┌───────────────┼───────────────┐
         │               │               │
         ▼               ▼               ▼

      [dog]         [chased]         [the]

      High           Medium           Low
    Importance     Importance      Importance
```

The diagram above illustrates a simplified view of attention. The current token does not treat every previous token equally. Instead, it determines which earlier information is most relevant to the prediction task.

### Multi-Head Attention (MHA)

Classic Multi-Head Attention (MHA) stores separate key-value state for multiple attention heads.

Each head can learn different relationships within the sequence and focus on different patterns of information. This flexibility makes MHA highly expressive.

However, maintaining separate key-value information for many heads causes the KV cache to grow rapidly. As context length increases, memory requirements can become substantial.

This is one reason why long-context inference can become expensive.

### Multi-Query Attention (MQA)

Multi-Query Attention (MQA) reduces memory usage by allowing multiple query heads to share a single key-value head.

This design significantly reduces KV cache growth and improves memory efficiency.

The trade-off is that MQA may be less expressive than full Multi-Head Attention because fewer independent key-value representations are available.

### Grouped-Query Attention (GQA)

Grouped-Query Attention (GQA) provides a middle ground between MHA and MQA.

Instead of sharing a single key-value head across all queries, query heads are organized into groups. Each group shares key-value information internally.

This approach preserves more flexibility than MQA while consuming less memory than full MHA.

As a result, GQA has become a common design choice in many modern local language models.

### Figure 1.11 — Comparing MHA, MQA, and GQA

```text
MHA

Q1 ──► KV1
Q2 ──► KV2
Q3 ──► KV3
Q4 ──► KV4


MQA

Q1 ──┐
Q2 ──┤
Q3 ──┤──► Shared KV
Q4 ──┘


GQA

Q1 ──┐
Q2 ──┘──► KV Group A

Q3 ──┐
Q4 ──┘──► KV Group B
```

This comparison highlights the trade-off between flexibility and memory efficiency. MHA provides maximum flexibility but requires the largest KV cache. MQA minimizes memory consumption through complete sharing. GQA balances the two approaches and has therefore become popular in many modern local models.

### Attention Kernels and Runtime Performance

Attention architecture is only part of the performance story.

Modern runtimes often use optimized attention kernels such as FlashAttention and SDPA-style implementations. These kernels reduce attention memory traffic and improve hardware utilization.

Reducing memory movement is important because memory bandwidth can become a major bottleneck during inference.

As a result, two systems running the same model on identical hardware can exhibit dramatically different performance characteristics depending on the quality of their attention implementations.

A runtime with highly optimized attention kernels may achieve significantly better performance than one without such optimizations.

### Why Attention Design Matters

Attention design has practical consequences beyond architecture diagrams.

Two models may both contain seven billion parameters and advertise identical context lengths, yet behave very differently during long-context inference.

A 7B model using full Multi-Head Attention at 128K context may consume enough memory to exhaust a 24 GB GPU.

A 7B model using Grouped-Query Attention with the same advertised context length may fit comfortably on the same hardware.

This illustrates an important principle: parameter count is not the entire story.

When comparing models, attention type, KV head configuration, supported context length, and runtime support are often just as important as model size itself.

When evaluating local language models, attention architecture should therefore be considered alongside parameter count rather than after it.

## 1.15 KV Cache

The KV cache is the model's working memory during generation.

Earlier in this chapter, we discussed how a decoder-only language model generates text one token at a time. Every newly generated token must be able to attend to previous tokens in the sequence. If the model were forced to recompute the attention state for the entire conversation every time it generated a new token, inference would become prohibitively expensive.

The KV cache exists to prevent this.

Rather than recomputing attention information from scratch for every generation step, the model stores key and value attention states associated with previously processed tokens. These stored states can then be reused when generating future tokens.

The result is a dramatic reduction in computational work during generation.

Without a KV cache, modern language model inference would be extremely inefficient. With a KV cache, generation becomes practical. However, this efficiency comes with a cost: memory consumption.

The size of the KV cache grows as more tokens enter the context window. A useful approximation for KV cache memory consumption is:

```text
KV Cache Size ∝

tokens × layers × kv_heads × head_dim × precision × 2
```

The final multiplier exists because the cache stores both keys and values.

Each additional token increases the amount of cached information. As conversations grow longer, the cache continues to grow alongside them.

### Figure 1.12 — KV Cache Growth

```text
Conversation Length
        │
        │
        │
        │
        ▼

1K Tokens   → Small KV Cache

4K Tokens   → Larger KV Cache

16K Tokens  → Much Larger KV Cache

32K Tokens  → Very Large KV Cache

64K Tokens  → Extremely Large KV Cache
```

The exact size depends on model architecture, attention design, precision format, and runtime implementation, but the overall relationship remains the same: more tokens require more cache memory.

### KV Cache in Practice

A useful rule of thumb for older Llama-style 7B models using Multi-Head Attention and FP16 KV storage is approximately:

```text
~0.5 MiB per token
```

Under this approximation:

```text
4K Tokens  ≈ 2 GiB KV Cache

32K Tokens ≈ 16 GiB KV Cache
```

These figures illustrate why context length can become a practical deployment constraint.

A model may fit comfortably into memory when idle, yet encounter memory pressure once large documents or long conversations are loaded.

This is because model weights are only one part of the memory budget.

The KV cache consumes memory independently of the weights and continues growing as context length increases.

### GQA and MQA Effects on KV Cache Size

Attention architecture significantly influences cache growth.

As discussed in the previous section:

* MHA stores separate key-value state for many heads.
* MQA allows query heads to share a single key-value representation.
* GQA organizes heads into shared groups.

Because MQA and GQA reduce the amount of stored key-value information, they can substantially reduce KV cache requirements.

This is one reason modern local models often prefer GQA or MQA-style architectures for long-context workloads.

The model may have the same parameter count as an older architecture while consuming significantly less cache memory.

### KV Cache Precision

The precision used to store cache values also influences memory consumption.

Many modern runtimes support:

* FP16 KV cache
* FP8 KV cache
* INT8 KV cache

For local users, FP8 and INT8 KV storage often provide substantial memory savings while maintaining practical performance.

The source material recommends treating approximately 8-bit KV storage as a practical compression floor for many local workloads in 2026.

### Sub-8-Bit KV Cache

Reducing KV cache precision below 8 bits is an active area of research.

Examples referenced in the source include:

* [KIVI Paper](https://arxiv.org/abs/2402.02750?utm_source=chatgpt.com)
* [KVQuant Paper](https://arxiv.org/abs/2401.18079?utm_source=chatgpt.com)

These systems demonstrate that 2-bit to 4-bit KV storage can function effectively when combined with specialized algorithms, calibration procedures, and custom kernels.

However, research results should not be confused with a simple runtime setting.

A carefully engineered research implementation is very different from enabling an aggressively compressed KV cache without validation.

For workloads where exact token recall matters—such as coding, tool usage, JSON generation, long-context retrieval, or structured outputs—sub-8-bit cache configurations should be benchmarked carefully before deployment.

### KV Cache Quantization vs Speculative Decoding

Another common source of confusion is the relationship between KV cache compression and speculative decoding systems.

These technologies solve different problems.

KV cache quantization reduces memory consumption.

Speculative decoding attempts to improve generation speed.

Examples referenced in the source include:

* [DFlash Paper](https://arxiv.org/abs/2602.06036?utm_source=chatgpt.com)
* [DDTree Paper](https://arxiv.org/abs/2604.12989?utm_source=chatgpt.com)

These approaches accelerate generation by drafting future tokens and verifying them.

They can reduce decode latency, but they do not eliminate the memory requirements associated with the KV cache.

The cache must still exist regardless of how efficiently tokens are generated.

### Why Models Sometimes Crash on Long Documents

A common misunderstanding among local users occurs when a model loads successfully but crashes after a large document is added.

The immediate assumption is often that the model itself does not fit into memory.

In many cases, the model weights fit perfectly.

The problem is that the working memory does not.

As context length grows, the KV cache expands. Eventually, the combined memory requirements of the weights and cache exceed available resources.

The weights fit.

The working memory does not.

## 1.16 Prefill and Decode

Language model inference operates in two distinct performance regimes:

* Prefill
* Decode

Although users often experience them as part of a single interaction, they represent fundamentally different types of computation.

Understanding this distinction helps explain why some workloads feel slow before generation begins while others feel slow during generation.

### Figure 1.13 — Prefill and Decode

```text
User Prompt
      │
      ▼

┌─────────────┐
│   Prefill   │
└─────────────┘

(Process Existing Tokens)

      │
      ▼

First Generated Token

      │
      ▼

┌─────────────┐
│   Decode    │
└─────────────┘

(Token-by-Token Generation)

      │
      ▼

Final Response
```

### Prefill

Prefill is the stage in which the model processes the prompt supplied by the user.

If a user submits a large document, every token within that document must be processed before generation can begin.

For example:

```text
20,000-token document
        ↓
20,000 tokens processed
        ↓
First output token appears
```

The time spent waiting for the first generated token is often dominated by prefill.

Although prefill can be computationally expensive, it benefits from substantial parallelism. Modern GPUs can process many prompt tokens simultaneously, making prefill relatively efficient compared with purely sequential workloads.

Nevertheless, very large prompts can still create noticeable latency.

### Decode

Decode begins after prompt processing is complete.

During decode, the model generates new tokens one at a time.

Each generated token becomes part of the sequence and influences future generation.

Because every new token depends on previously generated tokens, decode is inherently sequential.

This is the phase responsible for the familiar streaming effect in chat applications.

The speed of decode often determines whether a model feels responsive during generation.

### Long Prompts vs Long Answers

Different workloads stress different parts of the inference pipeline.

Long prompts primarily increase prefill cost.

Long responses primarily increase decode cost.

Long conversations increase both.

This relationship can be summarized as:

```text
Long Prompt      → Prefill Cost

Long Answer      → Decode Cost

Long Conversation
      ↓
Prefill + Decode + KV Growth
```

### Conversation Growth and Cache Growth

In a chat application, every turn contributes additional tokens to the conversation history.

As history grows, the KV cache grows with it.

A conversation containing 16K tokens carries the memory cost of those 16K tokens throughout subsequent generation.

This is one reason long-running conversations eventually slow down or exhaust memory resources.

Infinite chat history is not free.

The cache continues growing until memory becomes a limiting factor.

---

## 1.17 Decoding

After the model produces logits, text has not yet been generated.

The model has only assigned scores to possible next tokens.

Decoding is the process that converts those scores into an actual token selection.

Once a token is selected, it is appended to the context and the inference loop repeats.

### Figure 1.14 — Decoding Loop

```text
Logits
   │
   ▼

Probabilities
   │
   ▼

Decoding Policy
   │
   ▼

Selected Token
   │
   ▼

Append to Context
   │
   ▼

Repeat
```

The decoding process does not modify model weights.

Instead, it determines how the model's predictions are translated into generated text.

Different decoding strategies can significantly influence output style and behavior.

### What Decoding Controls

Decoding settings answer three practical questions:

#### Randomness

How much variation should be allowed?

Higher randomness permits more diverse outputs.

Lower randomness encourages consistency and predictability.

#### Tail Reach

How far into lower-probability tokens should the sampler be allowed to explore?

Restricting exploration produces safer and more focused outputs.

Allowing deeper exploration can increase creativity and diversity.

#### Boundaries

What mechanisms prevent undesirable behavior?

Examples include:

* Infinite loops
* Excessive repetition
* Schema violations
* Runaway generation

Boundary controls help keep outputs within desired constraints.

### Practical Decoding Approaches

Different tasks often benefit from different decoding behavior.

For precise work:

* Lower temperature
* Shorter generation limits
* Explicit stop sequences
* Constrained decoding

These settings help maintain consistency and structure.

For creative work:

* Higher temperature
* Higher diversity
* Multiple candidate generations

These settings encourage exploration and variety.

For coding tasks:

* Conservative first-pass generation
* Alternative samples when exploring solutions

This approach balances reliability with experimentation.

### Greedy Decoding

A common assumption is that selecting the highest-probability token at every step always produces the best result.

In practice, this is not necessarily true.

Greedy decoding can become brittle.

Because it never explores alternatives, it may become trapped in repetitive patterns or converge on generic responses.

Deterministic settings are often useful for evaluations and reproducibility.

For brainstorming and ideation, allowing controlled exploration can produce more varied and useful outputs.

The appropriate decoding strategy depends on the task being performed rather than a universal notion of correctness.

## 1.18 What a Model Package Contains

A runnable local language model is more than a single weight file.

This is one of the first lessons people learn when moving from hosted APIs to local inference. A beginner often downloads a model file, sees that it occupies several gigabytes of storage, and assumes that file contains everything required to run the model. In reality, modern language models are distributed as packages composed of multiple components that must work together correctly.

A model package typically contains:

* Architecture and configuration
* Weights
* Tokenizer
* Chat template
* Generation configuration
* License and model card

Although the weights are usually the largest component, they are only one piece of the system.

A useful way to think about a model package is to compare it to a computer program.

The executable code is important, but the program also depends on configuration files, libraries, data formats, and documentation. Removing any of those pieces can cause problems even if the executable itself remains intact.

Language models work in a similar way.

The weights contain learned knowledge, but the tokenizer determines how text is represented, the configuration describes how the model is constructed, and the chat template determines how conversations are formatted.

If any of these components are incorrect, the same weights can appear broken.

### Figure 1.15 — Components of a Model Package

```text
Model Package
      │
      ├── Architecture / Config
      ├── Weights
      ├── Tokenizer
      ├── Chat Template
      ├── Generation Config
      └── License / Model Card
```

A model package should therefore be viewed as a collection of components that travel together rather than a single file.

---

## 1.19 Architecture and Configuration

The architecture configuration describes how the model is built.

If the weights contain learned knowledge, the architecture defines the structure that uses that knowledge.

An important distinction is that architecture and weights are not the same thing.

A useful mental model is:

```text
Model
=
Architecture
+
Weights
```

The architecture specifies:

* How many layers exist
* How large internal representations are
* Which attention mechanism is used
* How positional information is encoded
* How many tokens exist in the vocabulary
* Which special tokens are reserved
* How much context the model supports

Without this information, a runtime would not know how to interpret the weight tensors stored within the model.

### Layer Count

A Transformer is composed of layers stacked on top of one another.

Each layer receives information from the previous layer, transforms that information, and passes the result forward.

A simplified view looks like:

```text
Input
  │
Layer 1
  │
Layer 2
  │
Layer 3
  │
 ...
  │
Layer N
  │
Output
```

The number of layers is known as the layer count.

Increasing the layer count allows information to undergo more transformations before a prediction is produced.

More layers generally allow more sophisticated processing, but they also increase:

* Memory consumption
* Computational cost
* Inference latency

Layer count is therefore one of the primary architectural characteristics used to distinguish model families.

---

### Hidden Size

Earlier in the chapter, token embeddings were introduced as vector representations.

Those vectors contain many numerical dimensions.

For example:

```text
"Paris"

↓

[0.12, -0.81, 1.54, 0.39, ...]
```

The number of dimensions in these vectors is known as the hidden size.

The hidden size determines how much information can be represented inside the model at each layer.

A useful analogy is to imagine hidden size as the width of a highway.

A wider highway allows more information to travel simultaneously.

A narrower highway limits how much information can be represented at once.

Larger hidden sizes generally provide greater representational capacity, but they also require more memory and computation.

This is one reason large models become expensive to run.

---

### Attention Type

The attention architecture determines how attention state is stored and reused.

Common attention types include:

* Multi-Head Attention (MHA)
* Multi-Query Attention (MQA)
* Grouped-Query Attention (GQA)

As discussed previously, attention architecture strongly influences:

* KV cache size
* Memory requirements
* Long-context performance
* Runtime efficiency

Two models with identical parameter counts may exhibit very different memory behavior if they use different attention mechanisms.

This is why attention type is part of the model configuration.

---

### RoPE Settings

RoPE stands for Rotary Position Embeddings.

RoPE provides positional information that tells the model where tokens appear within a sequence.

Consider the sentences:

```text
The dog chased the cat

The cat chased the dog
```

Both contain many of the same words, yet their meanings differ because the order differs.

The model therefore needs a way to encode position.

RoPE accomplishes this by rotating internal representations according to token position.

Conceptually:

```text
Position 1 → Rotate by θ

Position 2 → Rotate by 2θ

Position 3 → Rotate by 3θ
```

The exact mathematics involve trigonometric rotations applied within vector space, but the key idea is simple: RoPE allows the model to distinguish tokens that appear in different positions.

RoPE settings influence:

* Context length behavior
* Long-context scaling
* Positional representation

As context windows grow larger, RoPE configuration becomes increasingly important.

---

### Vocabulary Size

The tokenizer operates using a vocabulary.

The vocabulary contains every token recognized by the model.

Examples include:

* Whole words
* Word fragments
* Punctuation
* Special markers
* Control tokens

Vocabulary size affects several architectural components.

One important relationship is:

```text
Vocabulary Size
        ×
Hidden Size
        =
Embedding Table Size
```

A larger vocabulary may represent some text more efficiently because more token entries are available.

However, larger vocabularies also require larger embedding structures.

Vocabulary design therefore involves trade-offs rather than universally better choices.

---

### Special Tokens

Special tokens are tokens reserved for structural purposes.

Unlike ordinary text, they carry predefined meaning.

Examples include:

* Beginning-of-sequence markers
* End-of-sequence markers
* System-role markers
* User-role markers
* Assistant-role markers
* Tool-use markers

Special tokens help the model understand conversation structure and generation boundaries.

Later sections will examine these tokens in detail.

---

### Context Length

Context length specifies the maximum number of tokens that can participate in attention simultaneously.

Examples include:

```text
8K
32K
128K
256K
```

Context length is an architectural capability.

However, supporting a context length and using it efficiently are different things.

A model may technically support very large contexts while experiencing reduced speed or increased memory consumption at those lengths.

---

## 1.20 Weights

The weights are the learned parameters of the model.

When people download a model, they are usually downloading billions of numerical values that were learned during training.

These values are called weights because they influence how information flows through the network.

A useful mental model is:

```text
Architecture
=
Skeleton

Weights
=
Knowledge
```

The architecture determines how computation occurs.

The weights determine what the model has learned.

Both are required.

Without weights, the architecture is merely an empty structure.

Without architecture, the weights cannot be interpreted.

### What Is a Weight?

At its most basic level, a weight is a number.

Consider a simplified computation:

```text
Input
  ×
Weight
  =
Contribution
```

A neural network performs enormous numbers of operations similar to this.

Individual weights influence how strongly information flows between representations.

During training, these values are adjusted repeatedly until the network becomes capable of making useful predictions.

The final collection of learned values becomes the model's weights.

---

### Why Weights Store Knowledge

A common misconception is that model weights contain stored documents, sentences, or facts.

They do not.

Weights are not text files.

They are not databases.

They are not lookup tables.

Instead, they are learned numerical parameters that encode statistical patterns observed during training.

Knowledge emerges from the collective behavior of billions of weights interacting with one another.

This is why inspecting individual weights is generally not useful. Meaning exists in the network as a whole rather than in isolated values.

---

### The Scale of Modern Weights

Modern language models contain enormous numbers of parameters.

Examples include:

```text
2 Billion Parameters

7 Billion Parameters

14 Billion Parameters

70 Billion Parameters
```

Each parameter is a learned numerical value.

As parameter counts increase, the amount of stored information generally increases as well.

However, larger parameter counts also increase:

* Storage requirements
* Memory usage
* Computational cost

Parameter count is therefore an important characteristic, but it is not the only factor determining model quality.

---

### Architecture Versus Weights

One of the most important concepts for local inference is the distinction between architecture and weights.

Consider the following:

```text
Same Architecture
       +
Different Weights
       =
Different Model
```

The architecture may remain unchanged while entirely different weights are loaded.

In such cases, the resulting model can behave very differently despite sharing the same structural design.

This distinction becomes important when downloading models, applying fine-tunes, or converting between formats.

---

## 1.21 Weight Storage Formats

The source material mentions several common formats used to distribute model weights:

* safetensors
* GGUF
* GPTQ
* AWQ
* EXL2

Although all contain model parameters, they are designed for different purposes.

### safetensors

Safetensors is a weight storage format widely used throughout the Hugging Face ecosystem.

Its purpose is straightforward:

* Store model parameters efficiently.
* Load parameters safely.
* Avoid security issues associated with executable serialization formats.

Many modern model releases provide safetensors files as their primary distribution format.

When using Transformers-based workflows, safetensors is one of the most common formats encountered.

---

### GGUF

GGUF is a format commonly associated with llama.cpp and related local inference tools.

Unlike a simple tensor container, GGUF typically stores:

* Weights
* Quantization metadata
* Configuration information
* Runtime-relevant metadata

This makes GGUF particularly convenient for local deployment.

Many users choose GGUF because it allows a model to be packaged into a format that local runtimes can load directly.

---

### GPTQ

GPTQ is a quantized weight format.

Quantization reduces memory requirements by representing weights using fewer bits.

The objective is to make large models easier to run on limited hardware.

A simplified relationship is:

```text
Lower Precision
        ↓
Smaller Model
        ↓
Lower Memory Usage
```

The trade-off is that aggressive compression can sometimes reduce model quality.

GPTQ attempts to balance compression and performance.

---

### AWQ

AWQ stands for Activation-Aware Weight Quantization.

Like GPTQ, it is designed to reduce memory usage.

However, AWQ uses a different quantization strategy intended to preserve model quality while compressing weights.

The goal is not merely to make weights smaller, but to do so while maintaining useful behavior.

---

### EXL2

EXL2 is a format commonly associated with ExLlama-style runtimes.

It is designed for efficient GPU inference and aggressive compression.

EXL2 is often used by local users seeking high performance from limited hardware resources.

As with other quantized formats, the goal is to reduce memory requirements while preserving practical model quality.

---

### Why Multiple Formats Exist

A common question is:

> Why are there so many formats?

The answer is that different formats optimize for different goals.

Some prioritize compatibility.

Some prioritize safety.

Some prioritize memory efficiency.

Some prioritize inference speed.

As a result, the same model may be available in several formats depending on the runtime and hardware being targeted.

Understanding these formats helps explain why downloading a model is often only the first step. The format must also be compatible with the runtime that will execute it.

## 1.22 Tokenizers

Earlier in this handbook, tokenization was introduced as the process that converts text into tokens and tokens back into text. At first glance, tokenization may appear to be a simple preprocessing step. In reality, the tokenizer is one of the most important components of a language model.

A model never sees raw text.

When a user types:

```text id="9ph6xl"
Explain KV cache.
```

the model does not directly process those characters.

Instead, the tokenizer converts the text into a sequence of token identifiers.

Conceptually:

```text id="ph4nce"
Text
  │
  ▼
Tokenizer
  │
  ▼
Token IDs
  │
  ▼
Model
```

Every operation performed by the Transformer occurs on token representations rather than on raw text.

For this reason, the tokenizer should be viewed as part of the model rather than a separate utility.

A model trained with one tokenizer cannot simply be paired with a completely different tokenizer. The mapping between text and token IDs must remain consistent with the mapping used during training.

---

### Why Tokenizers Matter

Tokenizers influence:

* Context length usage
* Memory consumption
* Prompt processing cost
* Vocabulary size
* Multilingual efficiency
* Code generation efficiency

Consider the word:

```text id="bjlwm8"
internationalization
```

One tokenizer might represent this as:

```text id="xh8zsm"
international
ization
```

while another tokenizer might represent it as:

```text id="9nqz5w"
inter
national
ization
```

The resulting token count differs.

Because memory, latency, context windows, and KV cache growth are measured in tokens rather than words, tokenization directly affects practical performance.

---

### Token IDs

Every token has a numerical identifier.

For example:

```text id="ybp1k8"
"hello"
   ↓
15339
```

The actual values vary by tokenizer, but the principle remains the same.

The Transformer does not process words.

It processes token IDs.

Those token IDs are then converted into embeddings before entering the model.

Conceptually:

```text id="d7s1zi"
Text

"hello"

      ↓

Token ID

15339

      ↓

Embedding Vector

[0.12, -0.84, ...]
```

This conversion process forms the bridge between language and numerical computation.

---

### Vocabulary and Token IDs

The tokenizer vocabulary contains every token recognized by the model.

A simplified vocabulary might look like:

```text id="u7xlru"
0  → [PAD]
1  → [BOS]
2  → [EOS]
3  → hello
4  → world
5  → .
...
```

Real vocabularies often contain tens or hundreds of thousands of entries.

The vocabulary and tokenizer are tightly coupled.

Changing the vocabulary changes the meaning of token IDs.

As a result, tokenizers must remain compatible with the models that use them.

---

### Tokenizers Are Part of the Model

A common beginner mistake is to think:

> "The model is the weights."

The weights are important, but the tokenizer is equally critical.

The following combination does not work:

```text id="1ee1ga"
Model A Weights
+
Tokenizer B
```

unless both were designed to operate together.

The tokenizer determines how text is represented internally.

If the representation changes, the model may receive inputs it was never trained to interpret.

---

## 1.23 Chat Templates

Once text has been tokenized, the model still needs to understand the structure of a conversation.

This is where chat templates become important.

A chat template is a formatting scheme that tells the model:

* Which text is a system instruction
* Which text comes from the user
* Which text comes from the assistant
* Where responses begin
* Where responses end

A useful way to think about chat templates is as communication protocols.

Humans naturally understand conversational roles.

Language models do not.

Those roles must be explicitly represented in the sequence.

---

### Why Chat Templates Exist

Consider the following conversation:

```text id="3lh4ux"
You are a helpful assistant.

Explain KV cache.
```

A human reader immediately understands:

* The first sentence is an instruction.
* The second sentence is a question.

A language model does not automatically know that.

Chat templates provide explicit structure.

For example:

```text id="uvjlwm"
<|system|>
You are a helpful assistant.

<|user|>
Explain KV cache.

<|assistant|>
```

The role markers identify who is speaking.

Without these markers, the model may interpret the conversation differently than intended.

---

### Conversation Structure

Conceptually, a chat template transforms conversation into structured data.

```text id="ol1h2q"
System Message
       │
       ▼
User Message
       │
       ▼
Assistant Response
```

The model learns this structure during training.

When the runtime applies the correct template, the conversation resembles the data distribution the model encountered during training.

---

### Why Templates Affect Performance

The same model can appear intelligent with one template and broken with another.

For example:

```text id="jlwmjk"
Correct Template
      ↓
Expected Behavior
```

versus:

```text id="bf6mwj"
Incorrect Template
      ↓
Confused Behavior
```

Many cases of poor model performance are actually template problems rather than model problems.

---

## 1.24 Understanding ChatML

One of the most common template families is ChatML.

ChatML uses special role markers to distinguish different participants in a conversation.

A typical ChatML conversation looks like:

```text id="v5mzh4"
<|system|>
You are a helpful assistant.

<|user|>
Explain KV cache.

<|assistant|>
```

To a human, these appear to be labels.

To the model, they are special tokens with learned meaning.

The model does not interpret:

```text id="8o7qu5"
<|system|>
```

as ordinary text.

Instead, it interprets the token as a structural marker indicating that the following content belongs to the system role.

The same applies to:

```text id="uw7w84"
<|user|>
```

and

```text id="0lm4kr"
<|assistant|>
```

These markers are part of the language the model was trained to understand.

---

### System Messages

System messages contain high-level instructions.

Example:

```text id="44m8o4"
<|system|>
You are a helpful assistant.
```

The purpose of the system role is to define behavior.

It tells the model how it should approach the conversation.

---

### User Messages

User messages represent requests.

Example:

```text id="vlsb5v"
<|user|>
Explain KV cache.
```

The model learns that text appearing after the user marker represents a request or input from the user.

---

### Assistant Messages

Assistant messages indicate where the model's response begins.

Example:

```text id="jlwmg8"
<|assistant|>
```

When generation starts after this marker, the model understands that it should produce assistant-style text.

---

### ChatML Flow

```text id="krm1rv"
<|system|>
      │
      ▼
<|user|>
      │
      ▼
<|assistant|>
      │
      ▼
Generated Response
```

This structure is one of the most common conversation formats used by modern chat models.

---

## 1.25 BOS and EOS Tokens

Two of the most important special tokens in language models are BOS and EOS.

These tokens define sequence boundaries.

---

### BOS — Beginning of Sequence

BOS stands for Beginning of Sequence.

It marks the start of a token sequence.

Conceptually:

```text id="67vzcw"
[BOS]

Explain KV cache.
```

The model learns that a new sequence begins after this token.

BOS provides an anchor point that tells the model where generation starts.

---

### EOS — End of Sequence

EOS stands for End of Sequence.

It marks the end of a sequence.

Conceptually:

```text id="cw8jwn"
Explain KV cache.

[EOS]
```

The model learns that generation should stop when this token appears.

EOS helps define boundaries and prevents endless continuation.

---

### Why EOS Matters

Suppose the model generates:

```text id="ggm59l"
The KV cache stores attention state.
```

If EOS is generated afterward:

```text id="6wpkrn"
The KV cache stores attention state.

[EOS]
```

generation stops.

Without EOS, generation may continue indefinitely until a separate limit intervenes.

This is one reason why EOS handling is critical.

---

### Sequence Boundary Diagram

```text id="vfz4k8"
[BOS]
   │
User Prompt
   │
Assistant Response
   │
[EOS]
```

These markers provide structure that helps the model understand where sequences begin and end.

---

## 1.26 Template Families

Not all models use the same chat template.

Different model families are trained using different formats.

As a result, one model may expect ChatML while another expects an entirely different structure.

---

### INST Templates

A common example is the INST-style format.

For example:

```text id="s2qifj"
[BOS] [INST]
Explain KV cache.
[/INST]
```

This format uses instruction delimiters rather than explicit role markers.

The model learns that content inside the instruction block represents a user request.

Although the syntax differs, the underlying purpose remains the same: provide structure.

---

### Chat Templates vs Instruction Templates

Conceptually:

```text id="hmv5xw"
ChatML
      ↓
Role-Based Structure
```

versus:

```text id="r5z0m2"
INST
      ↓
Instruction-Based Structure
```

Both approaches provide information about how text should be interpreted.

They simply encode that information differently.

---

### Tool-Oriented Templates

Some models include additional markers for tool use.

Conceptually:

```text id="is0u6x"
System
User
Assistant
Tool
Assistant
```

These templates allow the model to distinguish tool outputs from normal conversation text.

---

### Reasoning-Oriented Templates

Some models introduce additional markers related to reasoning or intermediate computation.

The exact syntax varies between model families.

The key principle remains unchanged:

Special tokens provide structure.

The model learns to associate that structure with specific behaviors.

---

## 1.27 Why Templates Break Models

A common misconception among new local users is that all chat models accept the same prompt format.

They do not.

Consider a model trained on ChatML:

```text id="yuxnkt"
<|system|>
...
<|user|>
...
<|assistant|>
```

If that model receives an INST template instead:

```text id="xvt5eh"
[INST]
...
[/INST]
```

the structure no longer matches the format encountered during training.

The result may be:

* Reduced instruction following
* Strange formatting
* Repetition
* Lower-quality answers
* Unexpected behavior

The weights have not changed.

The tokenizer has not changed.

The hardware has not changed.

Only the template changed.

Yet behavior may change dramatically.

---

### Template Mismatch

A useful mental model is:

```text id="1x9nnv"
Correct Template
        ↓
Expected Behavior
```

```text id="qsm2lr"
Incorrect Template
        ↓
Unexpected Behavior
```

When evaluating a model, always verify:

* The tokenizer is correct.
* The chat template is correct.
* BOS tokens are correct.
* EOS tokens are correct.
* Role markers are correct.

A surprising number of "bad model" reports are actually formatting problems.

The template is effectively part of the model.

If the template is wrong, you are not truly testing the model that was trained.

## 1.28 Model Types

Not all language models are trained for the same purpose.

One of the most common mistakes among newcomers is assuming that every model behaves like a chatbot. In reality, modern language models are often optimized for different tasks, workflows, and interaction styles.

Two models may have identical parameter counts, use the same architecture, and even share a common base model, yet behave very differently because they have been trained for different objectives.

When evaluating a model, one of the first questions to ask is:

> What type of model is this?

Common categories include:

* Base models
* Instruct models
* Chat models
* Reasoning models
* Tool-tuned models

Understanding these categories helps explain why some models excel at conversation, others at coding, and others at structured tool use.

---

### Base Models

A base model is the closest representation of the raw pretrained model.

It has learned language patterns from training data but has not necessarily been optimized for assistant-style interactions.

Conceptually, a base model is a text continuation engine.

If given:

```text id="r3h1kb"
The capital of France is
```

the model attempts to predict what text most naturally follows.

It is not necessarily trying to be helpful.

It is trying to continue the sequence.

For example:

```text id="5qgh2o"
The capital of France is Paris.
It has a population of...
```

The model simply continues generating text.

### Base Model Behavior

A base model generally thinks in terms of completion rather than instruction following.

Conceptually:

```text id="7g0kdi"
Prompt
   ↓
Text Continuation
```

rather than:

```text id="20wr3f"
Prompt
   ↓
Helpful Assistant Response
```

This distinction is important.

A base model may possess substantial knowledge while still producing poor conversational results because it was not specifically optimized for assistant behavior.

---

### Instruct Models

An instruct model is designed to follow instructions more effectively.

During additional training, the model learns patterns associated with requests, commands, and tasks.

Examples include:

```text id="hy73hc"
Summarize this article.

Explain KV cache.

Write a Python function.
```

Rather than simply continuing text, the model learns that it should attempt to satisfy the user's request.

Conceptually:

```text id="v2vwxt"
Instruction
      ↓
Task Completion
```

Instruction tuning is one of the reasons modern language models feel substantially more useful than earlier completion-oriented systems.

---

### Chat Models

Chat models are optimized for conversational interaction.

In addition to instruction following, they learn:

* Conversational turn-taking
* Role separation
* Multi-turn interactions
* Assistant-style responses

A chat model understands structures such as:

```text id="m6mxni"
System
User
Assistant
User
Assistant
```

and learns to maintain conversational consistency across multiple turns.

Because chat models are trained using specific conversation formats, correct chat templates become especially important.

---

### Reasoning Models

Reasoning models are optimized for tasks requiring longer chains of thought, multi-step analysis, or extended problem solving.

Examples include:

* Mathematics
* Complex coding
* Planning
* Logical analysis

Reasoning-oriented models are often designed to spend more computation on difficult problems before producing a final answer.

Conceptually:

```text id="9ll4b5"
Question
    ↓
Extended Internal Reasoning
    ↓
Answer
```

The exact implementation varies between model families, but the goal remains the same: improve performance on tasks requiring multiple reasoning steps.

---

### Tool-Tuned Models

Tool-tuned models are optimized to interact with external tools.

Rather than responding only with text, they learn structured outputs that can trigger external actions.

Examples include:

* Function calls
* JSON schemas
* Tool invocation formats
* API requests

Conceptually:

```text id="n7ob1f"
User Request
      ↓
Tool Selection
      ↓
Tool Output
      ↓
Final Response
```

* Understand the relationship between precision, memory usage, and model size.
* Distinguish between weight quantization and KV-cache quantization.

---

## 1.31.1 Why Models Become So Large

Before discussing quantization, we first need to understand why language models consume so much memory.

A modern language model contains billions of learned numerical values called weights.

Earlier in this handbook we described weights as the learned knowledge of the model.

Conceptually:

```text
Architecture
      +
Weights
      =
Language Model
```

Every weight is stored as a number.

A model containing seven billion parameters contains seven billion numbers.

A model containing seventy billion parameters contains seventy billion numbers.

Those numbers must live somewhere.

They occupy storage on disk.

They occupy RAM when loaded.

They occupy VRAM during GPU inference.

The larger the numbers and the more precision used to represent them, the more memory the model requires.

This immediately raises a question:

> Does every weight need to be stored with maximum precision?

Quantization exists because the answer is often no.

---

## 1.31.2 What Is Numerical Precision?

To understand quantization, we first need to understand precision.

Computers represent numbers using bits.

A bit is the smallest unit of information.

A bit can contain only one of two values:

```text
0

or

1
```

Multiple bits can represent increasingly complex numbers.

For example:

```text
1 Bit
↓

2 Possible Values
```

```text
8 Bits
↓

256 Possible Values
```

```text
16 Bits
↓

65,536 Possible Values
```

As the number of bits increases, the amount of information that can be represented also increases.

A useful mental model is to imagine a ruler.

A ruler with only a few markings can measure approximately.

A ruler with many markings can measure precisely.

Precision works similarly.

More bits allow more precise numerical representation.

---

## 1.31.3 Why Precision Consumes Memory

Suppose we have a single weight.

If that weight is stored using:

```text
16 Bits
```

it consumes twice as much memory as a weight stored using:

```text
8 Bits
```

Likewise:

```text
4 Bits
```

requires only one quarter of the storage used by:

```text
16 Bits
```

Now imagine billions of weights.

Even small reductions in precision create enormous memory savings.

### Figure 1.31 — Precision and Storage

```text
7 Billion Parameters

        │

        ▼

┌─────────────────────┐
│ FP16                │
│ 16 Bits Per Weight  │
└─────────────────────┘

        │

        ▼

Large Memory Usage

        │

        ▼

Apply Quantization

        │

        ▼

┌─────────────────────┐
│ Q4                  │
│ 4 Bits Per Weight   │
└─────────────────────┘

        │

        ▼

Much Smaller Memory Usage
```

This is the core idea behind quantization.

---

## 1.31.4 What Is Quantization?

Quantization is the process of storing model weights using lower numerical precision.

Instead of storing weights using large formats such as:

* FP16
* BF16

the weights are represented using fewer bits.

Conceptually:

```text
Original Weights

        │

        ▼

High Precision

        │

        ▼

Large Memory Usage

        │

        ▼

Quantization

        │

        ▼

Lower Precision

        │

        ▼

Reduced Memory Usage
```

The goal is not simply to make the model smaller.

The goal is to make the model smaller while preserving useful behavior.

---

## 1.31.5 A Helpful Analogy

Imagine storing photographs.

You might have:

```text
Uncompressed Image
```

which preserves maximum detail.

Or:

```text
Compressed Image
```

which consumes less storage.

The compressed image may look nearly identical to a human observer even though some information has been removed.

Quantization works in a similar way.

The objective is to remove information that contributes little to model behavior while preserving the information that matters most.

---

## 1.31.6 Why Local Users Care About Quantization

For local inference, quantization often determines whether a model can run at all.

Consider a user with a GPU containing:

```text
12 GB VRAM
```

The user wants to run a:

```text
14B Model
```

At high precision, the model may not fit.

The runtime may fail before generation even begins.

Applying quantization may reduce memory requirements enough for the model to load successfully.

### Figure 1.32 — Why Quantization Matters

```text
14B Model

       │

       ▼

Requires More Memory

       │

       ▼

Does Not Fit GPU

       │

       ▼

Apply Quantization

       │

       ▼

Reduced Memory Usage

       │

       ▼

Model Loads Successfully
```

Without quantization:

```text
No Inference
```

With quantization:

```text
Practical Inference
```

This is why quantization is one of the most important technologies in local AI.

---

## 1.31.7 Common Quantization Levels

Not all quantization methods are equal.

Different precision levels involve different trade-offs between quality and memory consumption.

The 2026 rule of thumb for local users is as follows.

### FP16 and BF16

FP16 and BF16 should be treated as the quality baseline.

Use them when:

* Memory is abundant.
* Benchmarking models.
* Evaluating model quality.
* Measuring quantization impact.

Conceptually:

```text
Highest Quality

       +

Highest Memory Usage
```

These formats are useful because they provide a reference point.

When evaluating a quantized model, it is often helpful to compare its behavior against FP16 or BF16.

---

### Q8 / INT8

Q8 is often considered near-lossless.

For many chat, writing, and document tasks, quality differences may be difficult to notice.

Benefits include:

* Strong quality retention.
* Reduced memory usage.
* Broad compatibility.

Conceptually:

```text
Very High Quality

       +

Meaningful Memory Savings
```

---

### Q6 and Q5

Q6 and Q5 are frequently regarded as one of the best trade-off points.

They provide:

* Excellent quality.
* Good compression.
* Practical deployment characteristics.

Many experienced local users consider these formats the safest compromise between quality and efficiency.

---

### Q4

Q4 has become one of the most common deployment choices.

It is often considered the consumer sweet spot.

Benefits include:

* Large memory reduction.
* Good response quality.
* Broad hardware compatibility.

For many workloads:

* Chat
* Retrieval
* Document analysis
* General assistants

Q4 performs remarkably well.

---

### Q3 and Q2

These formats prioritize fitting larger models into limited hardware.

They should generally be viewed as emergency options rather than ideal deployments.

Quality degradation often appears first in:

* Mathematics
* Coding
* Multi-step reasoning
* Tool use
* JSON generation
* Long-context retrieval

The model may still chat effectively while struggling with more demanding tasks.

---

## 1.31.8 Bigger Models Are Not Always Better

A common misconception among new users is:

> Bigger parameter count always wins.

Consider:

```text
13B Model
      ↓
Q2
```

versus

```text
7B Model
      ↓
Q6
```

Many users automatically assume the 13B model is superior.

In practice:

```text
7B Q6
```

may outperform:

```text
13B Q2
```

on reasoning tasks, code generation, and instruction following.

Why?

Because the larger model has lost too much information through aggressive quantization.

A useful principle is:

> Do not worship parameter count.

Model quality depends on both model size and retained precision.

---

## 1.31.9 Weight Quantization vs KV Cache Quantization

These terms are frequently confused.

They are not the same thing.

### Weight Quantization

Weight quantization reduces the size of the model itself.

### Figure 1.33 — Weight Quantization

```text
Stored Model

      │

      ▼

Model Weights

      │

      ▼

Quantized Weights

      │

      ▼

Smaller Model
```

This affects:

* Disk storage
* VRAM usage
* RAM usage

---

### KV Cache Quantization

KV-cache quantization reduces the size of working memory during inference.

### Figure 1.34 — KV Cache Quantization

```text
Conversation

      │

      ▼

Tokens

      │

      ▼

KV Cache

      │

      ▼

Quantized Cache

      │

      ▼

Reduced Context Memory
```

This affects:

* Long conversations
* Long documents
* Context length scalability

The model and the KV cache are separate memory budgets.

Reducing one does not automatically reduce the other.

---

## 1.31.10 When Quantization Fails

Quantization failures rarely appear immediately.

Instead, degradation tends to emerge gradually.

Common symptoms include:

* Incorrect arithmetic
* Missing reasoning steps
* Lower coding accuracy
* Invalid JSON output
* Tool-call failures
* Reduced retrieval quality
* Subtle instruction-following mistakes

This explains why evaluation should always use realistic workloads.

A quantized model may perform perfectly in casual conversation while struggling in production tasks.

---

## 1.31.11 Common Misunderstandings

### Misconception 1

> Lower bits always means better.

Reality:

Lower bits reduce memory usage, but excessive compression can damage quality.

---

### Misconception 2

> Bigger models always win.

Reality:

A smaller model at higher precision may outperform a larger model compressed too aggressively.

---

### Misconception 3

> Weight quantization and KV-cache quantization are the same thing.

Reality:

They reduce different memory budgets and solve different problems.

---

### Misconception 4

> Q4 automatically means faster inference.

Reality:

Performance depends on:

* Hardware
* Memory bandwidth
* Runtime
* Attention kernels
* Quantization implementation

Quantization alone does not guarantee speed improvements.

---

## Key Takeaways

Quantization reduces memory requirements by storing weights using lower numerical precision.

FP16 and BF16 remain the quality baseline.

Q8 provides near-lossless behavior for many workloads.

Q6 and Q5 offer an excellent balance between quality and efficiency.

Q4 remains a popular consumer deployment target.

Q3 and Q2 should generally be used only when fitting a larger model is more important than preserving quality.

Weight quantization reduces model size.

KV-cache quantization reduces working-memory size.

These are separate concepts and should not be confused.

## 1.32.5 Load Safety: The First Security Rule Of Local AI

Before discussing individual formats, we need to discuss security.

Many beginners assume:

> A model file is just data.

Unfortunately, this is not always true.

Some serialization systems allow executable code to be embedded within files.

This creates an obvious security risk.

A malicious model file could potentially:

* Execute code
* Modify files
* Download software
* Install malware

The local AI security rule is therefore:

> Do not let a stranger's model file become a stranger's code execution.

This rule is surprisingly important.

As local AI becomes more popular, users increasingly download models from unfamiliar sources.

Understanding which formats are safer helps reduce risk.

# 1.34 VRAM Math for Local Models

## Learning Objectives

By the end of this chapter, readers will be able to:

* Understand where GPU memory is consumed during inference.

* Distinguish between weight memory, KV cache memory, and runtime overhead.

## 1.34.1 Why VRAM Matters

One of the first questions new local AI users ask is:

> Will this model fit on my GPU?

Unfortunately, the answer is rarely as simple as:

```text id="m8g24v"
Model Size
      ↓
GPU Size
      ↓
Works
```

Language model memory usage is influenced by many factors.

Examples include:

* Model weights
* Context length
* KV cache size
* Runtime overhead
* Quantization level
* Number of users
* Vision inputs
* Adapters
* Serving strategy

As a result, two users running the same model can observe dramatically different memory consumption.

Understanding VRAM math helps explain why.

---

## 1.34.2 What Is VRAM?

VRAM stands for Video Random Access Memory.

For local AI workloads, VRAM acts as the model's workspace.

A useful analogy is a workbench.

Imagine assembling a large machine.

You need space for:

* The machine itself
* The tools
* The instructions
* The parts currently being used

If the workbench is too small, the work becomes difficult or impossible.

VRAM behaves similarly.

The model must fit.

The cache must fit.

The runtime must fit.

Everything required for inference must share the same memory budget.

### Figure 1.52 — VRAM as a Workspace

```text
GPU VRAM

┌─────────────────────────────┐
│                             │
│   Model Weights             │
│                             │
│   KV Cache                  │
│                             │
│   Runtime Buffers           │
│                             │
│   Temporary Tensors         │
│                             │
└─────────────────────────────┘
```

If the workspace fills completely, problems begin to appear.

---

## 1.34.3 The Three Major Memory Consumers

Although many factors influence memory usage, most local deployments are dominated by three major consumers.

### Model Weights

The stored parameters of the model.

### KV Cache

Working memory used during generation.

### Runtime Overhead

Everything required by the runtime itself.

Conceptually:

### Figure 1.53 — The Three Memory Buckets

```text
Available VRAM

      │

      ▼

┌──────────────────────┐
│ Model Weights        │
└──────────────────────┘

┌──────────────────────┐
│ KV Cache             │
└──────────────────────┘

┌──────────────────────┐
│ Runtime Overhead     │
└──────────────────────┘
```

Every deployment must account for all three.

Ignoring any one of them often leads to incorrect estimates.

---

# Model Weight Memory

## 1.34.4 Weight Memory

The easiest memory component to estimate is model weights.

Earlier chapters explained that a model consists of billions of parameters.

Each parameter occupies memory.

The approximate formula is:

```text
Weight Memory

≈

Parameters × Bytes Per Parameter
```

This is not perfectly exact, but it is extremely useful for planning.

---

## 1.34.5 Bytes Per Parameter

Different quantization levels use different amounts of storage.

Useful approximations are:

```text
FP16 / BF16

≈ 2 Bytes Per Parameter
```

```text
INT8 / Q8

≈ 1 Byte Per Parameter
```

```text
Q4

≈ 0.5 Bytes Per Parameter
```

Additional metadata and format overhead may increase the final number slightly.

However, these estimates are generally accurate enough for quick planning.

---

## 1.34.6 Example: 7B Model

Suppose we have:

```text
7 Billion Parameters
```

Running in FP16.

Using the approximation:

```text
7B × 2 Bytes
```

we obtain:

```text
≈ 14 GB
```

of weight memory.

Conceptually:

### Figure 1.54 — 7B FP16

```text
7B Parameters

      │

      ▼

2 Bytes Per Parameter

      │

      ▼

≈ 14 GB Weights
```

Before any tokens are processed, approximately 14 GB of memory may already be occupied.

---

## 1.34.7 Example: 7B Q4

Now suppose the same model is quantized.

```text
7B Parameters

      │

      ▼

0.5 Bytes Per Parameter

      │

      ▼

≈ 3.5 GB Weights
```

### Figure 1.55 — Quantization Savings

```text
FP16

≈ 14 GB


Q4

≈ 3.5 GB
```

The model itself has not changed.

Only the storage precision changed.

This illustrates why quantization is so important.

# KV Cache Memory

## 1.34.8 The Memory Most Beginners Forget

Many users estimate only model weights.

Then the model crashes.

Why?

Because weight memory is only part of the story.

The second major consumer is the KV cache.

Earlier chapters introduced the KV cache as the model's working memory.

Every token added to context increases cache size.

This growth continues throughout the conversation.

---

## 1.34.9 KV Cache Growth

A useful mental model is:


## 1.34.17 A Practical Memory Formula

A realistic planning formula looks like:

```text

+

KV Cache

+

Runtime Overhead

+

+

```

For vision systems, adapters, or speculative decoding, additional terms may be required.

This formula is not exact.

However, it provides a useful framework for thinking about memory.

## 1.34.18 Leave Headroom

> Do not run at 99% VRAM utilization.

* Memory fragmentation
* Temporary allocation failures
* Runtime instability
* Unexpected spikes

```text

80–90% Usage


Avoid

99–100% Usage
```

The safest deployment is not the deployment that barely fits.

The safest deployment is the deployment with room to breathe.

## Key Takeaways

VRAM consumption is dominated by three major components:

* Model weights
* KV cache
* Runtime overhead

Weight memory can be estimated using:


Concurrency, vision inputs, adapters, speculative decoding, and MoE architectures can significantly increase memory requirements.

Always leave approximately 10–20% VRAM headroom to avoid instability and out-of-memory failures.

# Why VRAM Is Not Everything

## 1.35.8 The Capacity Illusion

One of the most common beginner mistakes is:

> More VRAM automatically means more speed.

This is not necessarily true.

Consider two GPUs.

Both contain:

```text id="f8m3zj"
24 GB VRAM
```

One may generate:

```text id="9p4uyf"
20 Tokens/Second
```

while another achieves:

```text id="xj6zvu"
60 Tokens/Second
```

Why?

Because capacity and performance are different characteristics.

---

## 1.35.9 Memory Bandwidth

Memory bandwidth describes how quickly information can move between memory and compute units.

A useful analogy is a highway.

### Figure 1.67 — Memory Bandwidth

```text id="58xq7q"
Narrow Highway

      │

      ▼

Limited Traffic Flow



────────────────────



Wide Highway

      │

      ▼

Higher Traffic Flow
```

A wider highway moves more cars.

A higher-bandwidth GPU moves more data.

Language models move enormous amounts of data.

As a result, memory bandwidth becomes extremely important.

---

## 1.35.10 Why Decode Is Often Memory-Bound

Earlier chapters introduced:

* Prefill
* Decode

These phases behave differently.

### Decode

During decode:

```text id="arcktl"
Generate One Token

      │

      ▼

Read Weights

      │

      ▼

Generate Token

      │

      ▼

Repeat
```

The GPU repeatedly streams model weights.

Relatively little computation occurs for each byte transferred.

As a result:

> Decode is often memory-bandwidth-bound.

The bottleneck becomes moving data rather than performing calculations.

---

## 1.35.11 Why Prefill Behaves Differently

Prefill is different.

During prefill:

```text id="mnw3h3"
Large Prompt

      │

      ▼

Many Tokens Processed

      │

      ▼

Parallel Computation
```

Because many tokens can be processed simultaneously:

> Prefill is often more compute-bound.

The GPU can utilize its computational resources more effectively.

This explains why some GPUs exhibit:

* Similar prefill speeds
* Very different decode speeds

The workload characteristics differ.

---

## 1.35.12 The Decode Pipeline

A simplified view looks like:

### Figure 1.68 — Decode

```text id="hcm0kg"
KV Cache

      │

      ▼

Load Weights

      │

      ▼

Attention

      │

      ▼

Generate Token

      │

      ▼

Repeat
```

This cycle repeats continuously during generation.

Memory movement becomes a dominant cost.

# CPU Offloading

## 1.35.13 The Most Common Local AI Trap

Many users eventually encounter CPU offloading.

Suppose a model almost fits.

Example:

```text id="75okb2"
GPU VRAM

95% Full
```

The runtime may decide to place some layers on the CPU.

Conceptually:

### Figure 1.69 — CPU Offload

```text id="j27xso"
GPU Layers

      │

      ▼

Fast Execution



CPU Layers

      │

      ▼

Much Slower Access
```

The model technically runs.

However, performance may collapse.

---

## 1.35.14 Why CPU Offload Hurts

GPUs and CPUs communicate across relatively slow interconnects.

Conceptually:

### Figure 1.70 — Offload Penalty

```text id="jjv3kh"
GPU

 │

 ▼

Transfer Data

 │

 ▼

CPU

 │

 ▼

Transfer Back

 │

 ▼

Continue Generation
```

Those transfers add latency.

The result can be dramatic reductions in token generation speed.

---

## 1.35.15 When CPU Offload Makes Sense

CPU offloading is not useless.

It can be valuable for:

* Testing models
* Quick experiments
* Temporary evaluation
* Compatibility checks

However:

> CPU offload is a compatibility strategy.

It is not a performance strategy.

A model that barely fits through aggressive offloading often provides a poor user experience.

---

## 1.35.16 The Hardware Planning Rule

When evaluating hardware, avoid asking:

> Can I technically run this?

Instead ask:

> Can I run this comfortably?

Comfortable operation includes:

* Acceptable token speed
* Reasonable context length
* Stable memory usage
* Future flexibility

The difference between these two questions is often the difference between a satisfying deployment and a frustrating one.

---

### Misconception 1

> If a model fits, performance will be good.

Reality:

Capacity and performance are different problems.

---

### Misconception 2

> VRAM is the only specification that matters.

Reality:

Memory bandwidth frequently has a major influence on token speed.

---

### Misconception 3

> Prefill and decode behave the same way.

Reality:

### Misconception 4

---

## Key Takeaways

Hardware planning involves more than VRAM capacity.

For many serious local users in 2026:

* 16 GB represents the practical minimum comfortable tier.
* 24 GB represents an excellent enthusiast tier.
* 48 GB and beyond opens substantially larger deployment possibilities.

Performance depends on memory bandwidth, compute throughput, runtime quality, context length, quantization, attention implementation, and workload characteristics.

Decode is often memory-bandwidth-bound.

Prefill is often more compute-bound.

The most frustrating local AI deployments are often those that barely fit and require heavy CPU offloading.

A model that runs comfortably is usually far more valuable than a model that merely runs.

# 1.36 Choosing a Model That Fits

## Learning Objectives

## 1.36.1 The Wrong Question

One of the most common questions in local AI communities is:

> What is the best model?

At first glance, this seems reasonable.

If one model scores higher on benchmarks than another, surely it must be the better choice.

In practice, this question is often unhelpful.

A more useful question is:

> What is the smallest model that successfully solves my real workload on my hardware?

This distinction is extremely important.

A model that achieves slightly higher benchmark scores but runs slowly, consumes excessive memory, or fails to fit comfortably within available resources may provide a worse overall experience.

A useful mental model is:

```text
Best Model On Paper

      ≠

Best Model For You
```

The goal is not to maximize parameter count.

The goal is to maximize useful work.

---

A deployment that consumes:


* Out-of-memory failures
* Fragmentation issues
* Runtime instability
* Reduced flexibility

A model that fits comfortably is usually more valuable than a model that barely fits.

# 1.36.21 What Controls Speed?

After choosing a model that fits your hardware, the next question naturally becomes:

> **How fast will it run?**

Many beginners assume that token generation speed depends on a single specification.

Some believe it is determined entirely by VRAM.

Others focus only on GPU FLOPs or parameter count.

In reality, none of these tells the complete story.

Token generation speed is the result of many components working together.

A model is only as fast as its slowest bottleneck.

For this reason, two users running the same model can observe dramatically different performance.

The model may be identical.

The prompt may be identical.

Even the GPU memory capacity may be identical.

Yet one system may generate responses much faster than the other.

Understanding why requires looking at the complete inference pipeline.

---

## 1.36.22 Speed Is A System Property

Unlike benchmark scores, inference speed is not determined by a single number.

Instead, it emerges from the interaction of multiple hardware and software components.

Conceptually:

### Figure 1.76 — Factors Affecting Token Speed

```text
                 Tokens Per Second

                         │

     ┌───────────────────┼───────────────────┐
     │                   │                   │
     ▼                   ▼                   ▼

 Model Size      Memory Bandwidth      GPU Compute

     │                   │                   │
     ├──────────────┬────┴────┬──────────────┤
     │              │         │              │
     ▼              ▼         ▼              ▼

Context Length  Attention  Quantization  Runtime

        │             │          │            │
        └─────────────┴──────────┴────────────┘

                         │

                         ▼

                 Final Token Speed
```

Every one of these factors contributes to the final user experience.

Improving only one of them does not guarantee faster inference.

---

## 1.36.23 Memory Bandwidth

Earlier chapters introduced VRAM capacity.

Capacity determines **whether a model fits**.

Memory bandwidth determines **how quickly the GPU can move data**.

These are completely different concepts.

A useful analogy is a water pipe.

Imagine two water tanks.

Both contain:

```text
100 Liters
```

One tank connects to a narrow pipe.

The other connects to a wide pipe.

Both contain the same amount of water.

However, the wider pipe can deliver water much faster.

Memory bandwidth behaves similarly.

Two GPUs may each contain:

```text
24 GB VRAM
```

yet one may move data significantly faster.

Conceptually:

### Figure 1.77 — Memory Capacity vs Memory Bandwidth

```text
24 GB VRAM

        │

        ▼

──────────────────────────

GPU A

Narrow Memory Bus

        │

        ▼

Lower Bandwidth

        │

        ▼

Lower Token Speed



──────────────────────────



GPU B

Wide Memory Bus

        │

        ▼

Higher Bandwidth

        │

        ▼

Higher Token Speed
```

For many local inference workloads, memory bandwidth has a larger influence on generation speed than raw compute performance.

---

## 1.36.24 Why Decode Is Usually Memory-Bound

Earlier chapters separated inference into two major phases:

* Prefill
* Decode

These phases stress hardware differently.

During decode, the model generates one token at a time.

For every new token, the GPU repeatedly reads model weights from memory.

Conceptually:

### Figure 1.78 — Decode Loop

```text
Previous Tokens

        │

        ▼

Load Model Weights

        │

        ▼

Run Attention

        │

        ▼

Generate One Token

        │

        ▼

Repeat
```

Notice what happens.

The same weights are read again and again.

The GPU spends much of its time moving data rather than performing massive parallel computation.

For this reason, decode is often described as **memory-bandwidth-bound**.

If memory movement becomes slow, token generation slows regardless of how powerful the compute hardware may be.

---

## 1.36.25 GPU FLOPs

GPU FLOPs describe the amount of mathematical computation a GPU can perform.

Unlike decode, prefill processes many tokens simultaneously.

Large portions of the model can therefore execute in parallel.

Conceptually:

### Figure 1.79 — Prefill

```text
Large Prompt

        │

        ▼

Many Tokens

        │

        ▼

Parallel Computation

        │

        ▼

Build KV Cache
```

Because much more computation occurs simultaneously, GPU compute performance becomes increasingly important.

This explains why:

* Faster GPUs often complete prefill much more quickly.
* Decode performance may improve only modestly.

A GPU with exceptional compute capability but modest memory bandwidth may perform extremely well during prefill while showing smaller improvements during decode.

---

## 1.36.26 VRAM Capacity

Capacity remains important even though it does not directly determine speed.

If either:

* Model weights

or

* KV cache

no longer fit inside GPU memory, the runtime may begin placing data on the CPU.

Conceptually:

### Figure 1.80 — GPU Memory Spill

```text
Model

      │

      ▼

GPU Memory Full

      │

      ▼

Move Layers

      │

      ▼

CPU Memory

      │

      ▼

Slower Generation
```

This is often called **CPU offloading**.

Although the model technically continues to run, performance can decline dramatically.

---

## 1.36.27 Attention Implementations

Modern runtimes contain highly optimized attention kernels.

Examples include:

* FlashAttention
* SDPA (Scaled Dot Product Attention)
* Paged Attention
* Runtime-specific optimized kernels

These implementations attempt to reduce unnecessary memory movement while improving computational efficiency.

Conceptually:

### Figure 1.81 — Optimized Attention

```text
Input Tokens

      │

      ▼

Attention Kernel

      │

      ├── Memory Optimizations
      ├── Faster Computation
      ├── Better Scheduling
      └── Reduced Overhead

      ▼

Output Tokens
```

Changing only the attention implementation can noticeably improve both memory consumption and inference speed without changing the model itself.

---

## 1.36.28 Quantization

Quantization influences speed in two different ways.

Smaller weights require less memory movement.

Less data movement often improves performance.

Conceptually:

```text
Larger Weights

      │

      ▼

More Memory Traffic

      │

      ▼

Potentially Slower
```

versus

```text
Smaller Weights

      │

      ▼

Less Memory Traffic

      │

      ▼

Potentially Faster
```

However, aggressive quantization introduces additional work.

The runtime may need to reconstruct higher-precision values during inference.

This process is called **dequantization**.

Dequantization itself consumes computation.

Therefore:

Smaller weights do **not** automatically guarantee higher performance.

The final result depends on the runtime, hardware, and quantization implementation.

---

## 1.36.29 Batch Size and Concurrency

A single user generates one sequence.

Servers frequently generate many sequences simultaneously.

Conceptually:

### Figure 1.82 — Batching

```text
Prompt A

Prompt B

Prompt C

Prompt D

      │

      ▼

Runtime Scheduler

      │

      ▼

Shared GPU Execution
```

Batching improves overall throughput because the GPU performs more work in parallel.

However, each active sequence still requires its own KV cache.

As batch size increases:

* Throughput usually increases.
* Memory usage also increases.

Finding the correct balance is an important deployment decision.

---

## 1.36.30 Prompt Length

Earlier chapters introduced the distinction between:

* Context length
* Prompt length

Long prompts increase the amount of work required during the prefill stage.

Conceptually:

```text
Longer Prompt

      │

      ▼

More Tokens

      │

      ▼

Longer Prefill

      │

      ▼

Later First Token
```

The user may therefore experience a noticeable delay before the first generated token appears.

---

## 1.36.31 Generated Length

Prompt length affects prefill.

Generated length affects decode.

Long responses require the decode loop to execute repeatedly.

Conceptually:

```text
Generate Token

      │

      ▼

Repeat

      │

      ▼

Repeat

      │

      ▼

Repeat
```

Long answers therefore expose decode performance much more clearly than short answers.

---

## 1.36.32 Speculative Decoding

Modern inference systems increasingly use speculative decoding techniques.

Instead of generating only one token at a time, a smaller draft model predicts several possible tokens.

The larger target model then verifies those predictions.

Examples include:

* EAGLE-style speculative decoding
* Multi-Token Prediction (MTP)
* DFlash
* DDTree

Conceptually:

### Figure 1.83 — Speculative Decoding

```text
Prompt

      │

      ▼

Draft Model

      │

      ▼

Draft Tokens

      │

      ▼

Target Model

      │

      ▼

Verification

      │

      ▼

Accepted Tokens
```

When supported by both the model and runtime, speculative decoding can significantly increase generation speed.

However, these techniques also introduce additional memory usage and implementation complexity.

---

## 1.36.33 The "Almost-Fit" Trap

One of the most frustrating local AI experiences occurs when a model almost fits into available memory.

The runtime begins moving some layers or portions of the KV cache between GPU memory and system memory.

Conceptually:

### Figure 1.84 — The Almost-Fit System

```text
GPU Memory

Nearly Full

        │

        ▼

Some Layers

Moved To CPU

        │

        ▼

Repeated Data Transfers

        │

        ▼

Very Slow Token Generation
```

The model technically works.

Responses eventually appear.

However, token generation speed may fall from smooth and interactive to frustratingly slow.

For this reason:

> A model that comfortably fits is usually far more valuable than one that barely fits.

---

## 1.36.34 Benchmark Your Actual Workload

Leaderboards provide useful reference points.

However, benchmark numbers rarely represent your own deployment.

A BF16 benchmark measured on enterprise hardware tells you very little about:

* Q4 quantization
* Consumer GPUs
* Your runtime
* Your prompt length
* Your context size
* Your workload

Instead, benchmark the exact configuration you intend to use.

Evaluate:

* Runtime
* Quantization
* Context length
* Prompt format
* Model
* Hardware

Only then will you understand how your local deployment actually performs.

---

## Common Misunderstandings

### Misconception 1

> Token speed depends only on GPU FLOPs.

**Reality:**

Memory bandwidth, runtime quality, attention kernels, and model size often have equal or greater influence.

---

### Misconception 2

> More VRAM automatically means a faster model.

**Reality:**

VRAM capacity determines whether the model fits.

Memory bandwidth largely determines how quickly data moves.

---

### Misconception 3

> Quantization always increases speed.

**Reality:**

Quantization often improves performance, but aggressive quantization may introduce dequantization overhead and quality loss.

---

### Misconception 4

> Leaderboard numbers predict my local experience.

**Reality:**

Real-world performance depends on your hardware, runtime, quantization, context length, and workload.

---

## Key Takeaways

Inference speed is a system property rather than a single hardware specification.

Model size, memory bandwidth, GPU compute, attention kernels, runtime quality, context length, quantization, batching, and speculative decoding all contribute to token generation speed.

Decode is frequently limited by memory bandwidth, while prefill often benefits more from GPU compute capability.

Long prompts primarily affect prefill latency, whereas long responses expose decode performance.

A model that barely fits and repeatedly transfers data between GPU and CPU may technically function but often provides an unsatisfactory user experience.

The most reliable benchmark is your own workload running on your own hardware with your intended runtime, quantization, context length, and prompt structure.

## 1.37.1 What Is Long Context?

Earlier chapters introduced the concept of a **context window**.

The context window is the amount of information a language model can consider at one time.

Every prompt, conversation, document, system instruction, tool response, and generated token occupies part of this context window.

For many years, language models could only process relatively small amounts of text.

Modern models have expanded this dramatically.

Today it is common to see models advertising context windows such as:

```text id="lc1"
8K Tokens

32K Tokens

128K Tokens

256K Tokens

1 Million Tokens
```

At first glance, these numbers seem almost magical.

A beginner might naturally conclude:

> "If larger context windows exist, I should always use the largest one."

Unfortunately, the reality is more complicated.

Long context is an extremely useful capability.

It is **not** a free capability.

Every additional token introduces additional computational work, memory consumption, and opportunities for error.

---

## 1.37.2 What Does "128K Context" Actually Mean?

Suppose a model advertises:

```text id="lc2"
128K Context Window
```

This does **not** mean the model permanently remembers 128,000 tokens.

Instead, it means the model can consider approximately that many tokens during a single inference session.

Conceptually:

### Figure 1.85 — Context Window

```text id="lc3"
System Prompt

        │

        ▼

Conversation History

        │

        ▼

Documents

        │

        ▼

Tool Results

        │

        ▼

Current User Prompt

        │

        ▼

Total Context
```

Every piece of information competes for the same context budget.

If the total exceeds the supported window, older information must usually be discarded or compressed.

---

## 1.37.3 Why Bigger Sounds Better

Imagine asking a model to summarize an entire technical book.

A small context window might require splitting the document into many pieces.

A much larger context window may allow the entire document to be processed in a single request.

This makes long context particularly attractive for:

* Large documents
* Multi-file projects
* Entire conversations
* Long transcripts
* Source code repositories

Conceptually:

### Figure 1.86 — Small vs Large Context

```text id="lc4"
Small Context

┌─────────────┐
│ Chapter 1   │
└─────────────┘

↓

Process Again

↓

Chapter 2

↓

Repeat



────────────────────────────



Large Context

┌───────────────────────────┐
│ Entire Document           │
└───────────────────────────┘

↓

Single Prompt
```

Long context can therefore simplify many workflows.

However, larger does not automatically mean better.

---

## 1.37.4 The Hidden Cost of Long Context

Every additional token increases the amount of work the model must perform.

Long context affects much more than memory usage.

It influences nearly every stage of inference.

Conceptually:

### Figure 1.87 — Cost of Longer Context

```text id="lc5"
Longer Context

        │

        ├── Larger KV Cache

        ├── More Attention Operations

        ├── Longer Prefill Time

        ├── Higher Memory Usage

        ├── Greater Computational Cost

        └── More Opportunities For Distraction

        │

        ▼

Higher Overall Cost
```

Unlike model weights, which remain fixed after loading, the computational work grows as the prompt becomes larger.

---

## 1.37.5 More Context Means More KV Cache

Earlier chapters introduced the KV cache as the model's working memory.

Every processed token contributes additional entries to the cache.

As context length increases:

```text id="lc6"
More Tokens

        │

        ▼

Larger KV Cache

        │

        ▼

More VRAM Usage
```

A model that comfortably fits with an 8K context window may exceed available memory when configured for 128K.

The weights remain unchanged.

Only the working memory increases.

This explains why increasing context length often has a much larger effect on memory usage than beginners expect.

---

## 1.37.6 Longer Context Means Longer Prefill

Before a model can begin generating a response, it must first process the entire prompt.

This stage is called **prefill**.

Conceptually:

### Figure 1.88 — Prefill

```text id="lc7"
Prompt

      │

      ▼

Tokenization

      │

      ▼

Prefill

      │

      ▼

KV Cache Built

      │

      ▼

Generation Begins
```

If the prompt contains only a few hundred tokens, prefill completes quickly.

If the prompt contains hundreds of thousands of tokens, prefill becomes substantially more expensive.

Users often notice this as a long delay before the first generated token appears.

---

## 1.37.7 Attention Also Becomes More Expensive

Earlier chapters explained that transformers use an attention mechanism.

Attention allows the model to relate different parts of the prompt to one another.

As context grows, the amount of information available for attention also grows.

Conceptually:

### Figure 1.89 — Expanding Attention

```text id="lc8"
Small Prompt

Token

 │

 ├── Nearby Tokens

 │

 ▼

Limited Relationships



────────────────────────────



Large Prompt

Token

 │

 ├── Thousands Of Tokens

 ├── Documents

 ├── Instructions

 ├── History

 ├── Tool Results

 └── Previous Reasoning

 │

 ▼

Much Larger Search Space
```

The model has significantly more information to consider.

This increases computational cost.

---

## 1.37.8 Bigger Context Does Not Guarantee Better Answers

A common misconception is:

> "If I give the model more information, the answers will always improve."

Unfortunately, this is not always true.

Very long prompts may contain:

* Redundant information
* Irrelevant details
* Contradictory instructions
* Duplicate facts
* Outdated information

The model must decide what deserves attention.

As the amount of irrelevant information increases, useful information may become harder to identify.

Long prompts can therefore reduce answer quality rather than improve it.

---

## 1.37.9 Quality Can Decay Across Distance

Another subtle challenge is long-range reasoning.

Many modern models can successfully process very long prompts.

However, maintaining consistent attention across extremely distant sections remains difficult.

For example:

```text id="lc9"
Beginning Of Document

↓

...

↓

Hundreds Of Pages

↓

...

↓

End Of Document
```

The model may accurately summarize the conclusion while overlooking a critical definition introduced near the beginning.

This phenomenon is often called **context degradation** or **attention decay across distance**.

Long-context support does not automatically guarantee perfect long-range recall.

---

# When Should You Use Long Context?

## 1.37.10 Good Use Cases

Long context is particularly valuable when large amounts of information genuinely need to be considered together.

Examples include:

* Whole-document analysis
* Codebase exploration
* Technical documentation
* Legal contracts
* Research papers
* Meeting transcripts
* Multi-file reasoning
* Large specification reviews

Conceptually:

### Figure 1.90 — Appropriate Long Context Tasks

```text id="lc10"
Large Documents

      │

      ▼

Long Context

      │

      ▼

Single Reasoning Process
```

These are situations where splitting the information into smaller pieces would significantly reduce understanding.

---

## 1.37.11 Long Context Is Not A Replacement For RAG

One of the most common beginner mistakes is assuming:

> "If my model supports 1 million tokens, I no longer need Retrieval-Augmented Generation."

This is incorrect.

Long context and RAG solve different problems.

RAG is designed to search very large collections of information.

Long context is designed to reason over information that has already been selected.

Conceptually:

### Figure 1.91 — RAG vs Long Context

```text id="lc11"
Knowledge Base

       │

       ▼

Retriever

       │

       ▼

       │

       ▼

Long Context Window

       │

       ▼

Language Model

       │

       ▼

Final Answer
```

In an effective system:

* RAG finds the relevant evidence.
* Long context allows the model to reason over that evidence.

The two techniques complement one another.

# Practical Prompting Habits

## 1.37.12 Repeat Critical Instructions

Important instructions should not appear only once.

Placing important instructions near both the beginning and the end of a prompt helps reinforce them.

Conceptually:

```text id="lc12"
Instructions

        │

Documents

        │

Evidence

        │

More Documents

        │

Instructions Again
```

This increases the likelihood that important constraints remain visible throughout inference.

---

## 1.37.13 Use Clear Structure

Large prompts become easier for both humans and models when they are well organized.

Useful techniques include:

* Section headings
* Numbered lists
* Delimiters
* Clearly labeled documents
* Consistent formatting

Conceptually:

### Figure 1.92 — Structured Prompt

```text id="lc13"
Document A

──────────────

Document B

──────────────

Document C

──────────────

Instructions

──────────────

Question
```

Good structure improves readability and reduces ambiguity.

---

## 1.37.14 Ask For Citations

When reasoning over large collections of information, require the model to justify its answers.

For example:

* Cite the relevant section.
* Reference the document.
* Quote the supporting paragraph.

This encourages the model to ground its responses in the supplied evidence rather than generating unsupported conclusions.

---

## 1.37.15 Compress Conversation History

Not every previous message remains useful forever.

Older interactions often contain information that is no longer relevant.

Instead of preserving an infinitely growing conversation, replace old history with concise summaries.

Conceptually:

### Figure 1.93 — Summary Memory

```text id="lc14"
Long Conversation

        │

        ▼

Summarization

        │

        ▼

Short Memory

        │

        ▼

Continue Conversation
```

Summary memory conserves context while preserving important information.

---

## 1.37.16 Think of Long Context as Expensive Attention

A useful mental model is:

Long context is **not** an unlimited notebook.

It is expensive attention.

Every additional token consumes memory.

Every additional token requires computation.

Every additional token competes for the model's attention.

The objective is therefore not to maximize context length.

The objective is to maximize the usefulness of every token that occupies the context window.

## Common Misunderstandings

### Misconception 1

> Larger context windows always produce better answers.

**Reality:**

Additional context often improves reasoning, but irrelevant information can reduce answer quality.

---

### Misconception 2

> Long context replaces Retrieval-Augmented Generation.

**Reality:**

RAG retrieves relevant information.

Long context reasons over that information.

The two approaches complement one another.

---

### Misconception 3

> Long context only increases memory usage.

**Reality:**

Long context also increases prefill time, attention computation, latency, and evaluation complexity.

---

### Misconception 4

> The model remembers every part of a very long document equally well.

**Reality:**

Quality may degrade across long distances, and important details can become harder to retrieve from extremely large prompts.

---

## Key Takeaways

Long-context models can process very large prompts, making them valuable for document analysis, codebases, legal review, transcripts, and multi-file reasoning.

However, longer context increases KV cache memory, prefill latency, attention computation, and overall inference cost.

Long context should be viewed as a complement to Retrieval-Augmented Generation rather than a replacement.

Well-structured prompts, repeated critical instructions, citations, and summary memory improve long-context performance.

The most effective prompts are not necessarily the longest—they are the ones that make the best use of the model's limited attention.

## 1.38.1 Language Models Are No Longer Limited To Text

Earlier chapters assumed that a language model receives only text as input.

Historically, this was true.

A prompt consisted entirely of words.

For example:

```text id="mm1"
User Prompt

        │

        ▼

Tokenizer

        │

        ▼

Language Model

        │

        ▼

Response
```

Modern open-weight ecosystems have expanded considerably.

Many models can now understand much more than plain text.

Examples include:

* Images
* Screenshots
* Diagrams
* Documents
* Audio
* Video

Instead of interacting with only one type of information, these models combine multiple forms of input.

This capability is known as **multimodality**.

---

## 1.38.2 What Is A Multimodal Model?

A multimodal model is a model that can process more than one type of data.

The word itself explains the idea.

```text id="mm2"
Multi

+

Modal

=

Multiple Types Of Input
```

Traditional language models have only one modality:

```text id="mm3"
Text
```

Modern multimodal models may accept:

```text id="mm4"
Text

+

Images

+

Audio

+

Video
```

Some models support only images in addition to text.

Others also process spoken language, videos, or combinations of multiple input types.

The exact capabilities depend on the architecture.

---

## 1.38.3 How Does A Model Understand Images?

A language model cannot directly understand pixels.

Just as earlier chapters explained that text must first be converted into tokens, images must also be converted into representations that the model can process.

This conversion is performed by a component called a **vision encoder**.

Conceptually:

### Figure 1.94 — Image Processing Pipeline

```text id="mm5"
Image

      │

      ▼

Vision Encoder

      │

      ▼

Image Embeddings

      │

      ▼

Language Model

      │

      ▼

Generated Response
```

The vision encoder performs a role similar to the tokenizer.

Instead of converting words into tokens, it converts visual information into numerical representations that the language model can understand.

Without this step, the language model would see only meaningless pixels.

---

## 1.38.4 Images Become Tokens Too

One of the most common beginner misconceptions is:

> "Images are separate from the context window."

In reality, they are not.

Although images are processed differently from text, they ultimately occupy the same finite context budget.

Conceptually:

### Figure 1.95 — Shared Context Window

```text id="mm6"
Context Window

┌───────────────────────────────┐
│ System Prompt                 │
├───────────────────────────────┤

├───────────────────────────────┤
│ Image Tokens                  │
├───────────────────────────────┤
│ Previous Conversation         │
├───────────────────────────────┤
│ Tool Results                  │
└───────────────────────────────┘
```

Every image consumes part of the available context.

The larger the image, the more visual tokens may be required.

This leaves fewer tokens available for text.

---

## 1.38.5 The Hidden Cost Of Images

Many users assume adding an image has little impact on inference.

In reality, images introduce several additional costs.

Examples include:

* Vision encoder computation
* Additional GPU memory
* Larger context windows
* Longer prefill times
* More attention computation

```text id="mm7"
      │

```

A multimodal request is therefore often more expensive than an equivalent text-only request.

## 1.38.6 High-Resolution Images Consume More Context

Image resolution matters.

A small image contains relatively little information.

A large, high-resolution image contains significantly more.

Most multimodal systems divide images into smaller regions or patches before processing them.

From the model's perspective:

> Image tokens and text tokens both occupy the same context window.

## 1.38.8 Why Multimodal Prompting Is Harder

Text-only prompting is relatively straightforward.

# Practical Limitations

## 1.38.9 Vision Models Can Still Hallucinate

Modern Vision Language Models have improved rapidly.

However, they remain imperfect.

They may:

* Invent objects that are not present.
* Misread text.
* Confuse similar objects.
* Miss small visual details.
* Produce incorrect descriptions.

This behavior is known as **visual hallucination**.

The model generates information that appears plausible but is unsupported by the image.

For this reason, important visual workflows should always include verification.

---

## 1.38.10 OCR Quality Varies

Many users expect a multimodal model to perform perfect Optical Character Recognition (OCR).

Reality is more complicated.

Performance depends on:

* Font quality
* Image resolution
* Lighting
* Rotation
* Compression
* Handwriting
* Language

Poor-quality scans often produce poor OCR.

Specialized OCR systems may outperform general-purpose multimodal models on difficult documents.

---

## 1.38.11 Charts And Tables Remain Challenging

Structured visual information introduces additional complexity.

Examples include:

* Financial reports
* Scientific graphs
* Engineering diagrams
* Complex tables
* Dense spreadsheets

These documents require both visual understanding and logical reasoning.

Small Vision Language Models often struggle with:

* Reading small labels.
* Understanding table structure.
* Interpreting chart relationships.
* Preserving numerical accuracy.

For high-stakes applications, evaluation with representative data is essential.

---

## 1.38.12 Evaluate Using Real Data

A common mistake is evaluating a multimodal model using only simple demonstration images.

For example:

```text id="mm11"
Picture Of A Dog

↓

Correct Description
```

This does **not** prove that the same model can reliably perform:

* Invoice extraction
* Medical document analysis
* Receipt processing
* Engineering drawings
* Legal forms
* Identity documents

Instead, evaluate using the actual documents your application will process.

Conceptually:

### Figure 1.100 — Real Evaluation

```text id="mm12"
Real Workload

      │

      ├── Receipts

      ├── Invoices

      ├── Charts

      ├── Tables

      ├── Contracts

      └── Screenshots

      │

      ▼

Benchmark Model
```

The best benchmark is always the workload you intend to deploy.

---

## 1.38.13 Practical Guidelines

When building multimodal systems:

* Count image tokens as part of the context window.
* Keep image resolution reasonable.
* Use the correct multimodal chat template.
* Test OCR with real documents.
* Verify chart and table understanding.
* Benchmark representative workloads rather than simple demonstration images.

Treat multimodal models as evolving tools rather than perfect visual reasoning systems.

---

## Common Misunderstandings

### Misconception 1

> Images do not consume context.

**Reality:**

Images are converted into visual tokens that occupy the same context window as text.

---

### Misconception 2

> Multimodal models understand raw pixels directly.

**Reality:**

Images must first be processed by a vision encoder before the language model can reason about them.

---

### Misconception 3

> OCR is always reliable.

**Reality:**

OCR quality depends heavily on image quality, document complexity, and the capabilities of the model.

---

### Misconception 4

> A successful photo demonstration proves production readiness.

**Reality:**

Invoice extraction, chart interpretation, tables, forms, and technical documents are substantially more difficult than describing everyday photographs.

---

## Key Takeaways

Multimodal language models extend traditional text-based models by accepting images and, in some cases, audio and video.

Visual information is processed by a vision encoder and converted into representations that share the same context window as text.

High-resolution images, audio, and video increase memory usage, prefill time, attention computation, and overall inference cost.

Multimodal prompting requires correct templates and careful runtime configuration.

Vision Language Models remain susceptible to hallucinations and variable OCR performance, particularly on complex documents, charts, and tables.

The most reliable evaluation strategy is to benchmark the model using the real documents, images, and workflows it will encounter in production rather than relying on simple demonstration examples.

## 1.39.16 PagedAttention

Earlier chapters explained that KV cache memory grows continuously during generation.

PagedAttention reduces wasted KV-cache memory by organizing cache allocation more efficiently.

The result is:

* Lower memory waste
* Better utilization
* Higher serving efficiency

It has become one of the most influential techniques in modern inference systems.

* DDTree

When successful, they significantly improve throughput.

## 1.40.1 Most Problems Are Not Model Problems

One of the biggest surprises for beginners is that most local AI failures are **not** caused by poor models.

When a model produces strange outputs, many users immediately conclude:

> "This model is bad."

In reality, the cause is often somewhere else.

Examples include:

* Incorrect chat templates
* Insufficient memory
* Wrong tokenizer
* Runtime incompatibility
* Retrieval failures
* Poor decoding settings

Learning to isolate the real cause saves enormous amounts of time.

# Performance Problems

## 1.40.7 Slow First Token

Users often notice that the first generated token appears much later than expected.

Earlier chapters introduced **prefill**.

Before generation begins, the runtime must process the entire prompt.

### Figure 1.114 — First Token Delay

```text id="fm6"
      │

      ▼

      │

      ▼
```

Large prompts increase prefill time.

## 1.40.8 Improving First Token Latency

Several techniques can reduce prefill time.

Examples include:

* Shorter prompts.
* Better retrieval.
* Prefix caching.
* Smaller context windows.
* Faster runtimes.
* Removing unnecessary conversation history.

Reducing prefill often produces a larger improvement than changing models.

---

## 1.40.9 Slow Streaming

Sometimes the first token appears quickly.

Subsequent tokens arrive very slowly.

This usually indicates that **decode** has become the bottleneck.

Earlier chapters explained that decode frequently depends on memory bandwidth.

Conceptually:

### Figure 1.115 — Slow Decode

```text id="fm7"
      ▼

      │
```

If weight movement becomes slow, every token becomes slow.

## 1.40.10 Improving Decode Performance

When streaming is slow, investigate:

* Memory bandwidth.
* CPU offloading.
* Quantization.
* Attention backend.
* Runtime optimizations.
* Speculative decoding support.

Sometimes the simplest solution is:

> Use a model that fits comfortably.

A slightly smaller model often produces a much better overall experience.

## Key Takeaways

Most local AI failures originate from memory limits, runtime configuration, prompt formatting, retrieval pipelines, or decoding settings rather than from the model itself.

Out-of-memory errors can usually be resolved by reducing model size, context length, batch size, or concurrency, or by selecting a more appropriate quantization.

Prompt formatting issues often arise from incorrect chat templates, tokenizers, BOS/EOS tokens, or tool schemas.

Performance problems should be separated into prefill latency and decode speed, as each has different causes and solutions.

Reliable document and tool workflows depend on retrieval quality, structured prompting, and careful runtime configuration.

Before replacing a model, always perform the simple configuration checks first—these "boring" checks solve far more problems than model swapping.

# 1.42.1 Local Does Not Mean Secure

One of the biggest selling points of local language models is privacy.

Unlike cloud-based services, prompts and responses can remain entirely on your own hardware.

## 1.42.4 Safe Weight Formats

## Key Takeaways

Running language models locally significantly improves privacy by allowing prompts and responses to remain on local hardware, but privacy alone does not guarantee security.

Secure local AI deployments require trusted model files, careful handling of remote code, protection against prompt injection and tool abuse, proper management of secrets, and disciplined version control.

Operational security practices such as running models with limited privileges, validating tool calls, reviewing telemetry settings, and maintaining reproducible configurations are far more effective than relying on assumptions that "local" automatically means "safe."

The strongest local AI systems are not simply private—they are private, secure, reproducible, and built on consistent operational discipline.

### Misconception 1

**Reality:**

Repository retrieval and relevant context must still be supplied.

---

### Misconception 2

> Lower temperature reduces coding quality.

**Reality:**

For many programming tasks, lower temperatures improve consistency and correctness.

---

### Misconception 3

> Local models can safely rewrite an entire codebase without supervision.

**Reality:**

Human review, testing, and validation remain essential regardless of where the model runs.

---

## Key Takeaways

Local language models are particularly valuable for software development because they protect private source code, reduce latency, lower API costs, and integrate naturally with developer tools.

The strongest coding assistants combine repository retrieval, targeted context, automated testing, and patch-based workflows rather than relying on conversational prompts alone.

Reliable coding workflows emphasize deterministic decoding, continuous evaluation, automated verification, and careful human review.

Keeping the model local makes development more private and more economical—but it does not replace sound software engineering practices.

This follows the **Principle of Least Privilege** introduced in the previous chapter.

* Required directories.

A narrowly scoped agent is significantly safer than one with unrestricted system access.

## Key Takeaways

Tool use transforms a language model from a conversational assistant into an autonomous agent capable of interacting with external systems.

Because agents can affect the real world, they require significantly stronger security practices than text-only chatbots.

# 1.46 RAG Beats Giant Prompts

## Learning Objectives

By the end of this chapter, readers will be able to:

* Understand the purpose of Retrieval-Augmented Generation (RAG).
* Learn why retrieval is usually a better strategy than placing entire document collections into a prompt.
* Understand every stage of a modern RAG pipeline.
* Recognize common failure points in retrieval systems.
* Learn why chunking is one of the most important design decisions in RAG.
* Understand the role of rerankers and why they cannot repair poor document ingestion.

---

# 1.46.1 Why Bigger Prompts Are Not Always Better

Modern language models support increasingly large context windows.

Examples include:

* 32K tokens
* 128K tokens
* 256K tokens
* 1 million tokens

Many beginners assume this means they should simply place every available document into a single prompt.

Conceptually:

### Figure 1.158 — The Giant Prompt Approach

```text id="rag1"
Document A

Document B

Document C

Document D

Document E

        │

        ▼

One Massive Prompt

        │

        ▼

Language Model
```

Although this approach sometimes works, it has several disadvantages.

Large prompts require:

* More memory.
* Longer prefill times.
* Larger KV caches.
* Higher computational cost.
* More attention operations.

As prompts grow larger, the model must spend increasing effort determining which information is actually relevant.

More information does not necessarily produce better answers.

---

## 1.46.2 What Is Retrieval-Augmented Generation?

Rather than providing every document to the model, Retrieval-Augmented Generation (RAG) retrieves only the information needed to answer the current question.

Instead of asking the model to search thousands of pages itself, an external retrieval system first identifies the most relevant pieces of information.

Only those pieces are included in the prompt.

Conceptually:

### Figure 1.159 — Retrieval-Augmented Generation

```text id="rag2"
Knowledge Base

        │

        ▼

Retriever

        │

        ▼

Relevant Chunks

        │

        ▼

Language Model

        │

        ▼

Grounded Answer
```

The language model spends its attention on carefully selected evidence rather than an entire library.

---

## 1.46.3 Why RAG Is Usually Better

A well-designed RAG system provides several important advantages.

It:

* Reduces prompt size.
* Decreases memory consumption.
* Improves response speed.
* Grounds answers in source documents.
* Makes citations easier.
* Scales to millions of documents.

Instead of expanding the context window indefinitely, RAG attempts to place only the most useful information in front of the model.

This usually produces higher-quality answers at a lower computational cost.

# Every Stage Can Fail

## 1.46.5 Document Ingestion

The first stage is document ingestion.

This process collects documents before they are indexed.

Examples include:

* PDFs
* Word documents
* HTML pages
* Markdown
* Plain text
* Wikis
* Emails
* Source code

If documents are incomplete, corrupted, or missing during ingestion, the remaining stages cannot recover that information.

A retrieval system cannot retrieve documents that were never indexed.

---

## 1.46.6 Parsing

Documents rarely arrive in a form suitable for retrieval.

PDFs must be interpreted.

Tables extracted.

Images processed.

Headers recognized.

Formatting preserved.

Conceptually:

### Figure 1.161 — Parsing

```text id="rag4"
      │

      ▼

      │

      ▼

```

Poor parsing can introduce significant errors.

Examples include:

* Broken tables.
* Missing paragraphs.
* Incorrect reading order.
* Corrupted OCR.
* Lost headings.

Once information is damaged during parsing, every later stage inherits those mistakes.

---

## 1.46.7 Chunking

After parsing, documents are divided into smaller pieces called **chunks**.

A chunk becomes the basic unit that retrieval systems search.

### Figure 1.162 — Chunking

## 1.46.8 Embeddings

Earlier chapters introduced embedding models.

Each chunk is converted into a numerical representation called an embedding.

Conceptually:

```text id="rag6"

      │

      ▼

      │

      ▼

```

Similar chunks produce vectors that are close together in vector space.

These vectors allow semantic retrieval rather than simple keyword matching.

---

## 1.46.9 Vector Index

Embeddings are stored inside a vector database or vector index.

Instead of searching raw text directly, retrieval searches these vector representations.

Conceptually:

```text id="rag7"

      │

      │

      ▼

```

Efficient indexing enables retrieval across very large document collections.

---

## 1.46.10 Retrieval

When a user submits a question, the system converts that question into another embedding.

The retriever then searches for nearby document vectors.

Conceptually:

### Figure 1.163 — Retrieval

```text id="rag8"

      │

      ▼

      │

      ▼

      │

      ▼

```

The retriever decides which evidence reaches the language model.

If it retrieves the wrong information, the model begins with the wrong evidence.

---

## 1.46.11 Reranking

Initial retrieval often returns many approximately relevant chunks.

A reranker performs a second evaluation.

Instead of asking:

> Which chunks are similar?

it asks:

> Which chunks are actually the most relevant to this specific question?

### Figure 1.164 — Reranking


---

## 1.46.12 Prompt Construction

After retrieval and reranking, the selected chunks are assembled into a prompt.

* Clear instructions.
* Retrieved evidence.
* Citations.
* User question.

Conceptually:

```text id="rag10"

+

      │

      ▼

```

Well-structured prompts help the model distinguish between instructions and supporting evidence.

---

## 1.46.13 Grounding Checks

Good RAG systems encourage answers that remain supported by retrieved evidence.

Conceptually:

```text id="rag11"

      │

      ▼

      │

      ▼

```

Answers lacking supporting evidence should be treated cautiously.

---

## 1.46.14 Evaluation

A RAG system should never be considered complete without evaluation.

Measure:

* Retrieval accuracy.
* Citation quality.
* Grounding.
* Hallucination rate.
* Latency.
* User satisfaction.

Evaluation identifies weaknesses before deployment.

# Why RAG Systems Fail

## 1.46.15 Every Stage Can Become a Bottleneck

Many users assume that poor RAG answers indicate a weak language model.

In practice, failure usually occurs much earlier.

Examples include:

Bad parsing:

Tables become unreadable.

Bad chunking:

Important context is separated across multiple chunks.

Bad retrieval:

Irrelevant paragraphs are selected.

Bad reranking:

Correct evidence appears too low in the ranking.

Conceptually:

### Figure 1.165 — Failure Propagation

```text id="rag12"

      │

      ▼

      │

      ▼

      │

      ▼

      │

      ▼

```

A powerful language model cannot answer correctly using evidence it never received.

# Chunking: The Silent Killer

## 1.46.16 Why Chunking Matters So Much

Chunking appears simple.

In reality, it often determines the success or failure of an entire RAG system.

If chunks are too small:

Important context disappears.

If chunks are too large:

Retrieval becomes less precise.

Finding the correct balance requires experimentation.

---

## 1.46.17 Fixed-Size Chunking

The simplest approach divides documents into equally sized pieces.

Conceptually:

```text id="rag13"
Document

────────────

Chunk

────────────

Chunk

────────────

Chunk
```

Although easy to implement, this strategy may split:

* Sentences.
* Tables.
* Code blocks.
* Paragraphs.

These boundaries often reduce retrieval quality.

---

## 1.46.18 Overlapping Chunks

One solution is overlap.

Instead of creating completely separate chunks, neighbouring chunks share some content.

Conceptually:

```text id="rag14"
Chunk 1

████████████



Chunk 2

      ████████████



Chunk 3

            ████████████
```

Overlap reduces the chance of losing important context at chunk boundaries.

The trade-off is increased storage and larger indexes.

---

## 1.46.19 Semantic and Hierarchical Chunking

Modern RAG systems increasingly prefer chunking based on document meaning rather than fixed lengths.

Examples include:

* Semantic chunking.
* Section-based chunking.
* Hierarchical chunking.
* Parent-document retrieval.

These methods attempt to preserve logical structure while improving retrieval accuracy.

However, no strategy is universally optimal.

Different document collections require different approaches.

---

## 1.46.20 There Is No Universal Chunk Size

One of the most common questions is:

> What is the best chunk size?

There is no single answer.

The optimal configuration depends on:

* Document type.
* Language.
* Retrieval model.
* Query style.
* Context length.
* Domain knowledge.

Always evaluate chunk size using your own documents rather than copying another project's configuration.

---

# Reranking Has Limits

## 1.46.21 What a Reranker Can Improve

A reranker often rescues imperfect retrieval.

It can identify the best candidates among retrieved chunks.

This frequently improves answer quality without requiring changes elsewhere in the pipeline.

---

## 1.46.22 What a Reranker Cannot Fix

A reranker only evaluates chunks that already exist.

It cannot recover:

* Missing documents.
* Lost tables.
* Broken parsing.
* Incorrect chunk boundaries.
* Information removed during ingestion.

Conceptually:

### Figure 1.166 — Limits of Reranking

```text id="rag15"
Answer Lost

During Ingestion

      │

      ▼

Retriever

      │

      ▼

Reranker

      │

      ▼

Still Missing
```

If the answer disappeared before indexing, no reranker can restore it.

---

## Common Misunderstandings

### Misconception 1

> Long context eliminates the need for RAG.

**Reality:**

Long-context models and RAG solve different problems. RAG scales better for large knowledge bases and provides stronger grounding.

---

### Misconception 2

> Poor RAG answers mean the language model is weak.

**Reality:**

Failures are often caused by document parsing, chunking, retrieval, or reranking rather than the language model itself.

---

### Misconception 3

> There is one perfect chunk size.

**Reality:**

Optimal chunking depends on the documents, retrieval model, and workload. It must be evaluated experimentally.

---

### Misconception 4

> A reranker can repair every retrieval problem.

**Reality:**

Rerankers can improve the ordering of retrieved chunks, but they cannot recover information that was lost during parsing, ingestion, or chunking.

---

## Key Takeaways

Retrieval-Augmented Generation improves efficiency and answer quality by retrieving only the most relevant evidence instead of relying on increasingly large prompts.

A modern RAG pipeline consists of multiple interconnected stages, including ingestion, parsing, chunking, embeddings, vector indexing, retrieval, reranking, prompt construction, grounding verification, and evaluation.

Every stage is a potential source of failure, and weaknesses introduced early in the pipeline often propagate to the final answer.

Chunking is one of the most influential design decisions in a RAG system. The choice of chunk size, overlap, and chunking strategy should always be evaluated using representative documents.

Rerankers are powerful tools for improving retrieval quality, but they cannot recover information that was lost before retrieval began. The most effective RAG systems optimize the entire pipeline rather than focusing exclusively on the language model.
