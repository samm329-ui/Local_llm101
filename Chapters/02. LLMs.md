# Title

# LLMs 101: A Practical Guide

## Introduction

Large Language Models are often discussed in terms of model size, GPU requirements, context windows, quantization methods, serving engines, and benchmark scores. While these topics are important, they are much easier to understand once the core mechanism of an LLM becomes intuitive.

At the center of every modern decoder-only language model is a simple loop. Text is converted into tokens. Those tokens are processed by a Transformer. Attention determines which earlier tokens are relevant to the current prediction. A Key-Value (KV) cache stores previously computed information so the model does not need to recompute the entire sequence repeatedly. The model then predicts a next token, appends it to the sequence, and repeats the process.

Everything else builds upon this foundation.

Concepts such as VRAM consumption, quantization, context length, chat templates, retrieval systems, serving engines, and model selection are ultimately consequences of the mechanics governing token generation. Understanding the loop first makes these later topics easier to reason about because they stop appearing as isolated technical concepts and instead become practical implications of how inference works.

This chapter focuses on the model side of LLMs. It introduces inference, tokens, tokenization, context windows, model weights, attention, KV caching, prefill, decode, and next-token prediction. The goal is to build intuition for how modern language models operate before moving on to deployment, hardware, and software considerations.

# Chapter 1: The LLM Inference Loop

## Learning Objectives

By the end of this chapter, readers will be able to:

* Explain the fundamental inference loop used by decoder-only language models.
* Understand the role of tokens during language generation.
* Distinguish between model weights, context, and KV cache.
* Describe how next-token prediction works.
* Explain the relationship between logits, probabilities, and decoding.
* Understand why generation speed is measured in tokens per second.
* Recognize the practical impact of context windows and token counts.
* Understand why tokenization influences memory usage, latency, and model behavior.

## 1.1 Start With the Loop

The most useful way to understand an LLM is to begin with the generation loop itself.

Many people imagine that a model receives a prompt, internally constructs a complete answer, and then reveals that answer to the user. This is not how a decoder-only language model operates.

Instead, generation occurs incrementally.

The model repeatedly predicts a single token. Once that token has been selected, it becomes part of the sequence. The model then predicts another token based on the updated sequence. This process continues until generation stops.

The loop can be described as:

1. Convert text into tokens.
2. Feed tokens into the model.
3. Process the sequence through the Transformer.
4. Use attention to determine which previous tokens are relevant.
5. Compute scores for possible next tokens.
6. Apply a decoding strategy.
7. Select one token.
8. Append the token to the sequence.
9. Update the KV cache.
10. Repeat.

Every response generated by a language model—from a single word to an entire book chapter—is ultimately produced through repeated execution of this loop.

Understanding this process provides the conceptual foundation for everything that follows.

## 1.2 What an LLM Actually Does

Running a model is called "Inference".

Inference is the process of using a trained model to generate predictions from an input sequence.

For a decoder-only language model, inference consists of repeatedly predicting the most appropriate next token based on the tokens already present in the sequence.

This process continues until one of several stopping conditions occurs:

* The model generates a stopping token.
* A predefined token limit is reached.
* The user manually stops generation.

A critical point is that the model is never generating an entire answer simultaneously.

Instead, every new token becomes part of the context that influences future predictions.

As a result, generation is inherently sequential.

The current state of the sequence determines what can be generated next.

This sequential behavior explains why language models can adapt their output continuously as generation progresses.

## 1.3 Weights, Context, and Working Memory

Three concepts are central to understanding inference:

### Model Weights

Model weights contain the "Patterns learned" during training.

They represent the model's accumulated knowledge about language, structure, formatting conventions, and relationships between tokens.

During inference, the weights remain fixed.

The model is not learning from the prompt while generating text. Instead, it is applying the patterns already encoded within its parameters.

Weights determine what the model has learned.

### Context

Context is the information currently visible to the model.

This includes:

* User prompts
* Conversation history
* System instructions
* Previously generated tokens

The context changes continuously throughout generation because each new token is appended to the sequence.

Context determines what the model is currently looking at.

### KV Cache

The Key-Value cache acts as a form of working memory during inference.

Without a KV cache, the model would need to recompute information about all previously processed tokens every time a new token was generated.

Such repeated computation would be inefficient and would dramatically reduce generation speed.

The KV cache stores previously computed information so that subsequent generation steps can reuse it.

This makes long conversations and long-form generation practical.

The weights provide learned knowledge.

The context provides current information.

The KV cache enables efficient continuation.

Together, these components make modern inference feasible.

## 1.4 The Mathematical View

An LLM can be viewed as a learned function.

Its purpose is to map a sequence of tokens to a probability distribution over possible next tokens.

The relationship can be expressed as:

f(theta, sequence) → probability distribution over next token

Where:

* theta represents the model weights.
* sequence represents the prompt and all generated tokens so far.

The output of the model is not text.

The output is a set of scores describing the likelihood of potential next tokens.

Text generation emerges because this prediction process is repeated many times.

Understanding this distinction is important because it highlights what the model is fundamentally doing: predicting what token should come next.

## 1.5 Logits, Probabilities, and Decoding

When the model evaluates a sequence, it produces raw numerical scores for every token in its vocabulary.

These raw scores are known as logits.

Logits are not probabilities.

They represent the model's relative preferences among possible next tokens.

A normalization process converts these logits into probabilities.

The resulting probability distribution indicates how likely each candidate token is relative to all others.

However, probabilities alone do not generate text.

A decoding process must select a specific token.

Decoding transforms a probability distribution into an actual token choice.

Once selected, that token is appended to the sequence, becoming part of the context for the next prediction.

Generation therefore consists of a cycle of prediction and selection.

The model predicts possibilities.

Decoding chooses one possibility.

The loop then continues.

## 1.6 Prefill and Decode

From a user's perspective, generation speed is often experienced in two distinct phases.

### Prefill

Before the first generated token appears, the model must process the entire prompt.

This phase is known as prefill.

During prefill, the model converts the prompt into internal representations and builds the information needed for subsequent generation.

Long prompts increase the amount of work required during this stage.

As a result, users may experience a noticeable pause before output begins.

A document containing thousands of tokens can produce substantial prefill latency even before the model generates its first word.

### Decode

After prompt processing is complete, the model enters the decode phase.

During decode, the model repeatedly predicts and generates one token at a time.

Each generation step updates the KV cache and extends the sequence.

Because decode controls how quickly text appears on screen, users often perceive decode speed as responsiveness.

However, prefill latency can be equally important when working with large prompts.

Understanding the distinction helps explain why long documents may feel slow even on systems that generate tokens rapidly once decoding begins.

## 1.7 Tokens

Language models do not operate directly on human-readable words.

They operate on tokens.

A token is a unit of text represented internally by an integer identifier.

Tokens can represent many different types of textual content.

Examples include:

* Entire words
* Word fragments
* Punctuation symbols
* Whitespace-prefixed text
* Byte-level fallback representations
* Special control markers

A token might correspond to a complete word such as "hello."

It might also correspond to only part of a word.

Long or uncommon words are frequently divided into multiple tokens.

Consequently, words and tokens should never be assumed to be equivalent.

The distinction is fundamental because every major resource consumed by an LLM is measured in tokens rather than words.

## 1.8 Tokenization

Tokenization is the process of converting text into tokens and converting tokens back into text.

The tokenizer serves as the translation layer between human language and the model's internal representation.

Different model families use different tokenizers.

Common approaches include:

* BPE-style tokenizers
* SentencePiece-style tokenizers

Because tokenizers differ, identical text may produce different token counts across different models.

A document that consumes one amount of context in one model may consume significantly more context in another.

This difference affects practical considerations such as memory usage, prompt length, context consumption, and latency.

Tokenization is therefore not merely a preprocessing step. It directly influences how efficiently a model can represent information.

## 1.9 Vocabulary Size

Every tokenizer operates using a vocabulary.

The vocabulary contains all token entries recognized by the tokenizer.

Vocabulary size affects how text is represented.

A larger vocabulary may allow some text patterns to be represented using fewer tokens.

However, vocabulary size also influences other parts of the model, including embedding and output projection structures.

This means vocabulary design involves trade-offs rather than simple improvements.

It also explains why comparing tokens-per-second performance across different model families can be misleading.

Different tokenizers may represent the same text with different numbers of tokens, making direct comparisons imperfect.

## 1.10 Why Tokens Matter

Tokens are the fundamental unit of work during inference.

They determine:

* Context window consumption
* KV cache growth
* Prompt processing cost
* Memory requirements
* Latency
* Efficiency for code and multilingual text
* Correct interpretation of chat formatting markers

Once text is viewed as tokens rather than words, many aspects of LLM behavior become easier to reason about.

Long context stops appearing magical and instead becomes a measurable computational cost.

Every additional token contributes to the work the model must perform.

## 1.11 Context Windows

A context window defines the maximum number of tokens the model can attend to simultaneously.

Modern models support a wide range of context lengths.

Examples include:

* 8K tokens
* 32K tokens
* 128K tokens
* 256K tokens
* Larger server-oriented configurations

However, supported context length should not be confused with practical context length.

A model may technically accept a large context while becoming slower as that limit is approached.

Similarly, coherence may degrade at extreme sequence lengths.

Therefore, context support should be evaluated in terms of actual workloads rather than advertised limits alone.

A context window is not free.

It consumes memory, increases computation, and can influence generation behavior.

Understanding token costs makes context windows easier to evaluate realistically.

## Practical Example

Consider a user submitting a large document for analysis.

The text is first converted into tokens.

The model processes those tokens during prefill.

Internal state is established and the KV cache is populated.

The model then computes logits for possible next tokens.

These logits are transformed into probabilities.

A decoding strategy selects a token.

The selected token is appended to the sequence.

The KV cache is updated.

The model repeats the process.

What appears to the user as a streaming response is actually a sequence of repeated next-token predictions occurring one token at a time.

## Common Misunderstandings

A common misconception is that an LLM writes an entire answer before displaying it. In reality, generation occurs one token at a time.

Another misunderstanding is that tokens are equivalent to words. Tokenization frequently divides words into smaller components, making token counts and word counts fundamentally different measurements.

Many users also assume that large context windows guarantee strong performance throughout the entire range. In practice, supported context length does not necessarily imply equal speed, cost, or quality at every size.

It is also common to think that responsiveness is determined only by generation speed. Prompt processing time can be equally important, particularly when working with long documents.

Finally, the KV cache is often mistaken for stored conversation text. It is actually a storage mechanism for previously computed information that allows efficient continuation of generation.

## Key Takeaways

Modern decoder-only language models operate through a repeated next-token prediction loop.

Text becomes tokens. Tokens are processed by the Transformer. Attention determines relevant context. The KV cache preserves previously computed information. The model predicts a next token, appends it to the sequence, and repeats the process.

Weights determine what the model has learned.

Context determines what it is currently examining.

The KV cache enables efficient inference.

Tokens are the fundamental unit of work, influencing memory usage, latency, context length, and generation cost.

Understanding these mechanics provides the foundation required for reasoning about all higher-level aspects of language model behavior and deployment.

## 1.13 Transformers

Most modern Large Language Models are based on the Transformer architecture. While users typically interact with models through chat interfaces and observe only the generated text, the actual computational machinery responsible for producing those outputs is the Transformer itself.

Most local chat-oriented language models are decoder-only Transformers. A decoder-only Transformer generates text by repeatedly predicting the next token while looking back at tokens that have already appeared in the sequence. Future tokens do not yet exist, so the model can only use information that is already present within the context window.

Everything discussed earlier in this chapter—including tokens, tokenization, model weights, context windows, KV caches, configuration settings, and chat templates—exists to support the computational engine underneath. Those concepts determine what information enters the model and how that information is represented. The Transformer is the system that processes those representations and converts them into predictions.

A useful way to think about the Transformer is as a stack of processing layers. Each layer receives a representation of the sequence, transforms that representation, and passes the result to the next layer. Through repeated transformation, the model gradually builds increasingly useful internal representations that can ultimately be used for next-token prediction.

A simplified Transformer layer contains several major components:

* Token embeddings
* Positional information
* Self-attention
* Feed-forward computation
* Layer normalization
* Residual connections
* Output projection

The relationship between these components can be visualized as follows.

### Token Embeddings

The first step inside the Transformer is the embedding stage.

Language models do not operate directly on token IDs. A token ID is merely an integer identifier. Before meaningful computation can occur, each token ID must be converted into a vector representation.

Embeddings perform this conversion. They transform discrete token identifiers into dense numerical representations that can participate in the mathematical operations performed throughout the network.

Embeddings therefore act as the bridge between symbolic language and numerical computation.

### Positional Information

Token embeddings alone do not contain information about order.

A sequence such as:

> The dog chased the cat

contains many of the same tokens as:

> The cat chased the dog

but the meanings differ because the token order differs.

The Transformer therefore requires positional information.

Many modern language models use Rotary Position Embeddings (RoPE). RoPE encodes position by rotating internal representations according to token location within the sequence. This allows the model to distinguish between tokens appearing at different positions while preserving useful relationships between them.

Without positional information, the model would struggle to understand sequence structure.

### Self-Attention

Once tokens have been embedded and positional information has been incorporated, the model applies self-attention.

Self-attention allows each token representation to examine previous token representations and determine which information is relevant.

Not every earlier token contributes equally to the next prediction. Some tokens may be highly relevant while others contribute very little. Self-attention allows the model to assign different levels of importance to different parts of the sequence.

This mechanism is one of the primary reasons Transformers can maintain coherence across long passages of text.

### Feed-Forward Computation (MLP)

After attention processing, the representation passes through a feed-forward network, often referred to as an MLP.

The MLP performs dense nonlinear computation that transforms the representation generated by attention.

A useful intuition is that attention determines what information matters, while the MLP reshapes that information into a more useful form.

A large fraction of a model's parameters often reside within these feed-forward layers. As models become larger, these layers account for a substantial portion of the total parameter count.

### Layer Normalization and Residual Connections

Modern language models may contain dozens or even hundreds of Transformer layers.

To maintain stable computation across such depth, Transformers use layer normalization and residual connections.

Layer normalization helps stabilize numerical values as they flow through the network.

Residual connections allow information from earlier stages to continue influencing later stages rather than being completely replaced.

Together, these mechanisms help information propagate effectively through deep networks.

### Output Projection

After the sequence has passed through the Transformer stack, the final hidden representation must be converted into next-token scores.

The output projection layer performs this conversion.

The result is a set of logits over the model vocabulary. These logits later become probabilities, which are then used by the decoding process discussed earlier in this chapter.

### Transformer Recap

The Transformer can be summarized as a sequence of repeated operations.

Tokens become vector representations through embeddings. Positional information keeps token order intact. Attention connects relevant parts of the sequence. Feed-forward networks reshape representations. Layer normalization and residual pathways stabilize computation. Output projection converts the final hidden state into logits over the vocabulary.

## 1.14 Attention

Attention is the mechanism that allows a token to determine which earlier tokens matter for the next prediction.

It is also one of the primary reasons local inference can become memory-sensitive.

When a model processes a sequence, it must preserve information about previous tokens so that future tokens can attend to them. As context length grows, the amount of information that must be stored and accessed can increase substantially.

For local inference workloads, attention design therefore has a direct impact on memory usage, context length behavior, and KV cache growth.

### Figure 1.10 — Conceptual View of Self-Attention

```text
Sequence

[The] [dog] [chased] [the] [cat]

                         ▲
                         │
                  Current Token

                         │
         ┌───────────────┼───────────────┐
         │               │               │
         ▼               ▼               ▼

      [dog]         [chased]         [the]

      High           Medium           Low
    Importance     Importance      Importance
```

The diagram above illustrates a simplified view of attention. The current token does not treat every previous token equally. Instead, it determines which earlier information is most relevant to the prediction task.

### Multi-Head Attention (MHA)

Classic Multi-Head Attention (MHA) stores separate key-value state for multiple attention heads.

Each head can learn different relationships within the sequence and focus on different patterns of information. This flexibility makes MHA highly expressive.

However, maintaining separate key-value information for many heads causes the KV cache to grow rapidly. As context length increases, memory requirements can become substantial.

This is one reason why long-context inference can become expensive.

### Multi-Query Attention (MQA)

Multi-Query Attention (MQA) reduces memory usage by allowing multiple query heads to share a single key-value head.

This design significantly reduces KV cache growth and improves memory efficiency.

The trade-off is that MQA may be less expressive than full Multi-Head Attention because fewer independent key-value representations are available.

### Grouped-Query Attention (GQA)

Grouped-Query Attention (GQA) provides a middle ground between MHA and MQA.

Instead of sharing a single key-value head across all queries, query heads are organized into groups. Each group shares key-value information internally.

This approach preserves more flexibility than MQA while consuming less memory than full MHA.

As a result, GQA has become a common design choice in many modern local language models.

### Figure 1.11 — Comparing MHA, MQA, and GQA

```text
MHA

Q1 ──► KV1
Q2 ──► KV2
Q3 ──► KV3
Q4 ──► KV4


MQA

Q1 ──┐

Q4 ──┘──► KV Group B
```

This comparison highlights the trade-off between flexibility and memory efficiency. MHA provides maximum flexibility but requires the largest KV cache. MQA minimizes memory consumption through complete sharing. GQA balances the two approaches and has therefore become popular in many modern local models.

### Attention Kernels and Runtime Performance

Attention architecture is only part of the performance story.

Modern runtimes often use optimized attention kernels such as FlashAttention and SDPA-style implementations. These kernels reduce attention memory traffic and improve hardware utilization.

Reducing memory movement is important because memory bandwidth can become a major bottleneck during inference.

As a result, two systems running the same model on identical hardware can exhibit dramatically different performance characteristics depending on the quality of their attention implementations.

A runtime with highly optimized attention kernels may achieve significantly better performance than one without such optimizations.

### Why Attention Design Matters

Attention design has practical consequences beyond architecture diagrams.

Two models may both contain seven billion parameters and advertise identical context lengths, yet behave very differently during long-context inference.

A 7B model using full Multi-Head Attention at 128K context may consume enough memory to exhaust a 24 GB GPU.

A 7B model using Grouped-Query Attention with the same advertised context length may fit comfortably on the same hardware.

This illustrates an important principle: parameter count is not the entire story.

When comparing models, attention type, KV head configuration, supported context length, and runtime support are often just as important as model size itself.

When evaluating local language models, attention architecture should therefore be considered alongside parameter count rather than after it.


### KV Cache in Practice

A useful rule of thumb for older Llama-style 7B models using Multi-Head Attention and FP16 KV storage is approximately:

text
~0.5 MiB per token

Under this approximation:

text
4K Tokens  ≈ 2 GiB KV Cache

32K Tokens ≈ 16 GiB KV Cache

These figures illustrate why context length can become a practical deployment constraint.

A model may fit comfortably into memory when idle, yet encounter memory pressure once large documents or long conversations are loaded.

This is because model weights are only one part of the memory budget.

The KV cache consumes memory independently of the weights and continues growing as context length increases.

### GQA and MQA Effects on KV Cache Size

Attention architecture significantly influences cache growth.

As discussed in the previous section:

* MHA stores separate key-value state for many heads.
* MQA allows query heads to share a single key-value representation.
* GQA organizes heads into shared groups.

Because MQA and GQA reduce the amount of stored key-value information, they can substantially reduce KV cache requirements.

This is one reason modern local models often prefer GQA or MQA-style architectures for long-context workloads.

The model may have the same parameter count as an older architecture while consuming significantly less cache memory.

### KV Cache Precision

The precision used to store cache values also influences memory consumption.

Many modern runtimes support:

* FP16 KV cache
* FP8 KV cache
* INT8 KV cache

For local users, FP8 and INT8 KV storage often provide substantial memory savings while maintaining practical performance.

The source material recommends treating approximately 8-bit KV storage as a practical compression floor for many local workloads in 2026.

### Sub-8-Bit KV Cache

Reducing KV cache precision below 8 bits is an active area of research.

Examples referenced in the source include:

* [KIVI Paper](https://arxiv.org/abs/2402.02750?utm_source=chatgpt.com)
* [KVQuant Paper](https://arxiv.org/abs/2401.18079?utm_source=chatgpt.com)

These systems demonstrate that 2-bit to 4-bit KV storage can function effectively when combined with specialized algorithms, calibration procedures, and custom kernels.

However, research results should not be confused with a simple runtime setting.

A carefully engineered research implementation is very different from enabling an aggressively compressed KV cache without validation.

For workloads where exact token recall matters—such as coding, tool usage, JSON generation, long-context retrieval, or structured outputs—sub-8-bit cache configurations should be benchmarked carefully before deployment.

### KV Cache Quantization vs Speculative Decoding

Another common source of confusion is the relationship between KV cache compression and speculative decoding systems.

These technologies solve different problems.

KV cache quantization reduces memory consumption.

Speculative decoding attempts to improve generation speed.

Examples referenced in the source include:

* [DFlash Paper](https://arxiv.org/abs/2602.06036?utm_source=chatgpt.com)
* [DDTree Paper](https://arxiv.org/abs/2604.12989?utm_source=chatgpt.com)

These approaches accelerate generation by drafting future tokens and verifying them.

They can reduce decode latency, but they do not eliminate the memory requirements associated with the KV cache.

The cache must still exist regardless of how efficiently tokens are generated.

### Why Models Sometimes Crash on Long Documents

A common misunderstanding among local users occurs when a model loads successfully but crashes after a large document is added.

The immediate assumption is often that the model itself does not fit into memory.

In many cases, the model weights fit perfectly.

The problem is that the working memory does not.

As context length grows, the KV cache expands. Eventually, the combined memory requirements of the weights and cache exceed available resources.

The weights fit.

The working memory does not.

## 1.16 Prefill and Decode

Language model inference operates in two distinct performance regimes:

* Prefill
* Decode

Although users often experience them as part of a single interaction, they represent fundamentally different types of computation.

Understanding this distinction helps explain why some workloads feel slow before generation begins while others feel slow during generation.

### Prefill

Prefill is the stage in which the model processes the prompt supplied by the user.

If a user submits a large document, every token within that document must be processed before generation can begin.

For example:

text
20,000-token document
        ↓
20,000 tokens processed
        ↓
First output token appears

The time spent waiting for the first generated token is often dominated by prefill.

Although prefill can be computationally expensive, it benefits from substantial parallelism. Modern GPUs can process many prompt tokens simultaneously, making prefill relatively efficient compared with purely sequential workloads.

Nevertheless, very large prompts can still create noticeable latency.

### Decode

Decode begins after prompt processing is complete.

During decode, the model generates new tokens one at a time.

Each generated token becomes part of the sequence and influences future generation.

Because every new token depends on previously generated tokens, decode is inherently sequential.

This is the phase responsible for the familiar streaming effect in chat applications.

The speed of decode often determines whether a model feels responsive during generation.

### Long Prompts vs Long Answers

Different workloads stress different parts of the inference pipeline.

Long prompts primarily increase prefill cost.

Long responses primarily increase decode cost.

Long conversations increase both.

This relationship can be summarized as:

   text
Long Prompt      → Prefill Cost

Long Answer      → Decode Cost

Long Conversation
      ↓
Prefill + Decode + KV Growth

### Conversation Growth and Cache Growth

In a chat application, every turn contributes additional tokens to the conversation history.

As history grows, the KV cache grows with it.

A conversation containing 16K tokens carries the memory cost of those 16K tokens throughout subsequent generation.

This is one reason long-running conversations eventually slow down or exhaust memory resources.

Infinite chat history is not free.

The cache continues growing until memory becomes a limiting factor.

## 1.17 Decoding

After the model produces logits, text has not yet been generated.

The model has only assigned scores to possible next tokens.

Decoding is the process that converts those scores into an actual token selection.

Once a token is selected, it is appended to the context and the inference loop repeats.

### Figure 1.14 — Decoding Loop

The decoding process does not modify model weights.

Instead, it determines how the model's predictions are translated into generated text.

Different decoding strategies can significantly influence output style and behavior.

### What Decoding Controls

Decoding settings answer three practical questions:

#### Randomness

How much variation should be allowed?

Higher randomness permits more diverse outputs.

### Practical Decoding Approaches

Different tasks often benefit from different decoding behavior.

For precise work:

* Lower temperature
* Shorter generation limits
* Explicit stop sequences
* Constrained decoding

These settings help maintain consistency and structure.

For creative work:

* Higher temperature
* Higher diversity
* Multiple candidate generations

These settings encourage exploration and variety.

For coding tasks:

* Conservative first-pass generation
* Alternative samples when exploring solutions

This approach balances reliability with experimentation.

### Greedy Decoding

A common assumption is that selecting the highest-probability token at every step always produces the best result.

In practice, this is not necessarily true.

Greedy decoding can become brittle.

Because it never explores alternatives, it may become trapped in repetitive patterns or converge on generic responses.

Deterministic settings are often useful for evaluations and reproducibility.

For brainstorming and ideation, allowing controlled exploration can produce more varied and useful outputs.

The appropriate decoding strategy depends on the task being performed rather than a universal notion of correctness.
