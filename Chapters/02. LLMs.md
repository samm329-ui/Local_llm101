# Title

# LLMs 101: A Practical Guide

## Introduction

Large Language Models are often discussed in terms of model size, GPU requirements, context windows, quantization methods, serving engines, and benchmark scores. While these topics are important, they are much easier to understand once the core mechanism of an LLM becomes intuitive.

At the center of every modern decoder-only language model is a simple loop. Text is converted into tokens. Those tokens are processed by a Transformer. Attention determines which earlier tokens are relevant to the current prediction. A Key-Value (KV) cache stores previously computed information so the model does not need to recompute the entire sequence repeatedly. The model then predicts a next token, appends it to the sequence, and repeats the process.

Everything else builds upon this foundation.

Concepts such as VRAM consumption, quantization, context length, chat templates, retrieval systems, serving engines, and model selection are ultimately consequences of the mechanics governing token generation. Understanding the loop first makes these later topics easier to reason about because they stop appearing as isolated technical concepts and instead become practical implications of how inference works.

This chapter focuses on the model side of LLMs. It introduces inference, tokens, tokenization, context windows, model weights, attention, KV caching, prefill, decode, and next-token prediction. The goal is to build intuition for how modern language models operate before moving on to deployment, hardware, and software considerations.

# Chapter 1: The LLM Inference Loop

## Learning Objectives

By the end of this chapter, readers will be able to:

* Explain the fundamental inference loop used by decoder-only language models.
* Understand the role of tokens during language generation.
* Distinguish between model weights, context, and KV cache.
* Describe how next-token prediction works.
* Explain the relationship between logits, probabilities, and decoding.
* Understand why generation speed is measured in tokens per second.
* Recognize the practical impact of context windows and token counts.
* Understand why tokenization influences memory usage, latency, and model behavior.

## 1.1 Start With the Loop

The most useful way to understand an LLM is to begin with the generation loop itself.

Many people imagine that a model receives a prompt, internally constructs a complete answer, and then reveals that answer to the user. This is not how a decoder-only language model operates.

Instead, generation occurs incrementally.

The model repeatedly predicts a single token. Once that token has been selected, it becomes part of the sequence. The model then predicts another token based on the updated sequence. This process continues until generation stops.

The loop can be described as:

1. Convert text into tokens.
2. Feed tokens into the model.
3. Process the sequence through the Transformer.
4. Use attention to determine which previous tokens are relevant.
5. Compute scores for possible next tokens.
6. Apply a decoding strategy.
7. Select one token.
8. Append the token to the sequence.
9. Update the KV cache.
10. Repeat.

Every response generated by a language model—from a single word to an entire book chapter—is ultimately produced through repeated execution of this loop.

Understanding this process provides the conceptual foundation for everything that follows.

## 1.2 What an LLM Actually Does

Running a model is called "Inference".

Inference is the process of using a trained model to generate predictions from an input sequence.

For a decoder-only language model, inference consists of repeatedly predicting the most appropriate next token based on the tokens already present in the sequence.

This process continues until one of several stopping conditions occurs:

* The model generates a stopping token.
* A predefined token limit is reached.
* The user manually stops generation.

A critical point is that the model is never generating an entire answer simultaneously.

Instead, every new token becomes part of the context that influences future predictions.

As a result, generation is inherently sequential.

The current state of the sequence determines what can be generated next.

This sequential behavior explains why language models can adapt their output continuously as generation progresses.

## 1.3 Weights, Context, and Working Memory

Three concepts are central to understanding inference:

### Model Weights

Model weights contain the "Patterns learned" during training.

They represent the model's accumulated knowledge about language, structure, formatting conventions, and relationships between tokens.

During inference, the weights remain fixed.

The model is not learning from the prompt while generating text. Instead, it is applying the patterns already encoded within its parameters.

Weights determine what the model has learned.

### Context

Context is the information currently visible to the model.

This includes:

* User prompts
* Conversation history
* System instructions
* Previously generated tokens

The context changes continuously throughout generation because each new token is appended to the sequence.

Context determines what the model is currently looking at.

### KV Cache

The Key-Value cache acts as a form of working memory during inference.

Without a KV cache, the model would need to recompute information about all previously processed tokens every time a new token was generated.

Such repeated computation would be inefficient and would dramatically reduce generation speed.

The KV cache stores previously computed information so that subsequent generation steps can reuse it.

This makes long conversations and long-form generation practical.

The weights provide learned knowledge.

The context provides current information.

The KV cache enables efficient continuation.

Together, these components make modern inference feasible.

## 1.4 The Mathematical View

An LLM can be viewed as a learned function.

Its purpose is to map a sequence of tokens to a probability distribution over possible next tokens.

The relationship can be expressed as:

f(theta, sequence) → probability distribution over next token

Where:

* theta represents the model weights.
* sequence represents the prompt and all generated tokens so far.

The output of the model is not text.

The output is a set of scores describing the likelihood of potential next tokens.

Text generation emerges because this prediction process is repeated many times.

Understanding this distinction is important because it highlights what the model is fundamentally doing: predicting what token should come next.

## 1.5 Logits, Probabilities, and Decoding

When the model evaluates a sequence, it produces raw numerical scores for every token in its vocabulary.

These raw scores are known as logits.

Logits are not probabilities.

They represent the model's relative preferences among possible next tokens.

A normalization process converts these logits into probabilities.

The resulting probability distribution indicates how likely each candidate token is relative to all others.

However, probabilities alone do not generate text.

A decoding process must select a specific token.

Decoding transforms a probability distribution into an actual token choice.

Once selected, that token is appended to the sequence, becoming part of the context for the next prediction.

Generation therefore consists of a cycle of prediction and selection.

The model predicts possibilities.

Decoding chooses one possibility.

The loop then continues.

## 1.6 Prefill and Decode

From a user's perspective, generation speed is often experienced in two distinct phases.

### Prefill

Before the first generated token appears, the model must process the entire prompt.

This phase is known as prefill.

During prefill, the model converts the prompt into internal representations and builds the information needed for subsequent generation.

Long prompts increase the amount of work required during this stage.

As a result, users may experience a noticeable pause before output begins.

A document containing thousands of tokens can produce substantial prefill latency even before the model generates its first word.

### Decode

After prompt processing is complete, the model enters the decode phase.

During decode, the model repeatedly predicts and generates one token at a time.

Each generation step updates the KV cache and extends the sequence.

Because decode controls how quickly text appears on screen, users often perceive decode speed as responsiveness.

However, prefill latency can be equally important when working with large prompts.

Understanding the distinction helps explain why long documents may feel slow even on systems that generate tokens rapidly once decoding begins.

## 1.7 Tokens

Language models do not operate directly on human-readable words.

They operate on tokens.

A token is a unit of text represented internally by an integer identifier.

Tokens can represent many different types of textual content.

Examples include:

* Entire words
* Word fragments
* Punctuation symbols
* Whitespace-prefixed text
* Byte-level fallback representations
* Special control markers

A token might correspond to a complete word such as "hello."

It might also correspond to only part of a word.

Long or uncommon words are frequently divided into multiple tokens.

Consequently, words and tokens should never be assumed to be equivalent.

The distinction is fundamental because every major resource consumed by an LLM is measured in tokens rather than words.

## 1.8 Tokenization

Tokenization is the process of converting text into tokens and converting tokens back into text.

The tokenizer serves as the translation layer between human language and the model's internal representation.

Different model families use different tokenizers.

Common approaches include:

* BPE-style tokenizers
* SentencePiece-style tokenizers

Because tokenizers differ, identical text may produce different token counts across different models.

A document that consumes one amount of context in one model may consume significantly more context in another.

This difference affects practical considerations such as memory usage, prompt length, context consumption, and latency.

Tokenization is therefore not merely a preprocessing step. It directly influences how efficiently a model can represent information.

## 1.9 Vocabulary Size

Every tokenizer operates using a vocabulary.

The vocabulary contains all token entries recognized by the tokenizer.

Vocabulary size affects how text is represented.

A larger vocabulary may allow some text patterns to be represented using fewer tokens.

However, vocabulary size also influences other parts of the model, including embedding and output projection structures.

This means vocabulary design involves trade-offs rather than simple improvements.

It also explains why comparing tokens-per-second performance across different model families can be misleading.

Different tokenizers may represent the same text with different numbers of tokens, making direct comparisons imperfect.

## 1.10 Why Tokens Matter

Tokens are the fundamental unit of work during inference.

They determine:

* Context window consumption
* KV cache growth
* Prompt processing cost
* Memory requirements
* Latency
* Efficiency for code and multilingual text
* Correct interpretation of chat formatting markers

Once text is viewed as tokens rather than words, many aspects of LLM behavior become easier to reason about.

Long context stops appearing magical and instead becomes a measurable computational cost.

Every additional token contributes to the work the model must perform.

## 1.11 Context Windows

A context window defines the maximum number of tokens the model can attend to simultaneously.

Modern models support a wide range of context lengths.

Examples include:

* 8K tokens
* 32K tokens
* 128K tokens
* 256K tokens
* Larger server-oriented configurations

However, supported context length should not be confused with practical context length.

A model may technically accept a large context while becoming slower as that limit is approached.

Similarly, coherence may degrade at extreme sequence lengths.

Therefore, context support should be evaluated in terms of actual workloads rather than advertised limits alone.

A context window is not free.

It consumes memory, increases computation, and can influence generation behavior.

Understanding token costs makes context windows easier to evaluate realistically.

## Practical Example

Consider a user submitting a large document for analysis.

The text is first converted into tokens.

The model processes those tokens during prefill.

Internal state is established and the KV cache is populated.

The model then computes logits for possible next tokens.

These logits are transformed into probabilities.

A decoding strategy selects a token.

The selected token is appended to the sequence.

The KV cache is updated.

The model repeats the process.

What appears to the user as a streaming response is actually a sequence of repeated next-token predictions occurring one token at a time.

## Common Misunderstandings

A common misconception is that an LLM writes an entire answer before displaying it. In reality, generation occurs one token at a time.

Another misunderstanding is that tokens are equivalent to words. Tokenization frequently divides words into smaller components, making token counts and word counts fundamentally different measurements.

Many users also assume that large context windows guarantee strong performance throughout the entire range. In practice, supported context length does not necessarily imply equal speed, cost, or quality at every size.

It is also common to think that responsiveness is determined only by generation speed. Prompt processing time can be equally important, particularly when working with long documents.

Finally, the KV cache is often mistaken for stored conversation text. It is actually a storage mechanism for previously computed information that allows efficient continuation of generation.

## Key Takeaways

Modern decoder-only language models operate through a repeated next-token prediction loop.

Text becomes tokens. Tokens are processed by the Transformer. Attention determines relevant context. The KV cache preserves previously computed information. The model predicts a next token, appends it to the sequence, and repeats the process.

Weights determine what the model has learned.

Context determines what it is currently examining.

The KV cache enables efficient inference.

Tokens are the fundamental unit of work, influencing memory usage, latency, context length, and generation cost.

Understanding these mechanics provides the foundation required for reasoning about all higher-level aspects of language model behavior and deployment.

## 1.13 Transformers

Most modern Large Language Models are based on the Transformer architecture. While users typically interact with models through chat interfaces and observe only the generated text, the actual computational machinery responsible for producing those outputs is the Transformer itself.

Most local chat-oriented language models are decoder-only Transformers. A decoder-only Transformer generates text by repeatedly predicting the next token while looking back at tokens that have already appeared in the sequence. Future tokens do not yet exist, so the model can only use information that is already present within the context window.

Everything discussed earlier in this chapter—including tokens, tokenization, model weights, context windows, KV caches, configuration settings, and chat templates—exists to support the computational engine underneath. Those concepts determine what information enters the model and how that information is represented. The Transformer is the system that processes those representations and converts them into predictions.

A useful way to think about the Transformer is as a stack of processing layers. Each layer receives a representation of the sequence, transforms that representation, and passes the result to the next layer. Through repeated transformation, the model gradually builds increasingly useful internal representations that can ultimately be used for next-token prediction.

A simplified Transformer layer contains several major components:

* Token embeddings
* Positional information
* Self-attention
* Feed-forward computation
* Layer normalization
* Residual connections
* Output projection

The relationship between these components can be visualized as follows.

### Token Embeddings

The first step inside the Transformer is the embedding stage.

Language models do not operate directly on token IDs. A token ID is merely an integer identifier. Before meaningful computation can occur, each token ID must be converted into a vector representation.

Embeddings perform this conversion. They transform discrete token identifiers into dense numerical representations that can participate in the mathematical operations performed throughout the network.

Embeddings therefore act as the bridge between symbolic language and numerical computation.

### Positional Information

Token embeddings alone do not contain information about order.

A sequence such as:

> The dog chased the cat

contains many of the same tokens as:

> The cat chased the dog

but the meanings differ because the token order differs.

The Transformer therefore requires positional information.

Many modern language models use Rotary Position Embeddings (RoPE). RoPE encodes position by rotating internal representations according to token location within the sequence. This allows the model to distinguish between tokens appearing at different positions while preserving useful relationships between them.

Without positional information, the model would struggle to understand sequence structure.

### Self-Attention

Once tokens have been embedded and positional information has been incorporated, the model applies self-attention.

Self-attention allows each token representation to examine previous token representations and determine which information is relevant.

Not every earlier token contributes equally to the next prediction. Some tokens may be highly relevant while others contribute very little. Self-attention allows the model to assign different levels of importance to different parts of the sequence.

This mechanism is one of the primary reasons Transformers can maintain coherence across long passages of text.

### Feed-Forward Computation (MLP)

After attention processing, the representation passes through a feed-forward network, often referred to as an MLP.

The MLP performs dense nonlinear computation that transforms the representation generated by attention.

A useful intuition is that attention determines what information matters, while the MLP reshapes that information into a more useful form.

A large fraction of a model's parameters often reside within these feed-forward layers. As models become larger, these layers account for a substantial portion of the total parameter count.

### Layer Normalization and Residual Connections

Modern language models may contain dozens or even hundreds of Transformer layers.

To maintain stable computation across such depth, Transformers use layer normalization and residual connections.

Layer normalization helps stabilize numerical values as they flow through the network.

Residual connections allow information from earlier stages to continue influencing later stages rather than being completely replaced.

Together, these mechanisms help information propagate effectively through deep networks.

### Output Projection

After the sequence has passed through the Transformer stack, the final hidden representation must be converted into next-token scores.

The output projection layer performs this conversion.

The result is a set of logits over the model vocabulary. These logits later become probabilities, which are then used by the decoding process discussed earlier in this chapter.

### Transformer Recap

The Transformer can be summarized as a sequence of repeated operations.

Tokens become vector representations through embeddings. Positional information keeps token order intact. Attention connects relevant parts of the sequence. Feed-forward networks reshape representations. Layer normalization and residual pathways stabilize computation. Output projection converts the final hidden state into logits over the vocabulary.

## 1.14 Attention

Attention is the mechanism that allows a token to determine which earlier tokens matter for the next prediction.

It is also one of the primary reasons local inference can become memory-sensitive.

When a model processes a sequence, it must preserve information about previous tokens so that future tokens can attend to them. As context length grows, the amount of information that must be stored and accessed can increase substantially.

For local inference workloads, attention design therefore has a direct impact on memory usage, context length behavior, and KV cache growth.

### Figure 1.10 — Conceptual View of Self-Attention

```text
Sequence

[The] [dog] [chased] [the] [cat]

                         ▲
                         │
                  Current Token

                         │
         ┌───────────────┼───────────────┐
         │               │               │
         ▼               ▼               ▼

      [dog]         [chased]         [the]

      High           Medium           Low
    Importance     Importance      Importance
```

The diagram above illustrates a simplified view of attention. The current token does not treat every previous token equally. Instead, it determines which earlier information is most relevant to the prediction task.

### Multi-Head Attention (MHA)

Classic Multi-Head Attention (MHA) stores separate key-value state for multiple attention heads.

Each head can learn different relationships within the sequence and focus on different patterns of information. This flexibility makes MHA highly expressive.

However, maintaining separate key-value information for many heads causes the KV cache to grow rapidly. As context length increases, memory requirements can become substantial.

This is one reason why long-context inference can become expensive.

### Multi-Query Attention (MQA)

Multi-Query Attention (MQA) reduces memory usage by allowing multiple query heads to share a single key-value head.

This design significantly reduces KV cache growth and improves memory efficiency.

The trade-off is that MQA may be less expressive than full Multi-Head Attention because fewer independent key-value representations are available.

### Grouped-Query Attention (GQA)

Grouped-Query Attention (GQA) provides a middle ground between MHA and MQA.

Instead of sharing a single key-value head across all queries, query heads are organized into groups. Each group shares key-value information internally.

This approach preserves more flexibility than MQA while consuming less memory than full MHA.

As a result, GQA has become a common design choice in many modern local language models.

### Figure 1.11 — Comparing MHA, MQA, and GQA

```text
MHA

Q1 ──► KV1
Q2 ──► KV2
Q3 ──► KV3
Q4 ──► KV4


MQA

Q1 ──┐
Q2 ──┤
Q3 ──┤──► Shared KV
Q4 ──┘


GQA

Q1 ──┐
Q2 ──┘──► KV Group A

Q3 ──┐
Q4 ──┘──► KV Group B
```

This comparison highlights the trade-off between flexibility and memory efficiency. MHA provides maximum flexibility but requires the largest KV cache. MQA minimizes memory consumption through complete sharing. GQA balances the two approaches and has therefore become popular in many modern local models.

### Attention Kernels and Runtime Performance

Attention architecture is only part of the performance story.

Modern runtimes often use optimized attention kernels such as FlashAttention and SDPA-style implementations. These kernels reduce attention memory traffic and improve hardware utilization.

Reducing memory movement is important because memory bandwidth can become a major bottleneck during inference.

As a result, two systems running the same model on identical hardware can exhibit dramatically different performance characteristics depending on the quality of their attention implementations.

A runtime with highly optimized attention kernels may achieve significantly better performance than one without such optimizations.

### Why Attention Design Matters

Attention design has practical consequences beyond architecture diagrams.

Two models may both contain seven billion parameters and advertise identical context lengths, yet behave very differently during long-context inference.

A 7B model using full Multi-Head Attention at 128K context may consume enough memory to exhaust a 24 GB GPU.

A 7B model using Grouped-Query Attention with the same advertised context length may fit comfortably on the same hardware.

This illustrates an important principle: parameter count is not the entire story.

When comparing models, attention type, KV head configuration, supported context length, and runtime support are often just as important as model size itself.

When evaluating local language models, attention architecture should therefore be considered alongside parameter count rather than after it.

## 1.15 KV Cache

The KV cache is the model's working memory during generation.

Earlier in this chapter, we discussed how a decoder-only language model generates text one token at a time. Every newly generated token must be able to attend to previous tokens in the sequence. If the model were forced to recompute the attention state for the entire conversation every time it generated a new token, inference would become prohibitively expensive.

The KV cache exists to prevent this.

Rather than recomputing attention information from scratch for every generation step, the model stores key and value attention states associated with previously processed tokens. These stored states can then be reused when generating future tokens.

The result is a dramatic reduction in computational work during generation.

Without a KV cache, modern language model inference would be extremely inefficient. With a KV cache, generation becomes practical. However, this efficiency comes with a cost: memory consumption.

The size of the KV cache grows as more tokens enter the context window. A useful approximation for KV cache memory consumption is:

```text
KV Cache Size ∝

tokens × layers × kv_heads × head_dim × precision × 2
```

The final multiplier exists because the cache stores both keys and values.

Each additional token increases the amount of cached information. As conversations grow longer, the cache continues to grow alongside them.

### Figure 1.12 — KV Cache Growth

```text
Conversation Length
        │
        │
        │
        │
        ▼

1K Tokens   → Small KV Cache

4K Tokens   → Larger KV Cache

16K Tokens  → Much Larger KV Cache

32K Tokens  → Very Large KV Cache

64K Tokens  → Extremely Large KV Cache
```

The exact size depends on model architecture, attention design, precision format, and runtime implementation, but the overall relationship remains the same: more tokens require more cache memory.

### KV Cache in Practice

A useful rule of thumb for older Llama-style 7B models using Multi-Head Attention and FP16 KV storage is approximately:

```text
~0.5 MiB per token
```

Under this approximation:

```text
4K Tokens  ≈ 2 GiB KV Cache

32K Tokens ≈ 16 GiB KV Cache
```

These figures illustrate why context length can become a practical deployment constraint.

A model may fit comfortably into memory when idle, yet encounter memory pressure once large documents or long conversations are loaded.

This is because model weights are only one part of the memory budget.

The KV cache consumes memory independently of the weights and continues growing as context length increases.

### GQA and MQA Effects on KV Cache Size

Attention architecture significantly influences cache growth.

As discussed in the previous section:

* MHA stores separate key-value state for many heads.
* MQA allows query heads to share a single key-value representation.
* GQA organizes heads into shared groups.

Because MQA and GQA reduce the amount of stored key-value information, they can substantially reduce KV cache requirements.

This is one reason modern local models often prefer GQA or MQA-style architectures for long-context workloads.

The model may have the same parameter count as an older architecture while consuming significantly less cache memory.

### KV Cache Precision

The precision used to store cache values also influences memory consumption.

Many modern runtimes support:

* FP16 KV cache
* FP8 KV cache
* INT8 KV cache

For local users, FP8 and INT8 KV storage often provide substantial memory savings while maintaining practical performance.

The source material recommends treating approximately 8-bit KV storage as a practical compression floor for many local workloads in 2026.

### Sub-8-Bit KV Cache

Reducing KV cache precision below 8 bits is an active area of research.

Examples referenced in the source include:

* [KIVI Paper](https://arxiv.org/abs/2402.02750?utm_source=chatgpt.com)
* [KVQuant Paper](https://arxiv.org/abs/2401.18079?utm_source=chatgpt.com)

These systems demonstrate that 2-bit to 4-bit KV storage can function effectively when combined with specialized algorithms, calibration procedures, and custom kernels.

However, research results should not be confused with a simple runtime setting.

A carefully engineered research implementation is very different from enabling an aggressively compressed KV cache without validation.

For workloads where exact token recall matters—such as coding, tool usage, JSON generation, long-context retrieval, or structured outputs—sub-8-bit cache configurations should be benchmarked carefully before deployment.

### KV Cache Quantization vs Speculative Decoding

Another common source of confusion is the relationship between KV cache compression and speculative decoding systems.

These technologies solve different problems.

KV cache quantization reduces memory consumption.

Speculative decoding attempts to improve generation speed.

Examples referenced in the source include:

* [DFlash Paper](https://arxiv.org/abs/2602.06036?utm_source=chatgpt.com)
* [DDTree Paper](https://arxiv.org/abs/2604.12989?utm_source=chatgpt.com)

These approaches accelerate generation by drafting future tokens and verifying them.

They can reduce decode latency, but they do not eliminate the memory requirements associated with the KV cache.

The cache must still exist regardless of how efficiently tokens are generated.

### Why Models Sometimes Crash on Long Documents

A common misunderstanding among local users occurs when a model loads successfully but crashes after a large document is added.

The immediate assumption is often that the model itself does not fit into memory.

In many cases, the model weights fit perfectly.

The problem is that the working memory does not.

As context length grows, the KV cache expands. Eventually, the combined memory requirements of the weights and cache exceed available resources.

The weights fit.

The working memory does not.

## 1.16 Prefill and Decode

Language model inference operates in two distinct performance regimes:

* Prefill
* Decode

Although users often experience them as part of a single interaction, they represent fundamentally different types of computation.

Understanding this distinction helps explain why some workloads feel slow before generation begins while others feel slow during generation.

### Figure 1.13 — Prefill and Decode

```text
User Prompt
      │
      ▼

┌─────────────┐
│   Prefill   │
└─────────────┘

(Process Existing Tokens)

      │
      ▼

First Generated Token

      │
      ▼

┌─────────────┐
│   Decode    │
└─────────────┘

(Token-by-Token Generation)

      │
      ▼

Final Response
```

### Prefill

Prefill is the stage in which the model processes the prompt supplied by the user.

If a user submits a large document, every token within that document must be processed before generation can begin.

For example:

```text
20,000-token document
        ↓
20,000 tokens processed
        ↓
First output token appears
```

The time spent waiting for the first generated token is often dominated by prefill.

Although prefill can be computationally expensive, it benefits from substantial parallelism. Modern GPUs can process many prompt tokens simultaneously, making prefill relatively efficient compared with purely sequential workloads.

Nevertheless, very large prompts can still create noticeable latency.

### Decode

Decode begins after prompt processing is complete.

During decode, the model generates new tokens one at a time.

Each generated token becomes part of the sequence and influences future generation.

Because every new token depends on previously generated tokens, decode is inherently sequential.

This is the phase responsible for the familiar streaming effect in chat applications.

The speed of decode often determines whether a model feels responsive during generation.

### Long Prompts vs Long Answers

Different workloads stress different parts of the inference pipeline.

Long prompts primarily increase prefill cost.

Long responses primarily increase decode cost.

Long conversations increase both.

This relationship can be summarized as:

```text
Long Prompt      → Prefill Cost

Long Answer      → Decode Cost

Long Conversation
      ↓
Prefill + Decode + KV Growth
```

### Conversation Growth and Cache Growth

In a chat application, every turn contributes additional tokens to the conversation history.

As history grows, the KV cache grows with it.

A conversation containing 16K tokens carries the memory cost of those 16K tokens throughout subsequent generation.

This is one reason long-running conversations eventually slow down or exhaust memory resources.

Infinite chat history is not free.

The cache continues growing until memory becomes a limiting factor.

---

## 1.17 Decoding

After the model produces logits, text has not yet been generated.

The model has only assigned scores to possible next tokens.

Decoding is the process that converts those scores into an actual token selection.

Once a token is selected, it is appended to the context and the inference loop repeats.

### Figure 1.14 — Decoding Loop

```text
Logits
   │
   ▼

Probabilities
   │
   ▼

Decoding Policy
   │
   ▼

Selected Token
   │
   ▼

Append to Context
   │
   ▼

Repeat
```

The decoding process does not modify model weights.

Instead, it determines how the model's predictions are translated into generated text.

Different decoding strategies can significantly influence output style and behavior.

### What Decoding Controls

Decoding settings answer three practical questions:

#### Randomness

How much variation should be allowed?

Higher randomness permits more diverse outputs.

Lower randomness encourages consistency and predictability.

#### Tail Reach

How far into lower-probability tokens should the sampler be allowed to explore?

Restricting exploration produces safer and more focused outputs.

Allowing deeper exploration can increase creativity and diversity.

#### Boundaries

What mechanisms prevent undesirable behavior?

Examples include:

* Infinite loops
* Excessive repetition
* Schema violations
* Runaway generation

Boundary controls help keep outputs within desired constraints.

### Practical Decoding Approaches

Different tasks often benefit from different decoding behavior.

For precise work:

* Lower temperature
* Shorter generation limits
* Explicit stop sequences
* Constrained decoding

These settings help maintain consistency and structure.

For creative work:

* Higher temperature
* Higher diversity
* Multiple candidate generations

These settings encourage exploration and variety.

For coding tasks:

* Conservative first-pass generation
* Alternative samples when exploring solutions

This approach balances reliability with experimentation.

### Greedy Decoding

A common assumption is that selecting the highest-probability token at every step always produces the best result.

In practice, this is not necessarily true.

Greedy decoding can become brittle.

Because it never explores alternatives, it may become trapped in repetitive patterns or converge on generic responses.

Deterministic settings are often useful for evaluations and reproducibility.

For brainstorming and ideation, allowing controlled exploration can produce more varied and useful outputs.

The appropriate decoding strategy depends on the task being performed rather than a universal notion of correctness.

## 1.18 What a Model Package Contains

A runnable local language model is more than a single weight file.

This is one of the first lessons people learn when moving from hosted APIs to local inference. A beginner often downloads a model file, sees that it occupies several gigabytes of storage, and assumes that file contains everything required to run the model. In reality, modern language models are distributed as packages composed of multiple components that must work together correctly.

A model package typically contains:

* Architecture and configuration
* Weights
* Tokenizer
* Chat template
* Generation configuration
* License and model card

Although the weights are usually the largest component, they are only one piece of the system.

A useful way to think about a model package is to compare it to a computer program.

The executable code is important, but the program also depends on configuration files, libraries, data formats, and documentation. Removing any of those pieces can cause problems even if the executable itself remains intact.

Language models work in a similar way.

The weights contain learned knowledge, but the tokenizer determines how text is represented, the configuration describes how the model is constructed, and the chat template determines how conversations are formatted.

If any of these components are incorrect, the same weights can appear broken.

### Figure 1.15 — Components of a Model Package

```text
Model Package
      │
      ├── Architecture / Config
      ├── Weights
      ├── Tokenizer
      ├── Chat Template
      ├── Generation Config
      └── License / Model Card
```

A model package should therefore be viewed as a collection of components that travel together rather than a single file.

---

## 1.19 Architecture and Configuration

The architecture configuration describes how the model is built.

If the weights contain learned knowledge, the architecture defines the structure that uses that knowledge.

An important distinction is that architecture and weights are not the same thing.

A useful mental model is:

```text
Model
=
Architecture
+
Weights
```

The architecture specifies:

* How many layers exist
* How large internal representations are
* Which attention mechanism is used
* How positional information is encoded
* How many tokens exist in the vocabulary
* Which special tokens are reserved
* How much context the model supports

Without this information, a runtime would not know how to interpret the weight tensors stored within the model.

### Layer Count

A Transformer is composed of layers stacked on top of one another.

Each layer receives information from the previous layer, transforms that information, and passes the result forward.

A simplified view looks like:

```text
Input
  │
Layer 1
  │
Layer 2
  │
Layer 3
  │
 ...
  │
Layer N
  │
Output
```

The number of layers is known as the layer count.

Increasing the layer count allows information to undergo more transformations before a prediction is produced.

More layers generally allow more sophisticated processing, but they also increase:

* Memory consumption
* Computational cost
* Inference latency

Layer count is therefore one of the primary architectural characteristics used to distinguish model families.

---

### Hidden Size

Earlier in the chapter, token embeddings were introduced as vector representations.

Those vectors contain many numerical dimensions.

For example:

```text
"Paris"

↓

[0.12, -0.81, 1.54, 0.39, ...]
```

The number of dimensions in these vectors is known as the hidden size.

The hidden size determines how much information can be represented inside the model at each layer.

A useful analogy is to imagine hidden size as the width of a highway.

A wider highway allows more information to travel simultaneously.

A narrower highway limits how much information can be represented at once.

Larger hidden sizes generally provide greater representational capacity, but they also require more memory and computation.

This is one reason large models become expensive to run.

---

### Attention Type

The attention architecture determines how attention state is stored and reused.

Common attention types include:

* Multi-Head Attention (MHA)
* Multi-Query Attention (MQA)
* Grouped-Query Attention (GQA)

As discussed previously, attention architecture strongly influences:

* KV cache size
* Memory requirements
* Long-context performance
* Runtime efficiency

Two models with identical parameter counts may exhibit very different memory behavior if they use different attention mechanisms.

This is why attention type is part of the model configuration.

---

### RoPE Settings

RoPE stands for Rotary Position Embeddings.

RoPE provides positional information that tells the model where tokens appear within a sequence.

Consider the sentences:

```text
The dog chased the cat

The cat chased the dog
```

Both contain many of the same words, yet their meanings differ because the order differs.

The model therefore needs a way to encode position.

RoPE accomplishes this by rotating internal representations according to token position.

Conceptually:

```text
Position 1 → Rotate by θ

Position 2 → Rotate by 2θ

Position 3 → Rotate by 3θ
```

The exact mathematics involve trigonometric rotations applied within vector space, but the key idea is simple: RoPE allows the model to distinguish tokens that appear in different positions.

RoPE settings influence:

* Context length behavior
* Long-context scaling
* Positional representation

As context windows grow larger, RoPE configuration becomes increasingly important.

---

### Vocabulary Size

The tokenizer operates using a vocabulary.

The vocabulary contains every token recognized by the model.

Examples include:

* Whole words
* Word fragments
* Punctuation
* Special markers
* Control tokens

Vocabulary size affects several architectural components.

One important relationship is:

```text
Vocabulary Size
        ×
Hidden Size
        =
Embedding Table Size
```

A larger vocabulary may represent some text more efficiently because more token entries are available.

However, larger vocabularies also require larger embedding structures.

Vocabulary design therefore involves trade-offs rather than universally better choices.

---

### Special Tokens

Special tokens are tokens reserved for structural purposes.

Unlike ordinary text, they carry predefined meaning.

Examples include:

* Beginning-of-sequence markers
* End-of-sequence markers
* System-role markers
* User-role markers
* Assistant-role markers
* Tool-use markers

Special tokens help the model understand conversation structure and generation boundaries.

Later sections will examine these tokens in detail.

---

### Context Length

Context length specifies the maximum number of tokens that can participate in attention simultaneously.

Examples include:

```text
8K
32K
128K
256K
```

Context length is an architectural capability.

However, supporting a context length and using it efficiently are different things.

A model may technically support very large contexts while experiencing reduced speed or increased memory consumption at those lengths.

---

## 1.20 Weights

The weights are the learned parameters of the model.

When people download a model, they are usually downloading billions of numerical values that were learned during training.

These values are called weights because they influence how information flows through the network.

A useful mental model is:

```text
Architecture
=
Skeleton

Weights
=
Knowledge
```

The architecture determines how computation occurs.

The weights determine what the model has learned.

Both are required.

Without weights, the architecture is merely an empty structure.

Without architecture, the weights cannot be interpreted.

### What Is a Weight?

At its most basic level, a weight is a number.

Consider a simplified computation:

```text
Input
  ×
Weight
  =
Contribution
```

A neural network performs enormous numbers of operations similar to this.

Individual weights influence how strongly information flows between representations.

During training, these values are adjusted repeatedly until the network becomes capable of making useful predictions.

The final collection of learned values becomes the model's weights.

---

### Why Weights Store Knowledge

A common misconception is that model weights contain stored documents, sentences, or facts.

They do not.

Weights are not text files.

They are not databases.

They are not lookup tables.

Instead, they are learned numerical parameters that encode statistical patterns observed during training.

Knowledge emerges from the collective behavior of billions of weights interacting with one another.

This is why inspecting individual weights is generally not useful. Meaning exists in the network as a whole rather than in isolated values.

---

### The Scale of Modern Weights

Modern language models contain enormous numbers of parameters.

Examples include:

```text
2 Billion Parameters

7 Billion Parameters

14 Billion Parameters

70 Billion Parameters
```

Each parameter is a learned numerical value.

As parameter counts increase, the amount of stored information generally increases as well.

However, larger parameter counts also increase:

* Storage requirements
* Memory usage
* Computational cost

Parameter count is therefore an important characteristic, but it is not the only factor determining model quality.

---

### Architecture Versus Weights

One of the most important concepts for local inference is the distinction between architecture and weights.

Consider the following:

```text
Same Architecture
       +
Different Weights
       =
Different Model
```

The architecture may remain unchanged while entirely different weights are loaded.

In such cases, the resulting model can behave very differently despite sharing the same structural design.

This distinction becomes important when downloading models, applying fine-tunes, or converting between formats.

---

## 1.21 Weight Storage Formats

The source material mentions several common formats used to distribute model weights:

* safetensors
* GGUF
* GPTQ
* AWQ
* EXL2

Although all contain model parameters, they are designed for different purposes.

### safetensors

Safetensors is a weight storage format widely used throughout the Hugging Face ecosystem.

Its purpose is straightforward:

* Store model parameters efficiently.
* Load parameters safely.
* Avoid security issues associated with executable serialization formats.

Many modern model releases provide safetensors files as their primary distribution format.

When using Transformers-based workflows, safetensors is one of the most common formats encountered.

---

### GGUF

GGUF is a format commonly associated with llama.cpp and related local inference tools.

Unlike a simple tensor container, GGUF typically stores:

* Weights
* Quantization metadata
* Configuration information
* Runtime-relevant metadata

This makes GGUF particularly convenient for local deployment.

Many users choose GGUF because it allows a model to be packaged into a format that local runtimes can load directly.

---

### GPTQ

GPTQ is a quantized weight format.

Quantization reduces memory requirements by representing weights using fewer bits.

The objective is to make large models easier to run on limited hardware.

A simplified relationship is:

```text
Lower Precision
        ↓
Smaller Model
        ↓
Lower Memory Usage
```

The trade-off is that aggressive compression can sometimes reduce model quality.

GPTQ attempts to balance compression and performance.

---

### AWQ

AWQ stands for Activation-Aware Weight Quantization.

Like GPTQ, it is designed to reduce memory usage.

However, AWQ uses a different quantization strategy intended to preserve model quality while compressing weights.

The goal is not merely to make weights smaller, but to do so while maintaining useful behavior.

---

### EXL2

EXL2 is a format commonly associated with ExLlama-style runtimes.

It is designed for efficient GPU inference and aggressive compression.

EXL2 is often used by local users seeking high performance from limited hardware resources.

As with other quantized formats, the goal is to reduce memory requirements while preserving practical model quality.

---

### Why Multiple Formats Exist

A common question is:

> Why are there so many formats?

The answer is that different formats optimize for different goals.

Some prioritize compatibility.

Some prioritize safety.

Some prioritize memory efficiency.

Some prioritize inference speed.

As a result, the same model may be available in several formats depending on the runtime and hardware being targeted.

Understanding these formats helps explain why downloading a model is often only the first step. The format must also be compatible with the runtime that will execute it.

## 1.22 Tokenizers

Earlier in this handbook, tokenization was introduced as the process that converts text into tokens and tokens back into text. At first glance, tokenization may appear to be a simple preprocessing step. In reality, the tokenizer is one of the most important components of a language model.

A model never sees raw text.

When a user types:

```text id="9ph6xl"
Explain KV cache.
```

the model does not directly process those characters.

Instead, the tokenizer converts the text into a sequence of token identifiers.

Conceptually:

```text id="ph4nce"
Text
  │
  ▼
Tokenizer
  │
  ▼
Token IDs
  │
  ▼
Model
```

Every operation performed by the Transformer occurs on token representations rather than on raw text.

For this reason, the tokenizer should be viewed as part of the model rather than a separate utility.

A model trained with one tokenizer cannot simply be paired with a completely different tokenizer. The mapping between text and token IDs must remain consistent with the mapping used during training.

---

### Why Tokenizers Matter

Tokenizers influence:

* Context length usage
* Memory consumption
* Prompt processing cost
* Vocabulary size
* Multilingual efficiency
* Code generation efficiency

Consider the word:

```text id="bjlwm8"
internationalization
```

One tokenizer might represent this as:

```text id="xh8zsm"
international
ization
```

while another tokenizer might represent it as:

```text id="9nqz5w"
inter
national
ization
```

The resulting token count differs.

Because memory, latency, context windows, and KV cache growth are measured in tokens rather than words, tokenization directly affects practical performance.

---

### Token IDs

Every token has a numerical identifier.

For example:

```text id="ybp1k8"
"hello"
   ↓
15339
```

The actual values vary by tokenizer, but the principle remains the same.

The Transformer does not process words.

It processes token IDs.

Those token IDs are then converted into embeddings before entering the model.

Conceptually:

```text id="d7s1zi"
Text

"hello"

      ↓

Token ID

15339

      ↓

Embedding Vector

[0.12, -0.84, ...]
```

This conversion process forms the bridge between language and numerical computation.

---

### Vocabulary and Token IDs

The tokenizer vocabulary contains every token recognized by the model.

A simplified vocabulary might look like:

```text id="u7xlru"
0  → [PAD]
1  → [BOS]
2  → [EOS]
3  → hello
4  → world
5  → .
...
```

Real vocabularies often contain tens or hundreds of thousands of entries.

The vocabulary and tokenizer are tightly coupled.

Changing the vocabulary changes the meaning of token IDs.

As a result, tokenizers must remain compatible with the models that use them.

---

### Tokenizers Are Part of the Model

A common beginner mistake is to think:

> "The model is the weights."

The weights are important, but the tokenizer is equally critical.

The following combination does not work:

```text id="1ee1ga"
Model A Weights
+
Tokenizer B
```

unless both were designed to operate together.

The tokenizer determines how text is represented internally.

If the representation changes, the model may receive inputs it was never trained to interpret.

---

## 1.23 Chat Templates

Once text has been tokenized, the model still needs to understand the structure of a conversation.

This is where chat templates become important.

A chat template is a formatting scheme that tells the model:

* Which text is a system instruction
* Which text comes from the user
* Which text comes from the assistant
* Where responses begin
* Where responses end

A useful way to think about chat templates is as communication protocols.

Humans naturally understand conversational roles.

Language models do not.

Those roles must be explicitly represented in the sequence.

---

### Why Chat Templates Exist

Consider the following conversation:

```text id="3lh4ux"
You are a helpful assistant.

Explain KV cache.
```

A human reader immediately understands:

* The first sentence is an instruction.
* The second sentence is a question.

A language model does not automatically know that.

Chat templates provide explicit structure.

For example:

```text id="uvjlwm"
<|system|>
You are a helpful assistant.

<|user|>
Explain KV cache.

<|assistant|>
```

The role markers identify who is speaking.

Without these markers, the model may interpret the conversation differently than intended.

---

### Conversation Structure

Conceptually, a chat template transforms conversation into structured data.

```text id="ol1h2q"
System Message
       │
       ▼
User Message
       │
       ▼
Assistant Response
```

The model learns this structure during training.

When the runtime applies the correct template, the conversation resembles the data distribution the model encountered during training.

---

### Why Templates Affect Performance

The same model can appear intelligent with one template and broken with another.

For example:

```text id="jlwmjk"
Correct Template
      ↓
Expected Behavior
```

versus:

```text id="bf6mwj"
Incorrect Template
      ↓
Confused Behavior
```

Many cases of poor model performance are actually template problems rather than model problems.

---

## 1.24 Understanding ChatML

One of the most common template families is ChatML.

ChatML uses special role markers to distinguish different participants in a conversation.

A typical ChatML conversation looks like:

```text id="v5mzh4"
<|system|>
You are a helpful assistant.

<|user|>
Explain KV cache.

<|assistant|>
```

To a human, these appear to be labels.

To the model, they are special tokens with learned meaning.

The model does not interpret:

```text id="8o7qu5"
<|system|>
```

as ordinary text.

Instead, it interprets the token as a structural marker indicating that the following content belongs to the system role.

The same applies to:

```text id="uw7w84"
<|user|>
```

and

```text id="0lm4kr"
<|assistant|>
```

These markers are part of the language the model was trained to understand.

---

### System Messages

System messages contain high-level instructions.

Example:

```text id="44m8o4"
<|system|>
You are a helpful assistant.
```

The purpose of the system role is to define behavior.

It tells the model how it should approach the conversation.

---

### User Messages

User messages represent requests.

Example:

```text id="vlsb5v"
<|user|>
Explain KV cache.
```

The model learns that text appearing after the user marker represents a request or input from the user.

---

### Assistant Messages

Assistant messages indicate where the model's response begins.

Example:

```text id="jlwmg8"
<|assistant|>
```

When generation starts after this marker, the model understands that it should produce assistant-style text.

---

### ChatML Flow

```text id="krm1rv"
<|system|>
      │
      ▼
<|user|>
      │
      ▼
<|assistant|>
      │
      ▼
Generated Response
```

This structure is one of the most common conversation formats used by modern chat models.

---

## 1.25 BOS and EOS Tokens

Two of the most important special tokens in language models are BOS and EOS.

These tokens define sequence boundaries.

---

### BOS — Beginning of Sequence

BOS stands for Beginning of Sequence.

It marks the start of a token sequence.

Conceptually:

```text id="67vzcw"
[BOS]

Explain KV cache.
```

The model learns that a new sequence begins after this token.

BOS provides an anchor point that tells the model where generation starts.

---

### EOS — End of Sequence

EOS stands for End of Sequence.

It marks the end of a sequence.

Conceptually:

```text id="cw8jwn"
Explain KV cache.

[EOS]
```

The model learns that generation should stop when this token appears.

EOS helps define boundaries and prevents endless continuation.

---

### Why EOS Matters

Suppose the model generates:

```text id="ggm59l"
The KV cache stores attention state.
```

If EOS is generated afterward:

```text id="6wpkrn"
The KV cache stores attention state.

[EOS]
```

generation stops.

Without EOS, generation may continue indefinitely until a separate limit intervenes.

This is one reason why EOS handling is critical.

---

### Sequence Boundary Diagram

```text id="vfz4k8"
[BOS]
   │
User Prompt
   │
Assistant Response
   │
[EOS]
```

These markers provide structure that helps the model understand where sequences begin and end.

---

## 1.26 Template Families

Not all models use the same chat template.

Different model families are trained using different formats.

As a result, one model may expect ChatML while another expects an entirely different structure.

---

### INST Templates

A common example is the INST-style format.

For example:

```text id="s2qifj"
[BOS] [INST]
Explain KV cache.
[/INST]
```

This format uses instruction delimiters rather than explicit role markers.

The model learns that content inside the instruction block represents a user request.

Although the syntax differs, the underlying purpose remains the same: provide structure.

---

### Chat Templates vs Instruction Templates

Conceptually:

```text id="hmv5xw"
ChatML
      ↓
Role-Based Structure
```

versus:

```text id="r5z0m2"
INST
      ↓
Instruction-Based Structure
```

Both approaches provide information about how text should be interpreted.

They simply encode that information differently.

---

### Tool-Oriented Templates

Some models include additional markers for tool use.

Conceptually:

```text id="is0u6x"
System
User
Assistant
Tool
Assistant
```

These templates allow the model to distinguish tool outputs from normal conversation text.

---

### Reasoning-Oriented Templates

Some models introduce additional markers related to reasoning or intermediate computation.

The exact syntax varies between model families.

The key principle remains unchanged:

Special tokens provide structure.

The model learns to associate that structure with specific behaviors.

---

## 1.27 Why Templates Break Models

A common misconception among new local users is that all chat models accept the same prompt format.

They do not.

Consider a model trained on ChatML:

```text id="yuxnkt"
<|system|>
...
<|user|>
...
<|assistant|>
```

If that model receives an INST template instead:

```text id="xvt5eh"
[INST]
...
[/INST]
```

the structure no longer matches the format encountered during training.

The result may be:

* Reduced instruction following
* Strange formatting
* Repetition
* Lower-quality answers
* Unexpected behavior

The weights have not changed.

The tokenizer has not changed.

The hardware has not changed.

Only the template changed.

Yet behavior may change dramatically.

---

### Template Mismatch

A useful mental model is:

```text id="1x9nnv"
Correct Template
        ↓
Expected Behavior
```

```text id="qsm2lr"
Incorrect Template
        ↓
Unexpected Behavior
```

When evaluating a model, always verify:

* The tokenizer is correct.
* The chat template is correct.
* BOS tokens are correct.
* EOS tokens are correct.
* Role markers are correct.

A surprising number of "bad model" reports are actually formatting problems.

The template is effectively part of the model.

If the template is wrong, you are not truly testing the model that was trained.

## 1.28 Model Types

Not all language models are trained for the same purpose.

One of the most common mistakes among newcomers is assuming that every model behaves like a chatbot. In reality, modern language models are often optimized for different tasks, workflows, and interaction styles.

Two models may have identical parameter counts, use the same architecture, and even share a common base model, yet behave very differently because they have been trained for different objectives.

When evaluating a model, one of the first questions to ask is:

> What type of model is this?

Common categories include:

* Base models
* Instruct models
* Chat models
* Reasoning models
* Tool-tuned models

Understanding these categories helps explain why some models excel at conversation, others at coding, and others at structured tool use.

---

### Base Models

A base model is the closest representation of the raw pretrained model.

It has learned language patterns from training data but has not necessarily been optimized for assistant-style interactions.

Conceptually, a base model is a text continuation engine.

If given:

```text id="r3h1kb"
The capital of France is
```

the model attempts to predict what text most naturally follows.

It is not necessarily trying to be helpful.

It is trying to continue the sequence.

For example:

```text id="5qgh2o"
The capital of France is Paris.
It has a population of...
```

The model simply continues generating text.

### Base Model Behavior

A base model generally thinks in terms of completion rather than instruction following.

Conceptually:

```text id="7g0kdi"
Prompt
   ↓
Text Continuation
```

rather than:

```text id="20wr3f"
Prompt
   ↓
Helpful Assistant Response
```

This distinction is important.

A base model may possess substantial knowledge while still producing poor conversational results because it was not specifically optimized for assistant behavior.

---

### Instruct Models

An instruct model is designed to follow instructions more effectively.

During additional training, the model learns patterns associated with requests, commands, and tasks.

Examples include:

```text id="hy73hc"
Summarize this article.

Explain KV cache.

Write a Python function.
```

Rather than simply continuing text, the model learns that it should attempt to satisfy the user's request.

Conceptually:

```text id="v2vwxt"
Instruction
      ↓
Task Completion
```

Instruction tuning is one of the reasons modern language models feel substantially more useful than earlier completion-oriented systems.

---

### Chat Models

Chat models are optimized for conversational interaction.

In addition to instruction following, they learn:

* Conversational turn-taking
* Role separation
* Multi-turn interactions
* Assistant-style responses

A chat model understands structures such as:

```text id="m6mxni"
System
User
Assistant
User
Assistant
```

and learns to maintain conversational consistency across multiple turns.

Because chat models are trained using specific conversation formats, correct chat templates become especially important.

---

### Reasoning Models

Reasoning models are optimized for tasks requiring longer chains of thought, multi-step analysis, or extended problem solving.

Examples include:

* Mathematics
* Complex coding
* Planning
* Logical analysis

Reasoning-oriented models are often designed to spend more computation on difficult problems before producing a final answer.

Conceptually:

```text id="9ll4b5"
Question
    ↓
Extended Internal Reasoning
    ↓
Answer
```

The exact implementation varies between model families, but the goal remains the same: improve performance on tasks requiring multiple reasoning steps.

---

### Tool-Tuned Models

Tool-tuned models are optimized to interact with external tools.

Rather than responding only with text, they learn structured outputs that can trigger external actions.

Examples include:

* Function calls
* JSON schemas
* Tool invocation formats
* API requests

Conceptually:

```text id="n7ob1f"
User Request
      ↓
Tool Selection
      ↓
Tool Output
      ↓
Final Response
```

These models are frequently used in agent systems and automation workflows.

## 1.29 What Local Really Means

The phrase "local LLM" is often used casually, but it can mean different things to different people.

Many newcomers assume that local simply means:

> "The model runs on my computer."

While technically true, this definition is incomplete.

Running a model locally involves several interacting components:

* Model weights
* Runtime software
* Hardware resources
* Tokenizer
* Chat template
* Configuration settings

A successful local deployment depends on all of them.

### Figure 1.17 — Components of Local Inference

```text id="1jkv9q"
Model Weights
      │
      ▼
Tokenizer
      │
      ▼
Runtime
      │
      ▼
Hardware
      │
      ▼
Generated Output
```

If any component is misconfigured, the overall experience may suffer even when the model itself is capable.

---

### Local Does Not Mean Unlimited

A common misconception is that local inference removes all constraints.

In reality, local systems remain limited by:

* VRAM
* System memory
* Storage
* Memory bandwidth
* Runtime efficiency
* Context length

Every token generated consumes computational resources.

Every token added to context consumes memory.

Every conversation increases KV cache requirements.

The underlying mechanics discussed throughout this handbook still apply.

---

### Local Does Not Mean Every Model Fits

A model may exist publicly without being practical for a particular system.

For example:

```text id="0dd6az"
70B Model
     ↓
Requires More Resources
```

compared with:

```text id="9b7jv2"
7B Model
     ↓
Requires Fewer Resources
```

Choosing a model therefore involves balancing:

* Capability
* Memory usage
* Latency
* Hardware constraints

The largest model is not always the best choice.

The best model is often the one that fits comfortably within available resources.

---

### Local Does Not Eliminate Architecture Constraints

Running locally does not change how Transformers work.

The model still obeys:

* Context limits
* Attention rules
* KV cache growth
* Decoding constraints
* Template requirements

Local deployment changes where inference occurs.

It does not change the underlying mechanics of inference.

```text id="vxnuj0"
Runtime
+
Hardware

Configuration

```

Success depends on the entire system rather than any individual component.

## 1.30 Local LLM Success Formula

By this point, many of the major components involved in local inference have been introduced.

The final step is understanding how they fit together.

A successful local deployment is rarely the result of a single factor.

Instead, it emerges from the interaction of many components.

### Figure 1.18 — Local LLM Success Formula

Every piece matters.

### Evaluating Local Performance

When troubleshooting local inference, work through the system methodically.

Questions to ask include:

* Is the model type appropriate?
* Is the tokenizer correct?
* Is the chat template correct?
* Are BOS and EOS tokens configured correctly?
* Is the runtime compatible with the model format?
* Does the hardware have sufficient memory?
* Is the context length realistic?

Many apparent model failures originate elsewhere in the stack.

A careful evaluation process often reveals that the issue lies in configuration rather than model capability.

## Key Takeaways

Language models come in several forms, including base, instruct, chat, reasoning, and tool-tuned variants. These model types may share architectures while exhibiting very different behavior because they have been optimized for different objectives.

Running a model locally involves more than downloading weights. A successful local deployment requires compatible weights, tokenizers, templates, runtimes, and hardware resources.

Local inference does not change the underlying mechanics of Transformers, attention, KV caches, or decoding. It simply changes where those computations occur.

Ultimately, successful local AI should be viewed as a system-level outcome. Model quality, tokenizer compatibility, template correctness, runtime support, and hardware resources all contribute to the final experience.
